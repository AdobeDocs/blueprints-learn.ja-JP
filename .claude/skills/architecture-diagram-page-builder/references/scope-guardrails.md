# スコープのガードレール：アーキテクチャページとユースケースパターンページ

ブループリントサイトでは、**アーキテクチャ図ページ**&#x200B;を&#x200B;**ユースケースパターンページ**&#x200B;から分離しています。これは、読み手のニーズが異なるためです。 このドキュメントでは、境界をまたぐコンテンツの場所と処理方法を定義します。

## 主な違い

- **アーキテクチャ ダイアグラム ページ**&#x200B;は、最上位レベルの視覚的参照です。 彼らは答えます：*「これらのシステムはどのように組み合わせるのですか？」 統合ポイントはどこにありますか？ データ フローの形状は何ですか？&quot;* 読者は自分を導くために来る。
- **ユースケースパターンページ**&#x200B;は実装ガイドです。 彼らは答えます：*「この機能を構築するにはどうすればよいですか？ どの機能が関与していますか？ 成果測定のKPI? 実装オプションは何ですか？&quot;* オーディエンスは、ユースケースがあり、それを提供する必要があるときにここにアクセスします。

## アーキテクチャページに属する

| カテゴリ | 例 |
| --- | --- |
| トップレベルアーキテクチャ | AEPとアプリケーション、Experience Cloud マーケティングテクチャ、ハブとエッジのトポロジの概要図 |
| システムデータフロー | リアルタイムとバッチのインジェストパス、ハブとエッジ間のプロファイル同期、ルックアップとアクティベーションフローの比較 |
| 統合ポイント | AEPはAJO、CJA、Target、Campaign、Marketo、Workfrontと連携します。SDKのバウンダリ、API サーフェス |
| デプロイメントトポロジ | Web SDKとモバイルSDKの導入、サーバーサイド転送、エッジノードの配置 |
| アプリケーションアーキテクチャ | 1つのアプリケーション（AJO、CJA、RTCDP）がシステムレベルで内部構造化される方法 |
| ユースケースパターンのポイント | 「このアーキテクチャは、リンクを含むパターン X、Y、Zをサポートしています。アーキテクチャページでは、そのコンテンツを&#x200B;**not**&#x200B;複製しています」 |

## アーキテクチャページに属していない

次のいずれかを書いている場合は、ユースケースパターンページにリダイレクトします（`use-case-pattern-builder` スキルを使用）。

| カテゴリ | なぜ他の場所にあるのか |
| --- | --- |
| KPIと測定式 | ユースケースパターンは結果を測定しますが、アーキテクチャページは結果を測定しません |
| ビジネス目標、ビジネスへの影響 | KBO コンテンツは`/help/blueprints/business-objectives/`未満です。パターンはそれを参照します |
| 戦術的なユースケース | 「カート放棄リマインダー」、「パーソナライズされたホームページヒーロー」など、これらはパターンコンテンツです |
| 機能（`A > B > C > D`） | capabilities構造は、ユースケースパターンテンプレートの一部です |
| ペルソナの物語 | 「マーケターのマリアは…」 スタイルシナリオは、アーキテクチャ参照ではなくパターンに属します |
| 実装オプション | マルチオプション実装ガイダンス（Best for, How it works, Advantages, Limitations）はパターン構造である |
| 基礎/サポート関数テーブル | これらはパターンページセクションです |
| ユースケースごとの前提条件チェックリスト | パターンは、これらを追跡します。代わりに、アーキテクチャページはパターンにリンクします |

## 注目すべきトリガーフレーズ

ユーザーが新しいページを記述する際にこれらのフレーズのいずれかを提供した場合は、一時停止してスコープを再確認します。

- 「KPI」
- 「ビジネスの影響」/「ビジネスの成果」
- 「戦術的なユースケース」/「シナリオの例」
- 「機能」
- 「実装オプション」
- 「最適な用途」
- 「利点と制限事項」
- 「前提条件」
- 「ペルソナ」/「関係者」
- 「測定」

これらは自動的にページの選定を解除するものではありませんが、ユーザーがアーキテクチャページではなくユースケースパターンページを必要としていることを示しています。 生成前に意図を確認する：

## コンテンツがドリフトする場合の対処方法

1. **ドリフトを特定します。** 境界を越えた特定のセクションまたは箇条書きを指します。
2. **ユーザーに2つのオプションを提供します：**
   - アーキテクチャページからセクションをトリミングします（最も一般的なものは、アーキテクチャページの焦点を維持します）。
   - 停止し、そのコンテンツの`use-case-pattern-builder`に切り替えます（ユーザーが実際にパターンページを必要としている場合）。
3. **確認を待ちます。** 静かにコンテンツを書き換えたりドロップしたりしないでください。
4. **アーキテクチャのみのコンテンツ**&#x200B;を保持している場合は、ディープコンテンツを、関連するパターン（既存または作成する予定）にリンクしている`## Use case patterns supported`の下の1つの箇条書きに置き換えます。

## Edgeケース

- **ページは半分のアーキテクチャ、半分のパターンです。** 2つのページに分割します。1つのアーキテクチャページ （このスキル）、1つのユースケースのパターンページ （`use-case-pattern-builder` スキル）です。 クロスリンクします。
- **アーキテクチャ ページでは、エンドツーエンドの単一のユースケースについて説明します。** これはユースケースパターンであり、アーキテクチャページではありません。 `use-case-pattern-builder`にリダイレクトします。
- **アーキテクチャページでは、特定の1つのシナリオのデータフローの例を表示する必要があります。** シナリオが例示のみであり、ページの大部分がシステムアーキテクチャレベルに留まる場合は許容されます。 例を1つの段落に残し、関連するパターンにリンクして詳細を確認します。

## クイックテスト

生成する前に、次の質問を行います。*「読者がこのページにアクセスし、トップレベルのアーキテクチャ参照を期待している場合、1つを取得しますか。それとも、半完成のユースケースのチュートリアルを取得しますか？」* 後者の場合、ページは`use-case-pattern-builder`に属します。
