# 既知の訪問者の web/アプリのパーソナライゼーション

このリファレンスプランは、[!DNL Adobe Journey Optimizer] （AJO）および [!DNL Adobe Real-Time Customer Data Platform] （RT-CDP）を使用して、識別された訪問者にパーソナライズされたコンテンツを配信するための包括的な実装ガイドを提供します。 これは、あらゆる実行可能な実装アプローチ、各フェーズで行う必要がある決定、設定をサポートするExperience Leagueのドキュメントを理解する必要があるソリューションアーキテクト、マーケティングテクニカルボロジ、および実装エンジニア向けに設計されています。

既知の訪問者による web/アプリのパーソナライゼーションは、認証済みデジタルエクスペリエンスの主要なパーソナライゼーションパターンです。 セッション内の行動シグナルのみに依存する匿名の訪問者パーソナライゼーションとは異なり、このパターンでは、過去の行動データ、セグメントメンバーシップ、ロイヤルティ層、購入履歴、ライフサイクルステージ、計算属性、傾向スコアなど、完全な統合プロファイルが活用されます。 （AJO web チャネル経由の） web ページ間、モバイルのアプリ内メッセージ間およびコンテンツカード間で、パーソナライゼーションをサポートします。

このガイドでは、すべての実行可能な実装オプション（セグメントベース、決定ベース、マルチサーフェス）と、トレードオフ、決定ガイダンス、[!DNL Adobe Experience League] のドキュメントへの参照を紹介します。

## ユースケースの概要

認証済みデジタルプロパティ（e コマースサイト、バンキングポータル、購読サービス、ロイヤルティプログラム、モバイルアプリ）を持つ組織は、各顧客のブランドとの関係を反映したパーソナライズされたエクスペリエンスを提供する必要があります。 ID 解決を通じて訪問者がログインまたは認識されると、プラットフォームは完全な統合プロファイルにアクセスし、特定の属性、動作、環境設定に合わせてカスタマイズされたコンテンツを配信できます。

このパターンは、特定の訪問者が web プロパティに到達したりモバイルアプリを開いたりするシナリオに対応しており、システムはリアルタイムのプロファイルデータとオーディエンスメンバーシップに基づいて、表示する最適なコンテンツ、オファーまたはプロモーションを決定する必要があります。 パーソナライゼーションの決定はエッジでミリ秒単位で行われるので、目に見える待ち時間なしでサブ秒のコンテンツ配信が可能になります。

このパターンは、決定論的なパーソナライゼーション（特定のコンテンツが特定のオーディエンスセグメントにマッピングされる）と動的な意思決定（AJO Decisioning が実施要件ルールとランキング戦略を評価して、プロファイルごとに最適なコンテンツを選択する）の両方をサポートしています。 Web ページ、モバイルのアプリ内メッセージ、コンテンツカードなど、複数のデジタルサーフェスに及び、顧客のデジタルジャーニー全体で一貫したパーソナライゼーションを可能にします。

## 主なビジネス目標

次のビジネス目標は、このユースケースパターンで対処されます。

### パーソナライズされた顧客体験の提供

コンテンツ、オファーおよびメッセージングを、個々の環境設定、行動およびライフサイクルステージに合わせて調整します。 詳しくは、[&#x200B; パーソナライズされたカスタマーエクスペリエンスの配信 &#x200B;](../../business-objectives/customer-experience/deliver-personalized-customer-experiences.md) を参照してください。

| KPI | 説明 |
| --- | --- |
| エンゲージメント | デジタルサーフェスをまたいでパーソナライズされたコンテンツを使用したインタラクションの頻度と深度 |
| コンバージョン率 | パーソナライズされたコンテンツの受信後に、目的のアクションを完了した訪問者の割合 |
| 顧客満足度（CSAT） | 適切にパーソナライズされたデジタルエクスペリエンスによって、全体的な満足度が向上します |

### Web サイトエンゲージメントの向上

関連するエクスペリエンスを通じて、サイトでの時間、セッションあたりのページ数、web コンテンツとのインタラクションを改善します。 詳しくは、[Web サイトエンゲージメントの向上 &#x200B;](../../business-objectives/acquisition-growth/increase-website-engagement.md) を参照してください。

| KPI | 説明 |
| --- | --- |
| Time On （Web）ページ | パーソナライズされたコンテンツ領域への滞留時間の増加 |
| エンゲージメント | パーソナライズされた要素（クリック、スクロール、インタラクション）を使用したインタラクション率 |
| コンバージョン率 | パーソナライズされたコンテンツからのコンバージョンとデフォルトコンテンツの比較を改善 |

### モバイルアプリのエンゲージメントの向上

パーソナライズされたアプリ内エクスペリエンスを通じて、毎日のアクティブな使用、機能の採用、アプリ内コンバージョンを促進します。

| KPI | 説明 |
| --- | --- |
| エンゲージメント | パーソナライズされたメッセージとコンテンツカードを使用したアプリ内インタラクション率 |
| 定着 | パーソナライズされたエクスペリエンスにより、アプリの保持率が向上しました |
| コンバージョン率 | パーソナライズされたオファーやレコメンデーションからのアプリ内コンバージョンの改善 |

## 戦術的な使用例

このパターンに対する一般的な戦術的な実装を次に示します。

- ロイヤルティ層またはライフサイクルステージ別のホームページのヒーローのパーソナライゼーション – 顧客が新規、アクティブ、リスクにさらされている、またはVIPのどれに該当するかによって、異なるヒーローバナーを表示します
- 購入履歴に基づく製品の推奨カルーセル – 過去の購入データと製品アフィニティスコアを使用して、関連する製品の提案を表示します
- 顧客セグメント別のパーソナライズされたプロモーションバナー – 価値が高い、リスクのある、新しい顧客セグメントに対して様々なプロモーションを表示します
- 機能の採用に基づくモバイルユーザー向けのアプリ内メッセージ – 使用パターンに基づいて、使用率の低い機能をユーザーに案内します
- アカウントダッシュボードでパーソナライズされたオファーを使用したコンテンツカード – 顧客のプロファイルに合わせてカスタマイズされた、永続的で割引可能なオファー
- お客様の階層に基づいてパーソナライズされた価格またはディスカウントの表示：階層固有の価格またはロイヤルティプログラムメンバーへの特別割引を表示します
- 所有製品に基づくクロスセルのレコメンデーションウィジェット – 現在のポートフォリオに基づいて補完的な製品やサービスを提案します
- 興味に基づいてパーソナライズされたナビゲーションまたはコンテンツの順序：示された環境設定に基づいてコンテンツモジュールまたはナビゲーション要素を並べ替える

## ユースケースパターン

ここでは、コアパターンとその機能チェーンについて説明します。

**既知の訪問者による web/アプリのパーソナライゼーション**

Web、アプリ内モバイル、コンテンツカードの各サーフェスをまたいで、リアルタイムのプロファイルとセグメントメンバーシップに基づいて、パーソナライズされたコンテンツ、オファーまたはプロモーションを、特定された訪問者に配信します。

**関数チェーン：** オーディエンス評価/Personalization Decisioning/サーフェス/チャネル設定/コンテンツ配信/インプレッショントラッキング/レポート

## アプリケーション

このユースケースパターンでは、次のアプリケーションが使用されています。

- **[!DNL Adobe Journey Optimizer]（AJO）** - Web チャネル設定、アプリ内チャネル設定、コンテンツカードチャネル設定、決定（オファーの選択とランキング）、メッセージオーサリング（パーソナライズされたコンテンツの作成）、キャンペーンの実行、コンテンツの実験およびレポート
- **[!DNL Adobe Real-Time Customer Data Platform]（RT-CDP）** — オーディエンス評価（エッジ、ストリーミングおよびバッチ）、Edge Networkによるリアルタイムプロファイルルックアップ、計算属性と傾向スコアを使用したプロファイルエンリッチメント
- **[!DNL Adobe Experience Platform]（AEP）** - プロファイルストア、ID サービス、Web SDK、Mobile SDK、データストリーム設定、Edge Network 配信

## アプリケーション関数

このプランは、アプリケーション機能カタログから次の機能を実行します。 関数は、番号付きステップではなく、実装フェーズにマッピングされます。

### [!DNL Journey Optimizer] （AJO）

| 関数 | 実装フェーズ | 説明 |
| --- | --- | --- |
| チャネル設定 | サーフェスとチャネルの設定 | パーソナライゼーション配信用の web、アプリ内およびコンテンツカードチャネルサーフェスの設定 |
| メッセージオーサリング | コンテンツオーサリング | 動的コンテンツ、パーソナライゼーション式、各サーフェスの条件ブロックを使用して、パーソナライズされたコンテンツのバリアントを作成します |
| キャンペーンの実行 | Campaign の設定とアクティブ化 | オーディエンス、サーフェスおよびコンテンツをバインドする web キャンペーン（スケジュール済みまたは API トリガー）を作成してアクティブ化します |
| 決定 | 決定の設定（オプション B/C） | 動的なパーソナライゼーションのための実施要件ルール、ランキング戦略およびオファー/コンテンツ項目を含んだ決定ポリシーを設定します |
| コンテンツ実験 | 最適化（オプション） | パーソナライズされたコンテンツバリアントに対して A/B テストを実行し、パフォーマンスを最適化します |

## 実装オプション

この節では、このユースケースパターンで使用可能な実装アプローチについて説明します。

### オプション A：セグメントベースの web パーソナライゼーション

**最適な対象：** 特定のコンテンツのバリエーションがオーディエンスセグメント（ロイヤルティ層固有のヒーローバナー、ライフサイクルステージメッセージ、顧客セグメントのプロモーションコンテンツ）に直接マッピングされる、決定論的なパーソナライゼーション。 コンテンツとセグメントのマッピングが適切に定義されており、動的なランキングや最適化が不要な場合に最適です。

**仕組み：**

セグメントベースのパーソナライゼーションでは、コンテンツのバリアントをオーディエンスセグメントに直接マッピングします。 ページの読み込み時に既知の訪問者のプロファイルがエッジ対象のオーディエンスに対して評価されると、システムは訪問者が属するセグメントを決定し、対応するコンテンツバリアントを表示します。 選択はセグメントメンバーシップの優先度に基づきます – 訪問者が複数のセグメントの対象になると、優先度が最も高いセグメントのコンテンツが表示されます。

このアプローチでは、条件付きコンテンツルールまたは複数の処理ターゲティング設定を持つAJO web チャネルキャンペーン（またはアプリ内/コンテンツカードキャンペーン）を使用します。 各オーディエンスセグメントは、特定のコンテンツエクスペリエンスに関連付けられています。 決定エンジンは必要ありません。コンテンツ選択ロジックは、完全にセグメント駆動型です。

コンテンツは、オーディエンスのメンバーシップまたはプロファイル属性に基づいて異なるコンテンツをレンダリングする動的コンテンツブロックを含んだAJO メッセージオーサリングインターフェイスを使用して作成されます。 Web SDKまたはモバイルSDKは、パーソナライズされたエクスペリエンスをEdge Networkでリアルタイムに提供します。

**主な考慮事項：**

- セグメントごとにコンテンツのバリアントを事前に作成する必要があります
- セグメント定義は、リアルタイム選定に対してエッジ適格である必要があります
- 新しいセグメントやコンテンツのバリアントを追加するには、キャンペーン設定を更新する必要があります
- コンテンツの選択は決定的です。同じセグメントには常に同じコンテンツが表示されます

**メリット：**

- コンテンツとセグメントのマッピングが明確で、実装と管理が簡単
- パーソナライゼーションロジックを関係者に理解して説明しやすい
- 決定のオーバーヘッドなし – コンテンツの解決の迅速化
- 各セグメントが受け取るコンテンツの完全な制御

**制限事項：**

- セグメント数やコンテンツのバリアント数が増加した場合の柔軟性の制限
- セグメントメンバーシップを超えるプロファイルレベルのシグナルに基づいてコンテンツ選択を動的に最適化できない
- 新しいコンテンツバリアントの追加には、キャンペーンの手動更新が必要です
- 複数のオファーが同じプレースメントで競合する場合、次善のオファーのシナリオをサポートしない

### オプション B：決定ベースのパーソナライゼーション

**最適な対象：** 実施要件ルールとランキング戦略（ホームページでの次善のオファー、パーソナライズされた製品レコメンデーション、動的なプロモーションバナー選択）を使用して、プロファイルごとに最適なコンテンツまたはオファーを選択する必要がある動的なパーソナライゼーション。 複数のオファーまたはコンテンツ項目が同じプレースメントで競合し、システムが最適なオファーを選択する必要がある場合に最適です。

**仕組み：**

意思決定ベースのパーソナライゼーションでは、AJO Decisioning を使用して各訪問者のプロファイルをコンテンツ項目またはオファーのカタログと照合し、実施要件ルールを適用して選定する項目を決定し、ランキング戦略（優先度ベース、式ベース、AI ランク）を適用して各プレースメントに最適な項目を選択します。

この実装には、プレースメント（コンテンツが表示される場所）の作成、実施要件ルールとコンテンツ表示域を使用したオファーの定義、オファーをコレクションに整理、ランキング戦略を使用してコレクションにプレースメントをバインドする決定ポリシーの作成が含まれます。 実行時に、訪問者がページを読み込むと、Edge Networkは、訪問者のプロファイルに対して決定ポリシーを評価し、選択したオファーコンテンツを返します。

このアプローチは、次善のオファー、キャッピングを使用したパーソナライズされたプロモーション、AI に最適化されたコンテンツ選択など、高度なパーソナライゼーションシナリオをサポートします。 オファーには、プロファイル単位およびグローバルキャッピングの制限、有効期限の日付範囲、プロファイル属性とオーディエンスメンバーシップを組み合わせた複雑な実施要件条件を設定できます。

**主な考慮事項：**

- より多くの事前設定（プレースメント、オファー、実施要件ルール、コレクション、決定）が必要です
- ランキング戦略は、優先度ベース（手動）、式ベース（カスタム式）または AI ランク付け（自動最適化）にすることができます
- AI ランキングでは、モデルトレーニングに 1,000 以上のコンバージョンイベントが必要です
- 高スループットのシナリオでは、オファーキャッピングカウンターにわずかな遅れが生じる場合があります
- フォールバックオファーは、パーソナライズされたオファーが適合しない場合に設定する必要があります

**メリット：**

- セグメントとコンテンツのマッピングをハードコーディングしない、プロファイルごとの動的なコンテンツ選択
- 複雑な実施要件条件とランキングロジックをサポート
- AI ランクオプションにより、手動の操作なしで自動最適化を実現
- オファーキャッピングにより、特定のコンテンツの過剰公開を防ぐ
- オファーの一元管理 – オファーは、キャンペーンやチャネルをまたいで再利用できます

**制限事項：**

- セグメントベースのパーソナライゼーションよりも実装が複雑になる
- AI ランキングには、トレーニングに十分なコンバージョンイベント量が必要です
- 意思決定評価では、直接セグメントベースのコンテンツ配信と比較して、限界遅延が追加されます
- 式ベースのランキングでは、意図しない優先順位付けを避けるために慎重なデザインが必要です

### オプション C：マルチサーフェスパーソナライゼーション（web + アプリ内+ コンテンツカード）

**最適な対象：** 複数のデジタルサーフェスにわたって一貫したパーソナライゼーションを実現し、web ページ、モバイルのアプリ内メッセージ、コンテンツカードに調整されたパーソナライズされたエクスペリエンスを提供します。 組織が web とモバイルアプリの両方のプロパティを持ち、すべてのデジタルタッチポイントにわたって統一されたパーソナライゼーションロジックを必要としている場合に最適です。

**仕組み：**

マルチサーフェスパーソナライゼーションは、オプション A （セグメントベース）またはオプション B （決定ベース）を拡張し、複数のAJO サーフェスタイプにわたってパーソナライズされたコンテンツを提供します。 Web、アプリ内、コンテンツカードなどの各サーフェスタイプには、異なるコンテンツ形式と配信メカニズムが使用される場合がありますが、基になるパーソナライゼーションロジック（オーディエンスメンバーシップまたは決定）は共有されます。

この実装には、各サーフェスタイプのチャネルサーフェスの設定、サーフェス固有のコンテンツ（web サーフェス用の web HTML/CSS、アプリ内向けの構造化メッセージ、コンテンツカード用のカードレイアウト）のオーサリング、適切なサーフェスをターゲットとするキャンペーンの作成が含まれます。 Web SDKが web サーフェス配信を処理するのに対して、Mobile SDKはアプリ内配信とコンテンツカード配信を処理します。

コンテンツカードは、アカウントダッシュボードやアプリのホーム画面で永続的に無効化できるパーソナライズされたメッセージを表示する場合に特に役立ちます。 アプリ内メッセージは、コンテキストに応じた、セッション固有のコミュニケーションに最適です。 Web パーソナライゼーションでは、ヒーローバナー、レコメンデーションウィジェットおよびプロモーションコンテンツを処理します。

**主な考慮事項：**

- 各サーフェスタイプには、独自のチャネルサーフェス設定とコンテンツオーサリングが必要です
- Web SDKとモバイル SDKを実装し、設定する必要があります
- コンテンツは、サーフェス形式（異なる寸法、レイアウト、インタラクションパターン）ごとに設計する必要があります
- 共有オーディエンスと決定ロジックにより、サーフェス間の一貫性が確保されます
- テストは、デバイス間のすべてのサーフェスタイプをカバーする必要があります

**メリット：**

- すべてのデジタルタッチポイントにわたる一貫したパーソナライゼーションエクスペリエンス
- 共有オーディエンスと決定のロジックにより重複が減る
- コンテンツカードは、永続的で割り込みのないパーソナライゼーションサーフェスを提供します
- アプリ内メッセージにより、モバイルでコンテキストに応じたセッション固有のパーソナライゼーションが可能になります

**制限事項：**

- 実装が非常に複雑 – 各サーフェスタイプに対して設定が必要
- Web SDKと Mobile SDKの両方を実装する必要があります
- コンテンツは、サーフェス形式ごとに設計および管理する必要があります
- テストの範囲は、サーフェスのタイプを追加するたびに増加します

### 適切なオプションを選択

適切な実装方法を選択するには、次の質問から始めてください。

1. **サーフェスの数** Web とモバイルアプリの両方でパーソナライゼーションが必要な場合は、オプション C （基盤となるロジックの A または B のいずれかに基づいて構築される）を選択します。 Web のみの場合は、A と B のいずれかを選択します。

2. **コンテンツの選択はどの程度動的ですか？** セグメントとコンテンツのバリアントとのマッピングが明確に定義されている場合（例：3 ～ 5 のロイヤルティ層で、それぞれに特定のヒーローバナーが付いている場合）、オプション A を選択します。複雑な実施要件とランキングを持つオファーカタログから選択する必要がある場合は、オプション B を選択します。

3. **AI に最適化された選択範囲は必要ですか？** プロファイルごとに最適なパフォーマンスを発揮するコンテンツをシステムが自動的に学習し、最適化する場合は、AI ランク判定を使用してオプション B を選択します。

4. **コンテンツバリアントはいくつですか？** 明確なセグメントマッピングを持つコンテンツバリアントの数が 10 個未満の場合は、オプション A を使用する方が簡単です。 実施要件のフィルタリングとランキングが必要なオファーが多数ある場合は、オプション B の方がスケールが大きくなります。

5. **他のチャネルにも拡張する予定はありますか？** 決定ロジックが最終的にメール、web、その他のチャネルをまたいでオファーを提供する必要がある場合、オプション B は、クロスチャネル Offer Decisioning で拡張される一元的な決定基盤を提供します。

## 実装フェーズ

このセクションでは、実装の各フェーズについて詳しく説明します。

### フェーズ 1：オーディエンスの定義と評価の設定

**アプリケーション関数：** RT-CDP：オーディエンス評価

**設定内容：** パーソナライゼーションコンテンツの選択を推進するオーディエンスを定義します。 これらのオーディエンスは、パーソナライズされたエクスペリエンス（ロイヤルティ層、ライフサイクルステージ、行動コホート、製品親和性グループ）を受け取る訪問者セグメントを表します。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：オーディエンス評価方法**
>
>パーソナライゼーションのために、オーディエンスメンバーシップをどの程度短時間で解決する必要がありますか。 これは、パーソナライゼーションをページの読み込み時に実行できるか、遅延を必要としているかに直接影響します。
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| エッジ評価 | 1 秒未満の選定が必要なリアルタイムの web/アプリのパーソナライゼーション | 単純なプロファイル属性チェックとセグメントメンバーシップに制限される。 時系列クエリや複雑な集計はありません。 既知の訪問者パーソナライゼーションに必要です。 |
>| ストリーミング評価 | 行動イベントに基づいてプロファイルがオーディエンスに入る/出る際の、ほぼリアルタイムの選定 | 単一のイベントおよび期間限定のウィンドウクエリをサポートします。 オーディエンスの変更は数分以内に反映されます。 わずかな遅延を許容できるアプリ内およびコンテンツカードのサーフェスに適しています。 |
>| バッチ評価 | 複雑な集計や履歴パターンに基づいて、セグメントのオーディエンスを毎日更新します | 完全なセグメントルール関数のサポート。 リアルタイムのパーソナライゼーションには適していませんが、複雑な事前計算済みセグメントでエッジオーディエンスを補完できます。 |

>[!NOTE]
>**決定：Personalization属性**
>
>オーディエンス定義とコンテンツ選択を推進する必要があるのは、どのプロファイル属性ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| プロファイル属性（ロイヤルティ層、ライフサイクルステージ） | 既知の顧客プロパティに基づく決定論的パーソナライゼーション | 安定した、明確に定義された属性。 コンテンツのバリエーションに簡単にマッピングできます。 プロファイルが適切に設定されている場合は、エッジで使用できます。 |
>| 行動シグナル（購入履歴、閲覧パターン） | 最近の行動とエンゲージメントパターンに基づくPersonalization | 計算属性またはストリーミングセグメントが必要です。 より動的ですが、維持がより複雑です。 |
>| 傾向スコア（[!DNL Customer AI]） | コンバージョン、チャーンまたは購入の可能性に基づく予測パーソナライゼーション | 計算属性が必要です。 高度なパーソナライゼーションを可能にしますが、ML モデルトレーニングデータが必要です。 |
>| 組み合わせアプローチ | ほとんどの実稼動実装 | 絞り込みのために、行動信号と傾向スコアを使用したプライマリセグメント化にプロファイル属性を使用します。 |

### フェーズ 2：決定の設定（オプション B および C のみ）

**Application 関数：** AJO:Decisioning

**設定内容：** 各訪問者に最適なコンテンツやオファーを動的に選択する意思決定インフラストラクチャを設定します。 これには、プレースメント（オファーが表示される場所）、オファー（使用可能なコンテンツ）、実施要件ルール（選定者）、ランキング戦略（最適な選択方法）、決定ポリシー（すべてがつながる方法）が含まれます。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：ランキング戦略**
>
>各訪問者に最適なオファーを選択するには、実施要件を満たすオファーをどのようにランク付けする必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 優先度ベース （手動） | 明確なオファー階層を持つシンプルなユースケース | 各オファーには、手動の優先度値があります。 最も優先度が高い実施要件を満たすオファーが優先されます。 理解と制御が容易。 |
>| 式ベース（カスタム式） | ランキングでプロファイル属性を考慮するタイミング | カスタムランキング式は、プロファイルデータを参照します（例：ロイヤルティ層ごとのスコア +最新性）。 柔軟性がありますが、式の設計とテストが必要です。 |
>| AI ランク （自動最適化） | オファーの選択をシステムで自動的に最適化するタイミング | ML モデルでは、どのオファーがどのプロファイルに対して最もパフォーマンスが高いかを学習します。 トレーニングには 1,000 以上のコンバージョンイベントが必要です。 高トラフィックのプレースメントに最適です。 |

>[!NOTE]
>**決定：オファーキャッピング**
>
>1 人の訪問者に対するオファーの表示回数、またはすべての訪問者に対するオファーの表示回数に制限を設ける必要がありますか。
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| プロファイルごとのキャップ | 疲労が同じオファーを繰り返し表示するのを防ぐ | 一定期間の訪問者あたりのインプレッション数を制限します。 パーソナライズされたエクスペリエンスの多様性を確保します。 |
>| グローバルキャップ | プロモーションまたは限定提供オファーの合計インプレッション数の制限 | すべての訪問者の合計インプレッション数に上限を設定します。 数量限定プロモーションに役立ちます。 |
>| キャップなし | エバーグリーンコンテンツまたは常に関連性の高いオファー | インプレッションの制限はありません。 ロイヤルティ層のバナーなどの永続的なコンテンツに適しています。 |

**UI ナビゲーション：** [!DNL Journey Optimizer]/コンポーネント/意思決定管理

**キー設定の詳細：**

- オファーが表示される各サーフェス（web バナー、アプリ内メッセージ領域、コンテンツカードスロット）のプレースメントを作成します
- プロファイル属性とオーディエンスメンバーシップを参照するセグメントルール式を使用して、実施要件ルールを定義します
- プレースメントごとにコンテンツ表示域を使用してパーソナライズされたオファーを作成する

- コレクション修飾子（タグ）を使用してオファーを整理し、コレクションにグループ化する
- 選択したランキング戦略で、コレクションにプレースメントをバインドする決定ポリシーを作成します

### フェーズ 3：サーフェスとチャネルの設定

**アプリケーション関数：** AJO:Channel Configuration

**設定内容：** パーソナライズされたコンテンツの配信先を定義するチャネルサーフェスを設定します。 各サーフェスタイプ（web、アプリ内、コンテンツカード）には、サーフェス URI、コンテンツ形式、配信パラメーターを指定する独自の設定が必要です。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：ターゲットサーフェス**
>
>パーソナライズされたコンテンツを受信するデジタルサーフェスはどれですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| Web チャネルのみ | web プロパティに重点を置いたPersonalization | Web SDKが必要です。 最も簡単な実装。 ヒーローバナー、プロモーションエリア、レコメンデーションウィジェットについて説明します。 |
>| アプリ内チャネルのみ | モバイルアプリエクスペリエンスに重点を置いたPersonalization | Mobile SDKが必要です。 アプリ内のコンテキストに応じたセッション固有のメッセージについて説明します。 |
>| コンテンツカードチャネルのみ | 永続的で無効なパーソナライズされたメッセージ | Mobile SDKまたは Web SDKが必要です。 カードは、閉じられるまで保持されます。 ダッシュボードやホーム画面に最適です。 |
>| 複数のサーフェス（オプション C） | Web とモバイルをまたいだ一貫性のあるパーソナライゼーション | Web SDKと Mobile SDKの両方が必要です。 各サーフェスには、個別の設定とコンテンツが必要です。 |

>[!NOTE]
>**決定：web サーフェス設定アプローチ**

>| マルチページアプリケーション（MPA） | 従来のサーバーレンダリング web ページ | Edge Network応答を介してページの読み込み時に配信されるコンテンツ。 ページレベルのサーフェス URI 設定が必要になる場合があります。 |

- アプリ内サーフェスの場合：アプリ ID とサーフェスタイプを使用してモバイルアプリサーフェスを設定します
- コンテンツカードサーフェスの場合：アプリ ID または web コンテキストを使用してコンテンツカードサーフェスを設定します

### フェーズ 4：コンテンツのオーサリング

**アプリケーション関数：** AJO：メッセージオーサリング

**設定内容：** 各サーフェス、セグメントまたはオファーのパーソナライズされたコンテンツのバリアントを作成します。 これには、視覚的なレイアウトのデザイン、プロファイル属性を参照するパーソナライゼーション式の追加、条件付きコンテンツブロックの設定、再利用可能なコンテンツフラグメントの作成などが含まれます。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：コンテンツアプローチ**
>
>このユースケースでは、パーソナライズされたコンテンツをどのように構造化する必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 条件付きコンテンツブロック | 同じレイアウト内の異なるコンテンツセクションは、オーディエンスによって異なります | 条件付きルールを含んだ単一のコンテンツアセット。 小規模なバリエーション（ヘッドライン、CTA テキスト、画像の入れ替え）に効率的です。 |
>| 処理ごとに異なるコンテンツバリアント | オーディエンスごとに根本的に異なるレイアウトまたはデザイン | 複数の完全なコンテンツアセット。 柔軟性は向上しますが、メンテナンスの手間が増えます。 コンテンツ実験に必要です。 |
>| 意思決定主導のコンテンツ | オファーカタログから動的に選択されたコンテンツ | オファー表示域はコンテンツを定義します。 コンテンツ管理は、オファーライブラリ内で一元的に行われます。 |

>[!NOTE]
>**決定：Personalizationの深度**
>
>パーソナライズするコンテンツの割合
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| サーフェスレベルのパーソナライゼーション | パーソナライズされるのは、特定の要素（ヒーロー画像、CTA、オファーバナー）のみです | 複雑さの軽減。 影響の大きい領域に焦点を当てたパーソナライゼーション。 最も一般的な出発点。 |
>| フルページのパーソナライゼーション | ページレイアウト全体またはコンテンツの順序をパーソナライズする | 複雑さの増大。 広範なコンテンツ作成が必要です。 最適なエクスペリエンスを提供します。 |
>| トークンレベルのパーソナライゼーション | インラインパーソナライゼーショントークン（名前、層、ポイントの残高） | 最も単純なフォーム。 プロファイル属性値を、それ以外の場合は静的なコンテンツに挿入します。 |

**UI ナビゲーション：** [!DNL Journey Optimizer]/キャンペーン/キャンペーンを作成/コンテンツを編集

**オプションが分岐する場所：**

**オプション A （セグメントベース）の場合：**

- Web デザイナーまたはコードエディターを使用して、オーディエンスセグメントごとにコンテンツのバリアントを作成します
- オーディエンスメンバーシップに基づく条件での動的コンテンツブロックの使用

- 決定プレースメントを埋め込むことで、決定出力を web ページまたはアプリに統合
- コンテンツは実行時に動的に選択され、オーサリングでは個々のオファー項目に焦点が当てられます

**オプション C （マルチサーフェス）の場合：**

- ターゲットサーフェスごとにサーフェス固有のコンテンツを作成します（web HTML/CSS、アプリ内構造化メッセージ、コンテンツカードレイアウト）。
- 各サーフェスの形式制約に適応しながら、サーフェス間で一貫したパーソナライゼーションロジックを維持します
- 各サーフェスタイプでのコンテンツレンダリングのテスト

### フェーズ 5：キャンペーンの設定とアクティブ化

**アプリケーション関数：** AJO：キャンペーンの実行

**設定内容：** 配信のためにオーディエンス、サーフェス、コンテンツをバインドするAJO キャンペーンを作成し、アクティブ化します。 Web パーソナライゼーションの場合、キャンペーンは、通常、1 回限りのスケジュールされた送信ではなく、即時または継続的なアクティブ化のために設定されます。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：キャンペーンタイプ**
>
>パーソナライゼーションキャンペーンはどのようにトリガーしますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| スケジュールされたキャンペーン （常時稼動） | 条件を満たすすべての訪問者に対して継続的に実行される継続的なパーソナライゼーション | 開始日を即時に、終了日を設定しない。 キャンペーンは、手動で停止するまでアクティブです。 Web パーソナライゼーションに最も一般的です。 |
>| スケジュールされたキャンペーン （時間制限） | 特定の昇格期間に関連付けられたPersonalization | 開始日と終了日を設定します。 キャンペーンは、終了日の後に自動的に停止します。 季節ごとのプロモーションや期間限定オファーに適しています。 |
>| API トリガーキャンペーン | 特定のアプリケーションイベントによってトリガーされるPersonalization | プログラムによってトリガーされます。 特定のシステムイベントに対する応答としてパーソナライゼーションを表示する場合にのみ役立ちます。 |

>[!NOTE]
>**決定：フリークエンシーキャップ**
>
>このパーソナライゼーションに対して、インプレッション頻度を制限する必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 頻度ルールの適用 | 訪問者を疲労させる可能性のあるプロモーションまたはオファーベースのパーソナライゼーション | 同じパーソナライゼーションが何度も表示されるのを防ぎます。 AJO business rules を介して設定されます。 |
>| フリークエンシーキャップなし | エバーグリーンコンテンツのパーソナライゼーション（ロイヤルティ層のバナー、ダッシュボードコンテンツ） | 疲労を引き起こさない、常に関連性の高いコンテンツ。 インプレッションの制限は必要ありません。 |

**UI ナビゲーション：** [!DNL Journey Optimizer]/キャンペーン/キャンペーンを作成

**キー設定の詳細：**

- Web、アプリ内またはコンテンツカードチャネルの選択、およびフェーズ 3 で設定したサーフェスの選択
- フェーズ 1 で定義したターゲットオーディエンスのバインド
- フェーズ 4 で作成したコンテンツのリンク

- コンテンツ実験の場合：アクティベーション前に、実験の切り替えを有効にして、処理を定義し、トラフィック配分と成功指標を設定します

## 実装に関する考慮事項

この節では、このユースケースパターンに関連するガードレール、一般的な落とし穴、ベストプラクティスおよびトレードオフの決定について説明します。

### ガードレールと制限

- Edge Network ルックアップでは、エッジ評価されたセグメント（[&#x200B; リアルタイム顧客プロファイルガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/guardrails) の応答時間SLAは 200 ミリ秒未満です
- サンドボックスあたり最大 4,000 個のセグメント定義 – [&#x200B; セグメント化ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/guardrails)
- Edge セグメントは、単純な属性チェックとセグメントメンバーシップクエリ （時系列クエリなし）に制限されています。[Edgeのセグメント化 &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/segmentation/methods/edge-segmentation)
- サンドボックスごとにEdgeでアクティブにできる結合ポリシーは 1 つだけです [&#x200B; 結合ポリシー &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/merge-policies/overview)
- サンドボックスあたり最大 10,000 件の承認されたパーソナライズされたオファー – [&#x200B; 意思決定管理ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/journey-optimizer/using/get-started/guardrails)
- 1 つの決定につき最大 30 個のプレースメント — [Journey Optimizer ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/journey-optimizer/using/get-started/guardrails)
- AI ランキングモデルでは、トレーニングに 1,000 以上のコンバージョンイベントが必要です
- シングルスコープリクエストの場合、SLAは P95 で 500 ms 未満です
- サンドボックスあたり最大 500 個のアクティブなライブキャンペーン — [Journey Optimizer ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/journey-optimizer/using/get-started/guardrails)
- サンドボックスあたり最大 25 個のアクティブな計算属性 – [&#x200B; 計算属性ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/computed-attributes/overview)

### よくある落とし穴

- **Edge結合ポリシーが設定されていません：** エッジアクティブな結合ポリシーがないと、Edge Networkは認証済みプロファイルを解決できず、パーソナライゼーションが失敗するか、匿名の動作にフォールバックします。 サンドボックスに結合ポリシーが 1 つだけ `isActiveOnEdge: true` まれていることを確認します。
- **オーディエンスがエッジ適格ではありません：** オーディエンスセグメントルール式で、時系列クエリ、複雑な集計またはバッチのみのセグメントへの `inSegment()` 参照が使用される場合、エッジ評価の対象とならず、リアルタイムパーソナライゼーションを強化できません。 オーディエンス定義中にエッジの実施要件を検証します。
- **認証中の ID ステッチギャップ：** 訪問者が匿名から認証済みに移行すると、プロファイルがまだ解決されていない瞬間が生じる場合があります。 ID ステッチが適切に設定されていること、および Web SDKがログイン後すぐに `identityMap` を使用して認証済み ID を送信していることを確認します。
- **決定でフォールバックオファーが見つからない：** フォールバックオファーが設定されておらず、パーソナライズされたオファーが訪問者に該当しない場合、決定は空のコンテンツを返し、エクスペリエンスが壊れます。 すべてのプレースメントをカバーするフォールバックオファーを常に設定します。
- **Web SDKが提案表示イベントを送信しない：** `renderDecisions` が `true` に設定されていても、表示イベントが送信されない場合、レポートには実際のインプレッションは反映されません。 ブラウザー開発者ツールでネットワークリクエストを調べて、イベントトラッキングを検証します。
- **ページの読み込み時にコンテンツがちらつく：** 決定の呼び出し中にパーソナライズされたコンテンツが事前に非表示になっていない場合、訪問者はデフォルトコンテンツが置き換えられる前に一時的に表示される場合があります。 事前非表示のスニペットまたは CSS ベースの事前非表示を使用して、ちらつきを排除します。

### ベストプラクティス

- セグメントベースのパーソナライゼーション（オプション A）から初期実装を開始し、オファーカタログの拡大と最適化のニーズの増加に合わせて意思決定ベース（オプション B）に進化します
- 可能な限り、リアルタイムのパーソナライゼーションにエッジ評価オーディエンスを使用します。ストリーミングオーディエンスとバッチオーディエンスを予約して、補足的なユースケースを実現します
- パーソナライズされたエクスペリエンスにコンテンツ実験を実装して、パーソナライゼーションがデフォルトコンテンツよりも測定可能な上昇率を促進することを検証します
- 正常な劣化戦略を使用したデザインパーソナライゼーション – プロファイルをエッジで解決できない場合は、エクスペリエンスが損なわれるのではなく、適切にデザインされたデフォルトコンテンツを表示します
- 計算属性を使用して、エンゲージメントスコア、製品の親和性、前回購入からの日数など、価値の高いパーソナライゼーションシグナルを作成します。これにより、セグメントベースのパーソナライゼーションと意思決定ベースのパーソナライゼーションの両方の品質が向上します
- パーソナライズされたコンテンツを常に最新の状態に保ち、ブランドに合わせ、あらゆる面で準拠するように、コンテンツガバナンスプロセスを維持します
- セグメント別にパーソナライゼーションパフォーマンスを監視して、パーソナライゼーションから最もメリットを得るオーディエンスと、その上昇率が最も高いオーディエンスを特定します

### トレードオフの決定

>[!NOTE]
>**トレードオフ：Personalizationの精度とコンテンツ管理の複雑さの比較**
>
>より詳細なパーソナライゼーション（より多くのセグメント、より多くのコンテンツのバリアント、より多くのサーフェス）を行うと、エクスペリエンスはカスタマイズされますが、コンテンツの作成、メンテナンス、ガバナンスに関する取り組みが比例的に多く必要になります。
>
>- **精度の向上のお勧め：** 顧客体験の向上、エンゲージメントの向上、コンバージョン率の向上
>- **精度の低さのお気に入り：** 実装の高速化、コンテンツのメンテナンス負担の軽減、ガバナンスの簡素化
>- **推奨事項：** 明確なコンテンツの差別化を図りながら、3 ～ 5 の大きな影響を与えるセグメント（ロイヤルティ層やライフサイクルステージなど）から始める。 測定されたパフォーマンス上昇率に基づいて精度を拡大 意思決定（オプション B）を使用すると、コンテンツ管理を比例的に増やすことなく、精度を拡張できます。

>[!NOTE]
>**トレードオフ：リアルタイム判定と決定論的コンテンツ制御の比較**
>
>意思決定ベースのパーソナライゼーション（オプション B）は、動的な最適化を提供しますが、各訪問者に表示されるコンテンツを直接制御することを減らします。 セグメントベースのパーソナライゼーション（オプション A）は、完全な決定論的制御を提供しますが、動的な最適化は制限されます。
>
>- **決定のメリット：** スケーラビリティ、自動最適化、複雑な実施要件シナリオ
>- **セグメントベースのお気に入り：** 予測可能性、コンプライアンス管理、関係者の透明性
>- **推奨事項：** 厳密な管理が必要な場合に、セグメントベースのパーソナライゼーションをコンプライアンスに敏感なコンテンツ（規制に関するメッセージ、階層固有の価格）に使用する。 動的な最適化によって価値が高まる場合、プロモーションコンテンツ、オファーおよび Recommendations に決定を使用します。

>[!NOTE]
>**トレードオフ：Edgeのデータの可用性とパーソナライゼーションの豊富さ**
>
>Edgeで評価されるパーソナライゼーションは、エッジプロファイルストアで使用できる属性に限定されます。 より豊富なパーソナライゼーションシグナル（完全な行動履歴、複雑な計算属性）には、サーバー側の検索または事前計算が必要になる場合があります。
>
>- **Edgeのみのお気に入り：** 秒未満の遅延、最もシンプルなアーキテクチャ
>- **事前計算済みのエンリッチメントのお勧め：** より豊富なパーソナライゼーションシグナル、より洗練されたオーディエンス定義
>- **推奨事項：** 計算済み属性を使用して、リッチな行動シグナルをエッジで使用可能なプロファイル属性に事前に集計します。 これにより、行動データの豊富さがエッジ評価の速度で提供されます。
