# Segment Match によるオーディエンスの共同作業

このガイドでは、[!DNL Real-Time CDP] および [!DNL Adobe Experience Platform] で [!DNL Segment Match] を使用したオーディエンスの共同作業のための包括的な実装リファレンスを提供します。 これは、プライバシーに配慮した方法で、サンドボックスや組織間でオーディエンスセグメントを共有および照合する必要があるソリューションアーキテクト、マーケティング技術者および実装エンジニア向けに設計されています。

このプランを使用して利用可能なアプローチを理解し、トレードオフを評価し、[!DNL Adobe Experience League] のプランに移動して詳細な設定手順を確認します。

[!DNL Segment Match] を使用すると、基になる PII を公開せずにセグメントメンバーシップ情報を共有することで、複数の [!DNL Experience Platform] 組織（または組織内のサンドボックス）でオーディエンスデータに関する共同作業を行うことができます。 参加者は、重複の推定、オーディエンスの共有およびダウンストリーム宛先への一致したプロファイルのアクティブ化を行うことができます。

## ユースケースの概要

組織は、厳格なプライバシー管理を維持しながら、パートナー、子会社、またはビジネスユニット間でオーディエンスデータを共同作業する必要がますます高まっています。 オーディエンスの共同作業は、[!DNL Segment Match] を通じた安全なセグメント共有を有効にすることで、このニーズに応えます。[!DNL Real-Time CDP] の機能では、ハッシュ化されたプライバシーセーフの識別子を使用して、2 つ以上の [!DNL Experience Platform] 組織（またはサンドボックス）がオーディエンスメンバーシップ情報を交換できます。

ビジネスシナリオには通常、貴重なオーディエンスセグメントを作成し、共同ターゲティング、抑制、エンリッチメントのためにパートナー組織（受信者）と共有したい 1 つの組織（送信者）が関与します。 共有する前に、両方の関係者がオーディエンスの重複を推定して、価値を評価できます。 共有すると、受信側の組織は、一致したオーディエンスを独自の宛先を通じてアクティブ化できます。

このパターンは、外部広告やマーケティングの宛先に対してではなく、組織またはサンドボックス間で動作するので、標準のオーディエンスアクティベーションとは異なります。 また、ID インフラストラクチャを使用してAdobe エコシステム内でネイティブに動作するので、データクリーンルームやサードパーティのコラボレーションプラットフォーム [!DNL Experience Platform] は異なります。

## 主なビジネス目標

このユースケースパターンでは、以下のビジネス目標がサポートされています。

### 新規顧客の獲得

ターゲットを絞った獲得キャンペーン、類似オーディエンス、有料メディア最適化を通じて、顧客基盤を拡大します。 Audience Collaboration を使用すると、組織はセグメントをパートナーオーディエンスと照合し、価値の高い重複を特定し、共同アクティベーションを通じて新規顧客にリーチすることで、新しい見込み客プールを見つけることができます。

- **KPI：新規顧客** 顧客獲得コスト、見込み客/リードのコンバージョン
- [新規顧客の獲得](/help/blueprints/business-objectives/acquisition-growth/acquire-new-customers.md)

### 顧客獲得コストの削減

ターゲティング効率の向上、獲得キャンペーンからの既存顧客の抑制、メディア支出の最適化を実現します。 抑制セグメントを組織やビジネスユニット間で共有することで、チームは既にコンバージョンされた顧客に無駄に費やされる費用を回避し、真に新しい見込み客に予算を集中させることができます。

- **KPI：顧客獲得コスト、リードあたりのコスト、効率**
- [顧客獲得コストの削減](/help/blueprints/business-objectives/cost-efficiency/reduce-customer-acquisition-cost.md)

### マーケティング費用と ROI の最適化

ターゲティング、アトリビューション、オーディエンス抑制、予算配分の改善を通じて、マーケティング投資回収率（ROI）を向上させます。[!DNL Segment Match] 組織間のオーディエンス抑制および共同ターゲティングを有効にして、重複を減らし、精度を向上させます。

- **KPI:** コスト削減、顧客獲得コスト、収益増
- [マーケティング費用と ROI の最適化](/help/blueprints/business-objectives/cost-efficiency/optimize-marketing-spend-roi.md)

## 戦術的な使用例

- **パブリッシャーと広告主のオーディエンスマッチング** - ブランドは、価値の高い顧客セグメントをメディアパブリッシャーと共有して、パーソナライズされた広告で一致したユーザーの重複とターゲットを推定し、PII を公開せずにキャンペーンの関連性を向上させます。
- **持株会社内でのクロスブランド抑制** – 親組織の下の複数のブランドが顧客セグメントを共有し、獲得キャンペーンから姉妹ブランドの既存の顧客を抑制し、広告費用の無駄を減らします。
- **小売メディアネットワークのオーディエンス強化** - retailerは、CPG ブランドパートナーと購入ベースのセグメントを共有し、コンバージョン率の高いretailer メディアネットワークの実績のある購入者をターゲットにすることができます。
- **共同マーケティングのパートナーオーディエンス検出** – 競合しない 2 つのブランドが、共同キャンペーンを開始する前に、オーディエンスの重複を評価して、パートナーシップの可能性を評価しています。重複見積もりを使用して、オーディエンスの整合性を検証しています。
- **データ協同組合によるセグメント共有** - データ協同組合の組織は、ハッシュ化されたオーディエンスセグメントを共有して、プライバシーのコンプライアンスとデータガバナンスの制御を維持しながら、ターゲティングのリーチを拡大します。
- **マルチサンドボックスオーディエンスのフェデレーション** - グローバル企業は、地域のサンドボックスをまたいでオーディエンスセグメントを共有し、地域のデータ常駐要件を尊重しながら、市場全体で一貫した顧客ターゲティングを可能にします。
- **ロイヤルティプログラムのパートナー間アクティベーション** - ロイヤルティ連合は、参加するマーチャントとロイヤルティ層のセグメントを共有して、各パートナーが共有の顧客ベースに層に適したプロモーションを提供できるようにします。
- **測定とアトリビューションコラボレーション** – 広告主は、メディアパートナーとコンバージョンセグメントを共有しているので、パートナーは、公開されたユーザーをコンバーターと照合して、キャンペーンの有効性を測定できます。

## 前提条件

- 送信者組織と受信者組織の両方に [!DNL Segment Match] の使用権限 [!DNL Real-Time CDP] プロビジョニングされています
- [!DNL Segment Match] は、共同作業に参加している組織またはサンドボックスに対して有効になっています
- [!DNL Segment Match] UI の送信者組織と受信者組織の間にパートナー接続が確立されました
- 両方の組織が互換性のある ID 名前空間を使用している（例：両方にハッシュ化されたメールが設定されている）
- Source オーディエンスが定義され、ゼロ以外の母集団で評価されています
- データガバナンスポリシーが設定され、データ使用ラベルが関連するデータセットとフィールドに適用されます
- 両側のユーザーには、[!DNL Segment Match] 接続、共有、フィードを管理するために必要な権限があります
- 同意フィールドは、共有前に同意ベースのフィルタリングを有効にするためにプロファイルに設定されます

## 実装オプション

次のオプションでは、[!DNL Segment Match] とのオーディエンスの共同作業を実装する様々なアプローチについて説明します。 組織の構造とコラボレーションの要件に最適なオプションを選択します。

### オプション A：直接セグメント共有（1 対 1）

このオプションは、広告主と出版社などの 2 つの特定の組織間の二国間パートナーシップ、または共同マーケティングの取り決めにおける 2 つのブランドに最適です。

**仕組み：**

1 対 1 の直接セグメント共有では、送信者組織が 1 つ以上の評価オーディエンスを選択し、特定のパートナー組織との共有を開始し、受信者がその共有を受け入れます。 重複の推定は自動的に実行され、共有が完了する前に、一致したプロファイルの割合とボリュームが両方の関係者に表示されます。

送信者は、照合に使用する ID 名前空間を定義し、共有するセグメントを選択します。 受信者は、受信した共有を確認して受け入れ、一致したオーディエンスをオーディエンスリストで使用して、ダウンストリームのアクティベーションを行えるようになります。 ハッシュ化された ID の重複のみが交換されます。基になる PII やプロファイル属性データが組織の境界を越えることはありません。

このアプローチは簡単で、両方の関係者が完全に制御できます。 送信者は何を共有し、誰と共有するかを正確に選択し、受信者は各共有を受け入れるか拒否するかを選択できます。

**主な考慮事項：**

- 2 つの組織間で明示的なパートナー接続を設定する必要があります
- 一致させるには、両方の組織が ID 名前空間で合意する必要があります
- 重複の推定は、コミットメント前の透明性を提供する
- 各共有を個別に管理および監視する必要があります

**メリット：**

- シンプルで理解が深い二国間ワークフロー
- 共有前の重複推定による完全な透明性
- 詳細な制御 – 送信者は、共有するセグメントを正確に選択します。
- プライバシーセーフ – ハッシュ化された識別子のみが照合に使用されます
- 受信者は、共有を選択的に承認または拒否できます

**制限事項：**

- 多数のパートナーと同時にコラボレーションする場合に効率的に拡張できない
- パートナーシップごとに個別の接続セットアップと管理が必要
- 新しいセグメントごとに共有設定を繰り返す必要があります

- [Segment Match のトラブルシューティング](https://experienceleague.adobe.com/ja/docs/experience-platform/segmentation/ui/segment-match/troubleshooting)

### オプション B：複数のパートナーによるセグメント配分（1 対多）

このオプションは、複数のブランド広告主と購入ベースのセグメントを共有する小売メディアネットワークや、子会社ブランドにセグメントを配信する持株会社など、複数のパートナーと同時にセグメントを共有する必要がある組織に最適です。

**仕組み：**

1 対多の配布モデルでは、送信者組織は複数のパートナー組織と [!DNL Segment Match] の接続を確立し、同じセグメントまたは異なるセグメントをそれぞれ共有します。 送信者は、パートナー接続のポートフォリオを管理し、関係とユースケースに基づいて、様々なオーディエンスセグメントを様々なパートナーと選択的に共有できます。

各パートナー接続は独立して動作します。重複の推定、共有の承認、アクティブ化は、パートナーごとに管理されます。 送信者は、各パートナーが受け取るセグメントを制御でき、差別化されたコラボレーション戦略を可能にします（例えば、プレミアムパートナーはより詳細なセグメントを受け取り、標準パートナーはより広範なセグメントを受け取る）。

このアプローチでは、オプション A と同じ基盤となる [!DNL Segment Match] メカニズムを使用しますが、複数の同時パートナーシップを管理するための運用フレームワークを使用して、これを大規模に適用します。

**主な考慮事項：**

- 複数のパートナーシップを管理するための堅牢な運用プロセスが必要
- ガバナンスポリシーでは、同じセグメントを複数の外部パーティと共有することを考慮する必要があります
- 各パートナーは、異なる ID 名前空間を使用できるので、柔軟な設定が必要になる場合があります
- 重複率はパートナーによって異なり、パートナーごとの評価が必要

**メリット：**

- パートナーのエコシステム全体でオーディエンスコラボレーションを拡張
- パートナーごとに差別化された共有戦略
- 1 つの組織からのすべてのアウトバウンドセグメント共有の一元管理
- 各パートナーシップでは、独立したガバナンスと同意制御を維持しています

**制限事項：**

- パートナーが増えるたびに運用が複雑になる
- 監視とトラブルシューティングは、パートナーごとに行う必要があります
- 新しい各パートナーとの連携にガバナンスレビューが必要
- パートナーは、お互いのデータや共有ステータスを表示しません

**Experience League:**

- [Segment Match の概要](https://experienceleague.adobe.com/ja/docs/experience-platform/segmentation/ui/segment-match/overview)

### オプション C：クロスサンドボックスオーディエンスのフェデレーション

このオプションは、生データを移動せずに内部境界を越えてオーディエンスセグメントを共有する必要がある複数の [!DNL Experience Platform] サンドボックス（地域サンドボックス、ブランド固有のサンドボックス、環境固有のサンドボックスなど）を持つ大企業に最適です。

**仕組み：**

クロスサンドボックスオーディエンスフェデレーションでは、別々の組織間で共有するのではなく、[!DNL Segment Match] を使用して同じ組織内のサンドボックス間でオーディエンスセグメントを共有します。 これにより、グローバルマーケティングチームは、中央のサンドボックスでセグメントを作成して、地域のサンドボックスと共有したり、ブランド固有のサンドボックスで抑制リストを相互に共有したりできます。

ワークフローは、直接セグメント共有（オプション A）を反映していますが、組織境界内で動作します。 サンドボックスからサンドボックスへの接続は [!DNL Segment Match] を通じて確立され、セグメントは、同じプライバシーセーフの一致プロセスを使用して共有されます。 受信側のサンドボックスは、一致したオーディエンスを、独自にローカルに設定された宛先を通じてアクティブ化できる新しいオーディエンスとして取得します。

このアプローチは、データの常駐に関する要件により、未加工の顧客データを地域間で移動できないが、オーディエンスレベルのコラボレーションが許可されている場合に特に役立ちます。

**主な考慮事項：**

- クロスサンドボックス共有 [!DNL Segment Match] サポートする使用権限が必要です
- ID 名前空間は、サンドボックス間で一貫している必要があります
- 各サンドボックスの結合ポリシーは、プロファイルの解決が異なり、マッチ率に影響を与える可能性があります
- ガバナンスポリシーは、サンドボックスごとに個別に適用されます

**メリット：**

- サンドボックスの境界を越えて生データを移動することなく、オーディエンスの共同作業を可能にします
- データの常駐および地域のコンプライアンス要件をサポート
- 既存の組織 ID インフラストラクチャを活用
- 共有が同じ組織内で行われるので、ガバナンスレビューが容易になります

**制限事項：**

- サンドボックス間で一貫した ID 名前空間設定が必要
- 一致率は、サンドボックス間の結合ポリシーの一貫性に依存します
- 組織間コラボレーションのニーズに対応していない
- スキーマと設定を同期するために、サンドボックスツールが必要になる場合があります

**Experience League:**

- [Segment Match の概要](https://experienceleague.adobe.com/ja/docs/experience-platform/segmentation/ui/segment-match/overview)
- [サンドボックスの概要](https://experienceleague.adobe.com/ja/docs/experience-platform/sandbox/home)

### 適切なオプションを選択

次の決定ガイダンスを使用して、適切な実装方法を選択してください。

1. **外部組織と共同作業していますか、それとも組織内で共同作業していますか？**
   - 外部組織：質問 2 に進みます。
   - 独自の組織内（サンドボックスをまたぐ）:**オプション C** （クロスサンドボックス連携）」を選択します。

2. **共同作業を行う外部パートナーは何人ですか？**
   - 1 つのパートナー：**オプション A** （直接セグメント共有）を選択します。
   - 複数のパートナー：**オプション B** （マルチパートナー配布）を選択します。

3. **地域をまたいで生データを移動できないようにするデータ常駐サービスの制約はありますか？**
   - はい：パートナーが社内か社外かに関係なく、**オプション C** を選択します。クロスサンドボックス共有を使用して、データのローカル性を維持します。

## 実装フェーズ

### フェーズ 1：セグメントの選択と準備

**アプリケーション関数：** [!DNL Real-Time CDP]：オーディエンス評価、[!DNL Real-Time CDP]：オーディエンス構成

このフェーズでは、[!DNL Segment Match] を通じて共有されるオーディエンスセグメントの定義と評価を行います。 ソースセグメントを共有対象として選択する前に、ゼロ以外の母集団を使用してソースセグメントを完全に評価する必要があります。 このフェーズでは、共有前にセグメントを絞り込むためのオプションのオーディエンス構成も扱います。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：オーディエンス定義のアプローチ**
>
>共有のソースオーディエンスはどのように作成する必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| セグメントビルダー（セグメントルール） | プロファイル属性、イベントまたはセグメントメンバーシップに基づく標準オーディエンス定義 | バッチ、ストリーミング、エッジの評価をサポート。条件の定義に最も柔軟に対応 |
>| オーディエンス構成 | 既存のセグメントに対するランク、分割、除外、エンリッチメントの操作を必要とする派生オーディエンス | バッチ評価のみをサポートします。サンドボックスあたり 10 個のコンポジションキャンバスに制限されています |
>| Federated Audience の構成 | データを [!DNL Experience Platform] に取り込まずに外部 Data Warehouse クエリから作成されたオーディエンス | [!DNL Federated Audience Composition] の使用権限が必要。データはウェアハウスに残る |

>[!NOTE]
>
>**決定：オーディエンス評価方法**

>[!DNL Segment Match] にはバッチ評価されたオーディエンスが必要です。 評価はどのようにスケジュールする必要がありますか？

>| カスタムスケジュール | パートナーのアクティベーションウィンドウに合わせた特定のタイミング要件 | カスタム cron スケジュールの設定（より複雑で正確） |

**UI ナビゲーション：** 顧客/オーディエンス/オーディエンスを作成/ルールを作成（[!DNL Segment Builder]）またはオーディエンスを作成（[!DNL Audience Composition]）

**キー設定の詳細：**

- プロファイル属性、行動イベント、セグメントメンバーシップを使用してオーディエンス条件を定義します
- オーディエンスが結合ポリシーを使用しており、[!DNL Segment Match] に使用される ID 名前空間と互換性があることを確認します
- 評価後にオーディエンス母集団がゼロ以外であることを確認

**オプション A （直接セグメント共有）の場合：**
単一のパートナーと共有する特定のセグメントを準備します。 量よりも品質に重点を置く – パートナーシップに明確な価値を提供するセグメントをキュレーションします。

**オプション B （マルチパートナー配布）の場合：**
様々なパートナーと共有できるセグメントのポートフォリオを準備します。 パートナーごとに異なるオーディエンス定義が必要な場合は、パートナー固有のセグメントの作成を検討します。 パートナーシップをまたいでセグメントを管理するには、一貫した命名規則を使用します。

**オプション C （クロスサンドボックス連携）の場合：**
送信側サンドボックスのソースオーディエンスで、受信側サンドボックスにある ID 名前空間が使用されていることを確認します。 結合ポリシーがサンドボックス間で揃っていることを確認します。

### フェーズ 2：マッチングとガバナンスの設定

**アプリケーション機能：** [!DNL Real-Time CDP]：同意とガバナンスの適用

このフェーズでは、組織またはサンドボックス間の [!DNL Segment Match] 接続を確立し、照合に使用される ID 名前空間を設定し、データガバナンスポリシーによる共有の許可を確保します。 ガバナンスの実施は、セグメントデータを共有する前にクリアする必要があるポリシーゲートとして機能します。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：一致させる ID 名前空間**
>
>送信者と受信者の間でプロファイルを照合するために使用される ID 名前空間はどれですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| ハッシュ化されたメール（SHA-256） | どちらの組織もメールアドレスを収集し、一貫してハッシュ化できます | 最も一般的な一致キー：消費者のユースケースでの高い一致率。両側で同じハッシュアルゴリズムを使用する必要があります。 |
>| ハッシュ化された電話番号 | メールは一貫して利用できるわけではありませんが、電話番号は次のとおりです | 多くの市場で電子メールよりも対象範囲が狭くなります。モバイルファーストオーディエンスに役立ちます。 |
>| カスタム名前空間（ハッシュ化されたロイヤルティ ID など） | 組織は、共通のロイヤルティ ID またはメンバーシッププログラム ID を共有します | 共有済みの顧客ベースに対する最高の一致率。既存の共有 ID インフラストラクチャが必要 |
>| 複数の名前空間 | マッチ率を最大限に高めることは重要で、どちらの組織も複数の一貫性のある識別子を持っています | マッチ率は向上するが、複雑さが増す。各名前空間を個別に設定する必要がある |

>[!NOTE]
>
>**決定：データガバナンスのレビュー**
>
>共有する前に完了する必要があるガバナンスチェックは何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 標準ガバナンス評価 | 標準データ使用ラベルとポリシーの一般的なユースケース | データセットラベルに対してマーケティングアクション「サードパーティに書き出し」を評価し、違反を解決してから共有する |
>| 同意フィルタリングによるガバナンスの強化 | 同意を明示的に検証する必要がある外部パートナーとの共有 | 同意を共有せずにプロファイルを除外する同意ベースのフィルタリングを追加します（例：consents.share.val = &#39;n&#39;）。より厳格だが安全 |
>| 内部統制の検証 | 同じ組織内でのクロスサンドボックス共有 | データは組織の境界内にとどまり、データ使用ラベルを検証するので、ガバナンス要件を軽減 |

**UI ナビゲーション：** 顧客/オーディエンス/Segment Match/パートナー接続

**キー設定の詳細：**

- 送信者組織と受信者組織の間で接続識別子を交換することで、パートナー接続を確立する
- 両側での照合に使用する ID 名前空間の設定
- セグメント共有に関連付けられたマーケティングアクションに対してデータガバナンスポリシー評価を実行します
- 同意フィールドがプロファイルに入力され、同意を共有しないプロファイルが除外されていることを確認します
- 共有に含まれるデータセットとスキーマフィールドのデータ使用ラベルを確認します

**オプションが分岐する場所：**

**オプション A （直接セグメント共有）の場合：**
単一のパートナー接続を確立します。 特定のパートナーと ID 名前空間を設定します。 ガバナンスレビューでは、二国間関係に焦点を当てる。

**オプション B （マルチパートナー配布）の場合：**
複数のパートナー接続を確立および管理します。 各パートナーは、個別のガバナンスレビューを必要とする場合があります。 各パートナーシップのガバナンス承認を文書化します。 パートナーのオンボーディングを効率化するために、ガバナンスチェックリストの作成を検討します。

**オプション C （クロスサンドボックス連携）の場合：**
組織内でサンドボックス間の接続を確立します。 共有は内部的に行われるので、通常、ガバナンスはよりシンプルです。 ID 名前空間がサンドボックス間で一貫していることを確認します。

### フェーズ 3：重複の推定

**アプリケーション関数：** [!DNL Real-Time CDP]：オーディエンス評価（重複推定用）

このフェーズでは、送信者のセグメントと受信者のプロファイルベースの間の重複推定が実行されます。 重複の推定では、完全なセグメント共有をコミットする前に、期待される一致の量と割合を両当事者に提供し、共同作業の価値に関する十分な情報に基づいた意思決定を可能にします。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：処理の重複しきい値**
>
>完全なセグメント共有の進行を正当化する最小重複率は何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 最小しきい値なし | 探索的パートナーシップ、または重複が価値を提供する場合 | 関係をテストする最初の共同作業に適しています |
>| 低しきい値（1～5%） | わずかな重複でも大量のデータが得られる大規模なオーディエンスの共同作業 | 大規模なオーディエンスベースを持つパブリッシャーと広告主の関係で一般的 |
>| Mediumしきい値（5～20%） | 有意義な重複が予想される標準的なパートナーシップ | 共同マーケティングまたは同じ業界のコラボレーションに典型的 |
>| 高しきい値（20%+） | 強力なオーディエンスの連携が前提条件となるパートナーシップ | ロイヤルティ提携または緊密に統合されたブランドパートナーシップに共通 |

**UI ナビゲーション：** 顧客/ オーディエンス / セグメントの一致/共有/見積りの重複

**キー設定の詳細：**

- 重複推定に含めるセグメントを選択
- 一致したプロファイルの数と割合を示す重複レポートを確認します
- 承認を得るために、双方の関係者と重複推定結果を共有する
- コラボレーションの有効性を測定するためのベースラインとしての重複指標のドキュメント化
- 重複が許容可能なしきい値を下回る場合は、続行する前に、セグメント定義または ID 照合の設定を調整することを検討してください

**Experience League ドキュメント：**

- [Segment Match の概要](https://experienceleague.adobe.com/ja/docs/experience-platform/segmentation/ui/segment-match/overview)

### フェーズ 4：オーディエンスの共有

**アプリケーション関数：** [!DNL Real-Time CDP]：オーディエンス評価（共有実行用）

このフェーズでは、送信者から受信者への実際のセグメント共有が実行されます。 送信者が選択したセグメントの共有を開始し、受信者が受信した共有を受け入れます。 承認されると、一致したオーディエンスが、ダウンストリームのアクティベーションで使用可能な新しいオーディエンスとして受信者のオーディエンスリストに表示されます。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：方向性の共有**
>
>この共同作業の共有モデルはどのようなものですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 一方向の共有（送信者から受信者） | 1 つのパーティのみがオーディエンスデータを提供する非対称パートナーシップ | 最も単純なモデル。送信者の共有、受信者のアクティブ化。広告主と発行者の関係で共通 |
>| 双方向の共有 | 両方の関係者がオーディエンスを相互に共有することでメリットが得られます | 両方の組織が送信者と受信者として同時に機能します。2 つの共有設定が必要です。共通の共有はマーケティング提携に含まれます |

>[!NOTE]

>**決定：共有更新頻度**

>| 繰り返し共有（バッチ評価と連携） | オーディエンスメンバーシップが変化し、最新の状態に保つ必要がある継続的なパートナーシップ | 更新ステータスの監視が必要です。本番環境のコラボレーションで最も一般的です。 |

**UI ナビゲーション：** 顧客/オーディエンス/Segment Match/共有/共有を作成（送信者）または共有を承認（受信者）

**キー設定の詳細：**

- 送信者が共有するセグメントを選択し、設定済みパートナーとの共有を開始します
- 受信者は、受信した共有の詳細（セグメント名、推定サイズ、使用されている ID 名前空間）を確認します。
- 受信者は共有を承認して、一致したオーディエンスをサンドボックスに作成します
- 一致したオーディエンスが、受信者のオーディエンスリストに期待される母集団で表示されることを確認します。
- 一致したオーディエンスに、ガバナンストラッキングに適したラベルが付けられていることを確認します

**オプションが分岐する場所：**

**オプション A （直接セグメント共有）の場合：**
パートナーと 1 回の共有を実行します。 共有ステータスを監視し、受信側で一致したオーディエンスを検証します。

**オプション B （マルチパートナー配布）の場合：**
各パートナーに対して個別に共有を実行します。 すべてのパートナーシップで共有ステータスを追跡します。 処理の負荷を管理するには、共有の開始を分散させることを検討してください。

**オプション C （クロスサンドボックス連携）の場合：**
クロスサンドボックス共有を実行します。 一致したオーディエンスは、受信側のサンドボックスのオーディエンスリストに表示されます。 受信側サンドボックスに、ダウンストリームアクティベーションに必要な宛先設定があることを確認します。

### フェーズ 5：マッチしたオーディエンスのアクティブ化

**Application function:** [!DNL Real-Time CDP]: Destination Configuration, [!DNL Real-Time CDP]: Audience Activation

このフェーズでは、ターゲティング、抑制またはダウンストリームで使用するために、一致したオーディエンス（受信側）を外部の宛先に対してアクティブ化します。 一致したオーディエンスは、受信者のサンドボックス内の他のオーディエンスと同様に扱われ、標準の宛先アクティベーションワークフローでアクティベートできます。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：一致したオーディエンスの宛先タイプ**
>
>マッチしたオーディエンスをどこでアクティベートするか。
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| Advertisingの宛先（Google、Meta、Trade Desk） | 広告のターゲティングまたは抑制のための、マッチしたオーディエンスの使用 | 宛先固有のレート制限と形式要件に従い、宛先接続と認証が必要です。 |
>| クラウドストレージの宛先（S3、Azure、GCS） | 外部システムで使用するために、一致したオーディエンスをファイルとして書き出す | ファイル形式のカスタマイズをサポート。バッチ書き出しのスケジュールが必要。ダウンストリーム処理に柔軟に対応 |
>| CRM/マーケティング自動化宛先 | CRM レコードを強化したり、オーディエンスデータが一致する自動マーケティングワークフローをトリガーしたりします | CRM スキーマへのフィールドマッピングが必要です。セールスとマーケティングの連携に役立ちます |
>| Personalizationの宛先（web、アプリ） | オンサイトパーソナライゼーションのための、マッチしたオーディエンスメンバーシップの使用 | 一致したオーディエンスまたはストリーミングアクティベーションのエッジ評価が必要です。待ち時間は宛先によって異なります |

>[!NOTE]
>
>**決定：アクティベーションスケジュール**

>一致したオーディエンスをどのくらいの頻度で宛先に書き出す必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |

>| 毎日の完全書き出し | 毎回完全なオーディエンスファイルを必要とする宛先 | データ量が多い。宛先のオーディエンスの状態が完全であることを確認する。一部の宛先には完全な書き出しが必要 |
>| オンデマンドアクティベーション | アドホックキャンペーンのローンチまたは時間依存アクティベーション | 手動トリガー：スケジュールされたケイデンスをバイパスします。バッチ宛先でのみ使用できます。 |

**UI ナビゲーション：** 接続/宛先/カタログ（宛先設定用）または参照/宛先を選択/オーディエンスをアクティブ化（アクティブ化用）

**キー設定の詳細：**

- 適切な認証資格情報を使用した宛先接続の設定

## 実装に関する考慮事項

実装前と実装中に次の考慮事項を確認して、一般的な問題を回避し、オーディエンスの共同作業を最適化します。

### ガードレールと制限

- [!DNL Segment Match] は、照合にハッシュ化された識別子を使用します。組織の境界を越える PII はありません。 [Segment Match の概要 &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/segmentation/ui/segment-match/overview) を参照してください。
- [!DNL Segment Match] では、バッチ評価されたオーディエンスのみを共有できます。 ストリーミングセグメントとエッジ評価されたセグメントは、共有する前にバッチ評価に変換する必要があります。
- サンドボックスあたり最大 4,000 個のセグメント定義が、ソースと受信した両方のセグメントに適用されます。 [&#x200B; セグメント化ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/guardrails) を参照してください。
- 重複の推定精度は、一致した識別子の量によって異なります。 オーディエンスが小さい場合、正確な見積もりが表示されない場合があります。
- アクティベーションガードレールは、他のオーディエンスと同じように、一致したオーディエンスに適用されます（宛先あたり最大 100 個のデータフロー）。 [&#x200B; アクティベーションガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/destinations/guardrails) を参照してください。
- 構成されたオーディエンスはバッチスケジュールで評価され、サンドボックスあたり 10 個のコンポジションキャンバスに制限されます。 [&#x200B; オーディエンス構成ガードレール &#x200B;](https://experienceleague.adobe.com/en/docs/experience-platform/segmentation/guardrails) を参照してください。

### よくある落とし穴

- **送信者と受信者の間の ID ハッシュの不一致：** 両方の組織でメールアドレスをハッシュ化するが、異なるハッシュアルゴリズム、正規化ルールまたは salt 値を使用している場合、一致率はほぼゼロになります。 両者は、接続を確立する前に、正確なハッシュ仕様（小文字のメールの SHA-256 など）に同意する必要があります。
- **ガバナンスレビューを行わずにオーディエンスを共有：** データ使用ポリシーを評価せずにセグメント共有を開始すると、コンプライアンス違反につながる可能性があります。 外部組織とセグメントを共有する前に、「サードパーティに書き出し」マーケティングアクションに対してガバナンスポリシー評価を常に実行してください。
- **ID カバレッジギャップによる一致率の低下：** 送信者のオーディエンスが主に ECID （匿名 cookie）によって識別されているが、一致する名前空間がハッシュ化されたメールである場合、匿名プロファイルにはメールアドレスがないので、一致率は非常に低くなります。 ソースオーディエンスが、設定された一致する ID 名前空間を十分にカバーしていることを確認します。
- **受信者側での共有の受け入れを忘れる：** 共有が明示的に受け入れられるまで、共有オーディエンスは受信者のオーディエンスリストに表示されません。 受信者と調整して、特に時間に敏感なキャンペーンでタイムリーに受け入れるようにします。
- **評価スケジュールのミスアライメントによる一致したオーディエンスの更新の停止：** 送信者のソースオーディエンスが毎日評価され、[!DNL Segment Match] の更新が毎週実行される場合、受信側の一致したオーディエンスは最新のメンバーシップを反映していない可能性があります。 評価を調整し、更新サイクルを共有します。

### ベストプラクティス

- [!DNL Segment Match] を設定する前に、組織間で正式なデータ共有契約を確立します。 これには、許可されたユースケース、データガバナンス要件、同意義務、終了手順が含まれます。
- すべての主要なキャンペーンの前に重複推定を検証ツールとして使用 – コミットする前に推定を実行して、一致したオーディエンスが最小サイズおよび品質のしきい値を満たしていることを確認します。
- パートナー名、ユースケース、日付（例：「PartnerX_HighValue_Suppression_2026Q1」）を含む共有セグメントに説明的な命名規則を適用し、組織間の明確さを維持します。
- マッチ率の推移を監視します。 一致率の低下は、ID カバレッジの低下、データ品質の問題、パートナーの顧客ベースの変更を示す場合があります。
- 一致する ID 名前空間が入力されていないプロファイルを除外するようにソースオーディエンスをセグメント化します。 これにより、一致率の割合が向上し、より正確な重複予測が提供されます。
- 双方向共有パートナーシップの場合は、更新の責任がある組織に関する混乱を避けるために、セグメントのメンテナンスと更新スケジュールの明確な所有権を指定します。

### トレードオフの決定

>[!NOTE]
>
>**トレードオフ：マッチ率とプライバシー制御**
>
>照合に多くの ID 名前空間（ハッシュ化されたメール、ハッシュ化された電話、デバイス ID）を使用すると、一致率は向上しますが、再識別の可能性がある表面積は広くなります。 名前空間を減らす（ハッシュ化されたメールのみ）と、プライバシー保護を強化できますが、一致するオーディエンスサイズが減少する可能性があります。
>
>- **複数の名前空間のメリット：** マッチ率が高く、マッチしたオーディエンスが多いほど、キャンペーンターゲティングに役立ちます
>- **単一の名前空間のメリット：** プライバシー体制の強化、ガバナンスの検証の簡素化、コンプライアンスリスクの軽減
>- **推奨事項：** 単一の名前空間（ハッシュ化されたメールが最も一般的です）から開始して、使用例にマッチ率が不十分な場合にのみ追加の名前空間を追加します。 追加した各名前空間のプライバシー影響評価を文書化します。

>[!NOTE]
>
>**トレードオフ：セグメントの精度と運用のシンプルさ**
>
>詳細にターゲティングされた多数のセグメントをパートナーと共有すると、キャンペーンターゲティングの柔軟性が向上しますが、送信者と受信者の両方の操作が複雑になります。 より少ない広範なセグメントを共有することで、管理が簡素化されますが、ターゲティングの精度が低下します。
>
>- **詳細なセグメントの活用：** 正確なターゲティング、キャンペーンの差別化、関連度の向上
>- **幅広いセグメントの優位点：** 管理のシンプル化、監視するシェアの減少、運用オーバーヘッドの削減
>- **推奨事項：** 新しいパートナーシップのために、少数の高価値セグメント（2～5）から始めます。 パートナーシップが成熟し、運用プロセスが確立されると、精度が向上します。 セグメント数の増加に伴う複雑さを管理するために、命名規則とドキュメントを使用します。

>[!NOTE]
>
>**トレードオフ：更新頻度と処理コストの比較**
>
>共有オーディエンスを頻繁に更新すると、一致したオーディエンスが最新の状態に保たれますが、処理負荷が増加し、ライセンス容量が多く消費される場合があります。 更新頻度の低さはコストを削減しますが、一致したオーディエンスが古くなるのを防ぎます。
>
>- **頻繁な更新のお気に入り：** 現在のオーディエンスメンバーシップ、キャンペーンの関連性の高さ、抑制の精度の向上
>- **更新頻度の低さが好ましい：** 処理コストの削減、監視の合理化、ライセンス消費の削減
>- **推奨事項：** 毎日の更新は、ほとんどの実稼動環境の共同作業に適しています。 フラッシュ販売やイベントベースのキャンペーンなど、時間に敏感なユースケースの場合は、キャンペーンが開始される直前にオンデマンドでの再評価と共有を検討してください。
