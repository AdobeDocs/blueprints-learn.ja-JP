# B2B オーディエンスのアクティベーション

このガイドでは、[!DNL Adobe Real-Time Customer Data Platform] （[!DNL RT-CDP]）B2B editionを使用してアカウントベースの B2B オーディエンスをアクティブ化するための包括的な実装リファレンスを提供します。 Web、メール、広告、CRM の各チャネルにわたってアカウントレベルのオーディエンスを構築、評価、アクティベートする必要があるソリューションアーキテクト、マーケティングテクノロジスト、実装エンジニアを対象としています。

アカウントプロファイルの統合から、オーディエンスの評価とアクティベーション、[!DNL Marketo Engage]、[!DNL LinkedIn]、CRM システムなどの B2B 固有の宛先まで、ライフサイクル全体をカバーしています。 すべての実行可能な実装アプローチには、組織に適したパスを選択するのに役立つ、トレードオフと決定ガイダンスが提示されます。

## ユースケースの概要

B2B マーケティングチームは、オーディエンスをターゲットにして、個人レベルではなくアカウントレベルでアクティブ化する必要があります。 ターゲティングの単位が単一の消費者プロファイルである B2C オーディエンスアクティベーションとは異なり、B2B オーディエンスアクティベーションには、人物と彼らが属するアカウントの関係を理解し、人物レベルのエンゲージメントシグナルと組み合わされたアカウントレベル属性に基づいてオーディエンスメンバーシップを評価し、アカウントベースのターゲティングをサポートする宛先にそれらのオーディエンスを配信する必要があります。

[!DNL RT-CDP] B2B editionは、アカウント、オポチュニティ、キャンペーン用の専用の XDM クラスと、人物とアカウントの関係をマッピングする B2B ID 解決を使用して、標準 [!DNL Real-Time Customer Data Platform] ールを拡張します。 これによりマーケターは、個人データ（業界、売上高、従業員数）、テクニカルデータ（テクノロジースタック、製品使用状況）、および行動データ（web 訪問、メールエンゲージメント、イベント出席）を、これらのアカウントに関連付けられた人物から統合して、アカウントオーディエンスを作成できます。

アクティブ化されたアカウントオーディエンスは、[!DNL LinkedIn] やディスプレイ広告に対するfunnel上位層の認知度キャンペーン、[!DNL Marketo Engage] でのミッドfunnel育成プログラム、CRM 統合によるfunnel下位層のセールス啓発など、需要創出funnel全体のユースケースを強化します。 アカウント抑制オーディエンスは、既存の顧客、クローズして失われたアカウント、アクティブな販売サイクルに既に入っているアカウントを除外することで、無駄な支出を防ぎます。

## 主なビジネス目標

このユースケースパターンでは、以下のビジネス目標がサポートされています。

### リードジェネレーションの向上

フォーム、イベント、コンテンツ、マルチチャネルエンゲージメントを通じて、セールスパイプラインでより適格なリードを生成します。

**KPI：見込み客**、リードあたりのコスト、リードコンバージョン

[リードジェネレーションの向上についての詳細情報](/help/blueprints/business-objectives/acquisition-growth/increase-lead-generation.md)

### リードの選定とコンバージョンの向上

スコアリング、育成、パーソナライズされたフォローアップにより、リードの品質を高め、パイプラインの進行を加速させます。

**KPI:** リードコンバージョン、見込み客/リードコンバージョン、効率

[リードの選定とコンバージョンの向上に関する詳細情報](/help/blueprints/business-objectives/qualification-sales-b2b/improve-lead-qualification-conversion.md)

### 新規顧客の獲得

ターゲットを絞った獲得キャンペーン、類似オーディエンス、有料メディア最適化を通じて、顧客基盤を拡大します。

**KPI：新規顧客** 顧客獲得コスト、見込み客/リードのコンバージョン

[新規顧客の獲得に関する詳細情報](/help/blueprints/business-objectives/acquisition-growth/acquire-new-customers.md)

### マーケティング費用と ROI の最適化

ターゲティング、アトリビューション、オーディエンス抑制、予算配分の改善を通じて、マーケティング投資回収率（ROI）を向上させます。

**KPI:** コスト削減、顧客獲得コスト、収益増

## 戦術的な使用例

次のシナリオは、このパターンを実際にどのように適用できるかを示しています。

- **[!DNL LinkedIn]** でのアカウントベースの広告 – B2B editionからアクティブ化されたアカウントリストを使用して、[!DNL LinkedIn] でのスポンサー付きコンテンツおよび InMail キャンペーンで、理想的な顧客プロファイル（ICP）に一致す [!DNL RT-CDP] アカウントをターゲットにします
- **[!DNL Marketo Engage]育成プログラムのターゲティング** - アカウントオーディエンスをアクティブ化して、アカウントレベルの選定条件に基づいて、関連するリードおよび連絡先をターゲット育成ストリームに登録で [!DNL Marketo Engage] るようにします
- **CRM アカウントリストの同期** – 選定されたアカウントリストを、販売チームの表示、テリトリーの割り当て、アウトバウンド予測のワークフローのために [!DNL Salesforce] または [!DNL Microsoft Dynamics] にプッシュします
- **有料メディアのアカウント抑制** – 有料の獲得キャンペーンからのアクティブな販売サイクルの既存の顧客、成約済みのアカウントまたはアカウントを抑制して、無駄な支出を減らします
- **インテントベースのアカウントターゲティング** - サードパーティのインテント信号をアカウントレベルでのファーストパーティエンゲージメントデータと組み合わせて、市場内アカウントのオーディエンスを特定しアクティブ化します
- **既存のアカウントに対する製品のクロスセル** - 1 つの製品ラインのみを使用してアカウントのオーディエンスを作成し、クロスセルキャンペーン用のメールおよび広告チャネルに対してアクティブ化します
- **イベントおよびウェビナーターゲティング** - アカウントオーディエンスを広告およびメールチャネルに対してアクティブ化し、ターゲットアカウントからのイベント登録を促進します
- **競合するディスプレイスメント キャンペーン** – 広告およびメールチャネルを通じてアクティブ化されたカスタマイズされたメッセージを使用した、競合他社製品を使用したターゲットアカウント。
- **階層型アカウントエンゲージメント** – 集計したユーザーレベルのアクティビティに基づいて、アカウントをエンゲージメント層（高、中、低）にセグメント化し、層ごとに差別化されたキャンペーンをアクティブ化します
- **パートナー共同マーケティングオーディエンス** - クラウドストレージの宛先を通じて、チャネルパートナーや共同マーケティングプログラムとアカウントオーディエンスセグメントを共有します

## アプリケーション

次のアプリケーションを使用して、このユースケースパターンを実装します。

- **[!DNL Real-Time CDP]B2B edition** - アカウントプロファイルの統合、B2B ID の解決、アカウントオーディエンスの評価、B2B 固有の宛先設定、アカウントオーディエンスのアクティベーションのためのコアプラットフォーム
- **[!DNL Adobe Experience Platform]（AEP）** - B2B XDM データモデリング、CRM およびマーケティング自動化ソースからのデータ取り込み、ID サービス、ガバナンスの基盤インフラストラクチャ
- **[!DNL Marketo Engage]** — アクティブ化されたアカウントオーディエンスによってフィードされるリード育成プログラム、スコアリングおよびキャンペーン実行のプライマリ B2B マーケティング自動化先

## 実装オプション

次のオプションでは、このユースケースパターンを実装する様々なアプローチを説明します。 各オプションを確認し、要件に最適なものを選択します。

### オプション A:[!DNL Marketo Engage] への Audience Activation のストリーミング

**最適な対象：**&#x200B;[!DNL Marketo Engage] を主要な B2B マーケティング自動処理プラットフォームとして使用している組織では、トリガー育成プログラム、リードスコアの更新、アカウントの選定または選定解除に合わせたキャンペーンメンバーシップの変更など、ほぼリアルタイムのオーディエンスメンバーシップの更新が必要です。

**仕組み：**

このオプションでは、[!DNL RT-CDP] のネイティブ [!DNL Marketo Engage] 宛先コネクタを使用して、アカウントオーディエンスメンバーシップの変更を [!DNL Marketo Engage] に直接ストリーミングします。 アカウントがオーディエンスセグメントに適合するか、オーディエンスセグメントから離脱すると、オーディエンスに関連するリードおよび連絡先 [!DNL Marketo]、セグメントメンバーシップ属性で更新されます。[!DNL Marketo] スマートキャンペーンは、これらのメンバーシップの変更に基づいてトリガーを設定できます。

[!DNL Marketo Engage] の宛先はストリーミング宛先です。つまり、オーディエンスメンバーシップの変更は、スケジュールされたバッチではなく、発生した時点で増分的に送信されます。 これにより、アカウントの選定の変更に対応する必要があるキャンペーンのアクションまでの時間を短縮できます。 フィールドマッピング [!DNL RT-CDP] は、プロファイル属性をリード/連絡先フィールド [!DNL Marketo] 接続し、[!DNL RT-CDP] のアカウントレベルのデータを使用して [!DNL Marketo] レコードをエンリッチメントできます。

**主な考慮事項：**

- API アクセスを持つアクティブな [!DNL Marketo Engage] 購読が必要
- ストリーミングアクティベーションは、完全なオーディエンススナップショットではなく、増分更新を送信します
- [!DNL Marketo] の人物レベルのレコードは、アカウントレベルのレコードではなく更新されます
- セグメ [!DNL Marketo] トメンバーシップフィールドの更新に応じて行動するようにスマートキャンペーンを設定する必要があります。

**メリット：**

- [!DNL Marketo] でのオーディエンスメンバーシップのほぼリアルタイムの更新
- ネイティブの双方向コネクタにより統合が容易
- 増分更新により、データ転送ボリュームを最小限に抑えます
- [!DNL Marketo] ーディエンスの変化に基づいて、ナーチャリングプログラムをすぐにトリガーにすることができます

**制限事項：**

- アクティベーションターゲットとして [!DNL Marketo Engage] に制限
- ストリーミング評価の実施要件制約は、基になるオーディエンス定義に適用されます
- [!DNL RT-CDP] アカウントオーディエンスとリード/連絡先レコード [!DNL Marketo] マッピングが必要です
- ユーザーレベルの更新をアカウントレベルのアクションに変換するには、複雑な [!DNL Marketo] プログラムロジックが必要になる場合があります

**Experience League:**

- [Marketo Engage先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/adobe/marketo-engage)
- [Marketo Engageの宛先に対するオーディエンスのアクティブ化](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/adobe/marketo-engage#activate)

### オプション B：広告プラットフォームに対するオーディエンスのバッチアクティベーション

**以下に最適：** アカウントベースの広告キャンペーン：[!DNL LinkedIn]、ディスプレイネットワーク、その他の広告プラットフォームで、毎日のオーディエンス更新で十分であり、主な目標はリアルタイムの応答ではなく、アカウントレベルのリーチと認知です。

**仕組み：**

このオプションは、アカウントオーディエンスを広告プラットフォームの宛先（[!DNL LinkedIn] Matched Audiences、[!DNL Google] Customer Match、[!DNL Facebook] Custom Audiences または [!DNL The Trade Desk]）に対して、スケジュールされたバッチ頻度でアクティブ化します。 アクティベーションデータフローは、Advertising Platform がユーザーベースとの照合に使用する、マッピングされた ID フィールド（通常、関連する連絡先のハッシュ化されたメールアドレス）を使用して、アカウントオーディエンスメンバーを書き出します。

特に、[!DNL LinkedIn] Matched Audiences 宛先では、メールベースのマッチングに加えて、会社名とドメインベースのマッチングを受け入れ、真のアカウントレベルのターゲティングを可能にしまし [!DNL LinkedIn]。 その他の広告プラットフォームには通常、対象アカウントに関連付けられた連絡先からユーザーレベルの識別子（メール、電話）が必要です。

バッチアクティベーションは、設定可能なスケジュール（毎日、6 時間ごとなど）で実行されます およびは、宛先のタイプに応じて、完全なオーディエンスのスナップショットまたは増分変更を書き出します。 このアプローチは、オーディエンスの鮮度要件が分単位ではなく時間単位で測定される、継続的な認知キャンペーンに適しています。

**主な考慮事項：**

- オーディエンスの鮮度はバッチスケジュール（通常は毎日）によって異なります
- Advertising プラットフォームには、独自の一致率制限があります
- [!DNL LinkedIn] はアカウントレベルのマッチングをサポートしています。他のプラットフォームではユーザーレベルの識別子が必要です。
- 書き出しファイル形式と ID の要件は、宛先によって異なります

**メリット：**

- 複数の広告プラットフォームを同時にサポート
- バッチ処理では、大量のオーディエンスを効率的に処理します
- アカウント抑制オーディエンスにより、無駄な広告費用が削減されます
- [!DNL LinkedIn] Matched Audiences により、ネイティブアカウントレベルのターゲティングが可能になります

**制限事項：**

- オーディエンスの更新はリアルタイムではありません。待ち時間はバッチスケジュールによって異なります
- 一致率は、プラットフォームと使用可能な ID フィールドによって異なります
- 一部のプラットフォームは、真のアカウントレベルのマッチングをサポートしておらず、ユーザーレベルのみをサポートしています
- 広告プラットフォームごとに個別の宛先設定が必要です

**Experience League:**

- [LinkedIn でマッチしたオーディエンスの宛先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/social/linkedin)
- [Google カスタマーマッチの宛先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/advertising/google-customer-match)
- [宛先カタログ](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/overview)

### オプション C：クラウドストレージに対するファイルベースのアクティベーション

**最適な対象：** CRM インポート、データウェアハウスのエンリッチメント、パートナーのデータ共有、ファイルベースの引き渡しが望ましいカスタム統合など、アカウントオーディエンスがダウンストリームで使用される方法に最大限の柔軟性を必要とする組織。

**仕組み：**

このオプションでは、アカウントオーディエンスを CSV、JSON または Parquet ファイルとして、スケジュールに沿ってクラウドストレージの宛先（[!DNL Amazon S3]、[!DNL Azure Blob Storage]、[!DNL Google Cloud Storage]、SFTP）に書き出します。 書き出しには、アカウント属性、関連する連絡先からの人物レベルのフィールド、オーディエンスメンバーシップメタデータが含まれます。 ダウンストリームシステムは、独自の読み込みプロセスを通じてこれらのファイルを使用します。

ファイルベースのアクティベーションは、書き出し形式、フィールドの選択、スケジュールを最も詳細に制御します。 完全な書き出し（各実行ごとにオーディエンススナップショットを完了）と増分書き出し（前回の実行以降の変更のみ）の両方をサポートします。 この方法は、ターゲットシステムにネイティブの [!DNL RT-CDP] 宛先コネクタがない場合や、組織がデータをターゲットシステムに読み込む前に前処理または変換する必要がある場合に一般的に使用されます。

**主な考慮事項：**

- クラウドストレージインフラストラクチャ（[!DNL S3]、[!DNL Azure Blob]、[!DNL GCS]、SFTP）が必要です
- ダウンストリームのインポートプロセスを構築し、維持する必要があります
- ファイル形式（CSV、JSON、Parquet）は、ダウンストリームシステムの要件に一致する必要があります
- 書き出しスケジュールは、ダウンストリーム処理ウィンドウと一致させる必要があります

**メリット：**

- オーディエンスデータの消費方法に関する最大の柔軟性
- ファイルを読み取ることができる任意のダウンストリームシステムをサポート
- 書き出し形式、フィールド、スケジュールを完全に制御
- 1 回の書き出しで複数の下流コンシューマーにサービスを提供できます
- 完全エクスポートモードと増分エクスポートモードをサポート

**制限事項：**

- すべてのオプションで最も長い待ち時間（バッチのみ）
- ダウンストリーム読み込みワークフローの作成および維持が必要です
- ネイティブ統合なし – 追加の開発作業が必要
- ファイル管理（クリーンアップ、アーカイブ）は別々に処理する必要があります

**Experience League:**

- [Amazon S3 の宛先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/cloud-storage/amazon-s3)
- [Azure Blob ストレージの宛先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/cloud-storage/azure-blob)
- [バッチ宛先に対するオーディエンスのアクティブ化](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/api/connect-activate-batch-destinations)

### オプション D:CRM システムへのストリーミングアクティベーション

**最適な対象：** セールスチームの可視化、テリトリー割り当ての更新、自動化されたセールスワークフローのために、アカウントの選定の変更をほぼリアルタイムで CRM （[!DNL Salesforce] または [!DNL Dynamics]）に反映する必要があるセールスマーケティングとマーケティングの連携のユースケース。

**仕組み：**

このオプションは、ストリーミング宛先コネクタ（CRM、[!DNL Microsoft Dynamics] な [!DNL Salesforce]）を使用して、アカウントオーディエンスメンバーシップの変更を CRM アカウントまたは連絡先レコードに直接プッシュします。 アカウントがオーディエンスに適合するか、オーディエンスから離脱すると、CRM レコードが更新され、セグメントメンバーシップを示すカスタムフィールドが表示されます。 その後、営業チームは、これらのフィールドに基づいて CRM レポート、ビューおよびアラートを作成できます。

ストリーミングコネクタは、オーディエンスメンバーシップの変更が発生すると、増分更新を送信します。 フィールドマッピングは [!DNL RT-CDP] アカウント属性と人物属性を CRM フィールドに接続し、[!DNL RT-CDP] の統合データを使用して CRM レコードを強化できるようにします。 このアプローチは、マーケティングインテリジェンス（アカウントの選定）とセールスの実行（CRM ワークフロー）の間のループを閉じます。

**主な考慮事項：**

- 適切な書き込み権限を持つ CRM API アクセスが必要です
- オーディエンスメンバーシップデータを受信するには、CRM カスタムフィールドを作成する必要があります
- ストリーミングボリュームは、CRM API レートの制限内に収まる必要があります
- フィールドの更新に対応するように CRM ワークフローオートメーションを構成する必要があります

**メリット：**

- セールスチームの可視化に役立つ、ほぼリアルタイムの CRM の更新
- オーディエンスの変更によってトリガーされた自動 CRM ワークフローを有効にします
- [!DNL RT-CDP] からの統合アカウントデータで CRM レコードを強化
- アカウントレベルと連絡先レベルの両方のフィールドのアップデートをサポート

**制限事項：**

- CRM API のレート制限により、大量のアップデートがスロットルされる可能性があります
- CRM のカスタマイズが必要（カスタムフィールド、ワークフロー）
- ネイティブコネクタとして [!DNL Salesforce] および [!DNL Microsoft Dynamics] に制限
- ストリーミング評価制約は、基になるオーディエンスに適用されます

**Experience League:**

- [Salesforce CRM の宛先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/crm/salesforce)
- [Microsoft Dynamics 365 の宛先](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/crm/microsoft-dynamics-365)

### 適切なオプションを選択

次の決定ガイダンスを使用して、適切なアクティベーションアプローチを選択してください。

1. **主な目標が B2B リード育成で、[!DNL Marketo Engage]** れを使用する場合は、オプション A を選択します。ネイティブのストリーミングコネクタは、マーケティング自動化ワークフローでアクションまでの時間を最速にします。

2. **広告を通じたアカウントベースの認知と需要の創出を主な目標としている場合**、オプション B を選択します。[!DNL LinkedIn] マッチしたオーディエンスは、アカウントレベルで最も強力なターゲティングを提供します。 幅広いリーチのために他の広告プラットフォームを使用してください。

3. **アカウントオーディエンスをネイティブ [!DNL RT-CDP] コネクタのないシステムにフィードする必要がある場合**、またはデータ形式とダウンストリーム処理を最大限に制御する必要がある場合は、オプション C を選択します。これは、パートナーのデータ共有やデータウェアハウスのエンリッチメントにも最適な選択肢です。

4. **主な目標がセールスの有効化と、マーケティング対象アカウントの CRM での可視性である場合**、オプション D を選択します。これにより、ほぼリアルタイムで CRM レコードが更新され、マーケティングからセールスへのハンドオフループが閉じられます。

ほとんどの B2B 組織は、複数のオプションを同時に実装します。例えば、育成プログラムの場合はオプション A、[!DNL LinkedIn] 広告の場合はオプション B、CRM 同期の場合はオプション D です。 これらのオプションは相互に排他的ではありません。同じアカウントオーディエンスを複数の宛先に対して同時にアクティブ化できます。

### フェーズ 1：アカウントプロファイルのエンリッチメント

このフェーズでは、CRM、マーケティング自動化、サードパーティソースからのデータを統合することで、統合アカウントプロファイルを確立します。

**アプリケーション機能：** [!DNL RT-CDP] B2B：アカウントプロファイルの統合、[!DNL RT-CDP] B2B:B2B ID の解決

**構成内容：** このフェーズでは、CRM、マーケティング自動化、サードパーティソースからのデータを統合することで、統合アカウントプロファイルを確立します。 B2B ID 解決は、ユーザーレベルのエンゲージメントデータ（メール開封数、web 訪問数、コンテンツのダウンロード数）を集計し、アカウントレベルのオーディエンス評価で使用できるように、ユーザーとアカウントの関係をマッピングします。 このフェーズは、すでに導入されている必要がある基本機能 F2、F3、および F4 に基づいて構築されます。

### フェーズ 4:Audience Activation

このフェーズでは、評価済みのアカウントオーディエンスを設定済みの宛先に公開します。

**Application 関数：** [!DNL RT-CDP] B2B：アカウントAudience Activation、[!DNL RT-CDP]:Audience Activation

**設定内容：** このフェーズでは、評価済みのアカウントオーディエンスを設定済みの宛先に公開します。 アクティベートは、アカウントオーディエンス（ソース）を外部の宛先（ターゲット）に接続するデータフローを作成し、属性マッピングを適用し、設定されたスケジュールまたはストリーミング動作に従って書き出しを開始します。 また、不適格なアカウントをアクティブ化から除外するように抑制オーディエンスを設定します。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：書き出しのタイプ**
>
>アクティベーションで、オーディエンスの完全なスナップショットをエクスポートするか、増分変更のみをエクスポートするかを指定してください。
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 増分書き出し | ストリーミング宛先（[!DNL Marketo]、CRM）またはダウンストリームシステムがデルタを処理するバッチ宛先 | 書き出しあたりのデータ量が少ない。 処理の高速化。 状態を管理するためにダウンストリームシステムが必要です。 ストリーミング宛先のデフォルト。 |
>| 完全な書き出し | ダウンストリームシステムが各実行の完全なオーディエンスを置き換えるバッチ宛先 | より大きなデータ量だが、よりシンプルなダウンストリームロジック。 ダウンストリームシステムが状態を維持しない場合に役立ちます。 ファイルベースの宛先で使用できます。 |

>[!NOTE]
>**決定：アクティブ化の範囲**
>
>宛先ごとに有効化する必要があるオーディエンスの数
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| データフローごとの単一オーディエンス | オーディエンスと宛先のマッピングが明確なシンプルなアクティベーション | 監視とトラブルシューティングが最も簡単です。 1 つのオーディエンスの失敗が他のオーディエンスに影響を与えることはありません。 |
>| データフローごとの複数のオーディエンス | 同じ宛先に送信される複数の関連オーディエンス | 宛先接続のより効率的な使用。 すべてのオーディエンスは、同じフィールドマッピングおよびスケジュールを共有します。 |

**UI ナビゲーション：** 接続/宛先/参照/宛先を選択/オーディエンスをアクティブ化

**キー設定の詳細：**

- オーディエンスリストからターゲットオーディエンスを選択
- フェーズ 3 で設定されたフィールドマッピングを確認します。
- バッチ宛先の場合：書き出しスケジュール（時間、頻度）を設定します
- ストリーミング宛先の場合：アクティベーションは設定直後に開始されます
- オプションで、除外機能を使用して抑制オーディエンスを追加する
- データフローを検証するための最初のテストアクティベーションの実行をトリガーする

**オプションが分岐する場所：**

**オプション A （[!DNL Marketo Engage] ストリーミング）の場合：**

有効化するアカウントオーディエンスを選択します。 アクティベーションが直ちにストリーミングを開始します。 リード/連絡先レコードがセグメントメンバーシップフィールドで更新されていることを [!DNL Marketo] で確認します。 これらのフィールド [!DNL Marketo] 変更に基づいてトリガーにするスマートキャンペーンを設定します。

**オプション B （Advertising プラットフォームバッチ）の場合：**

アカウントオーディエンスを選択し、毎日の書き出しスケジュールを設定します。 最初の書き出しが完了したら、広告プラットフォームで、オーディエンスが表示され、メンバー数が入力されていることを確認します。 プラットフォームが初期オーディエンスファイルを処理するまで 24～48 時間待ちます。

**オプション C （クラウドストレージファイルベース）の場合：**

アカウントオーディエンスを選択し、書き出しスケジュールとファイル形式を設定します。 最初の書き出し後、ファイルが、期待される形式とコンテンツでターゲットのストレージの場所に表示されることを確認します。 ダウンストリームのインポートプロセスで、書き出されたファイルが正常に使用されることを確認します。

**オプション D （CRM ストリーミング）の場合：**

有効化するアカウントオーディエンスを選択します。 アクティベーションが直ちにストリーミングを開始します。 CRM で、アカウントレコードまたは連絡先レコードが、マッピングされたフィールドを使用して更新されていることを確認します。 更新されたフィールドに対して機能するように、CRM レポート、リストビュー、ワークフロー自動化を設定します。

### フェーズ 5：ガバナンスと監視

このフェーズでは、アカウントのオーディエンスのアクティベーションがデータガバナンスポリシーおよび同意環境設定に準拠し、進行中のアクティベーションデータフローが正常性について監視されます。

**アプリケーション機能：** [!DNL RT-CDP] B2B: B2B データガバナンス、[!DNL RT-CDP]：同意とガバナンスの適用

**構成内容：** このフェーズでは、アカウントのオーディエンスのアクティベーションがデータガバナンスポリシーおよび同意環境設定に準拠し、進行中のアクティベーションデータフローが正常性について監視されます。 B2B データガバナンスでは、アカウントの機密属性（売上高、サードパーティプロバイダーからの従業員数）に関する制限が適用されますが、同意の実施により、ユーザーレベルのコミュニケーションでオプトアウトの環境設定が確実に適用されます。 監視は、アクティベーションデータフローが正常に完了していることを確認します。

**このフェーズの決定ポイント：**

>[!NOTE]
>**決定：ガバナンスの実施レベル**
>
>B2B アクティベーションには、どのように厳密にデータガバナンスを適用する必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 完全適用（違反に対してブロック） | 機密データを使用した実稼動のアクティベーション | ガバナンスポリシーに違反するアクティベーションを防ぎます。 違反を解決するために、反復フィールドマッピングの調整が必要になる場合があります。 |
>| 警告して続行 | 開発環境またはテスト環境 | 有効化を続行できますが、警告がログに記録されます。 潜在的な問題を特定するために、初期セットアップ中に役立ちます。 |

>[!NOTE]
>**決定：モニタリングのアプローチ**
>
>アクティベーションの正常性はどのように監視する必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| アラートベースの監視 | 障害を迅速に把握する必要がある実稼動環境 | 宛先のアクティブ化の失敗、ソースフローの失敗、データフローの遅延に関するアラートを設定します。 アラート購読の設定が必要です。 |
>| 手動での監視 | 開発または少量のアクティベーション | 宛先 UI でのデータフロー実行を定期的に確認します。 オーバーヘッドは少ないが、エラー検出が遅延するリスクがある。 |
>| 両方 | 複雑な複数の宛先のアクティベーションを伴う実稼動環境 | 重大なエラーに関するアラートと、トレンドの定期的なダッシュボードレビュー。 ほとんどの B2B 実装で推奨されます。 |

**UI ナビゲーション：** プライバシー/ ポリシー（ガバナンス用）、接続/宛先/参照/宛先を選択/データフロー実行（監視用）

## 実装に関する考慮事項

以下の節では、実装を成功させるための追加のガイダンスを示します。

### ガードレールと制限

このユースケースパターンに適用される次のプラットフォームガードレールと制限をレビューします。

- アカウントオーディエンスを含む、サンドボックスあたり最大 4,000 個のセグメント定義 – [&#x200B; セグメント化ガードレール &#x200B;](https://experienceleague.adobe.com/en/docs/experience-platform/profile/guardrails)
- アカウントオーディエンスは主にバッチ評価を使用して評価されます。ストリーミングの実施要件は、単純なアカウント属性条件に制限されます
- 宛先接続あたり最大 100 個のデータフロー – [&#x200B; 宛先ガードレール &#x200B;](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/guardrails)
- バッチ宛先では、ファイルセグメントあたり最大 500 万のプロファイルを書き出します
- ストリーミング宛先には、宛先パートナーによって設定された 1 秒あたりのスループットの制限（[!DNL Marketo] API レート制限など）があります
- （オーディエンスコンポジションから）構成されたオーディエンスは、バッチ評価に制限され、ストリーミングを使用できません
- オーディエンス構成キャンバスあたり最大 10 個の構成ブロック
- [!DNL LinkedIn] 件のマッチしたオーディエンスをアクティブ化するには、オーディエンスサイズが最小（通常は 300 人のメンバー）である必要があります
- CRM ストリーミングの宛先は、CRM プロバイダーの API レート制限（例：一括 API の 1 日の制限） [!DNL Salesforce] 従います
- B2B edition ライセンス [!DNL RT-CDP]、ビジネスアカウントプロファイルの合計数を管理します。[RT-CDP product description](https://helpx.adobe.com/legal/product-descriptions/real-time-customer-data-platform-b2b-edition-prime-and-ultimate-packages.html)

### よくある落とし穴

このユースケースパターンを実装する際には、次の一般的な問題に注意してください。

- **不完全な人物とアカウントのマッピング：** 人物レコード（リード、連絡先）が B2B ID 解決を通じてアカウントレコードに適切にリンクされていない場合、人物の属性やアクティビティに依存するアカウントオーディエンスは、適格なアカウントをアンダーカウントします。 アカウントオーディエンスを構築する前に、F4 でユーザーとアカウントの関係を検証します。

- **オーディエンスドリフトの原因となる古い CRM データ：** CRM ソースコネクタが通常のスケジュールで実行されていない場合や、サイレントに失敗している場合、アカウント属性（業界、収益、ステータス）が古くなります。 これにより、オーディエンスに、選定されなくなったアカウントが含まれたり、選定する必要のあるアカウントが除外されたりします。 ソースコネクタデータフローの正常性を監視します。

- **Advertising Platform 一致率の期待値：** [!DNL LinkedIn] 以外の広告プラットフォームに対してアカウントオーディエンスをアクティブ化した場合、一致率は、対象アカウントに関連付けられた連絡先に有効なユーザーレベルの識別子（ハッシュ化されたメール）があるかどうかによって決まります。 メールアドレスを持つ連絡先が関連付けられていないアカウントは、一致しません。 一致率を監視し、連絡先データのエンリッチメントを検討します。

- **[!DNL Marketo]フィールドマッピングの位置合わせ不良：** [!DNL Marketo Engage] にストリーミングする場合、[!DNL RT-CDP] フィールドマッピングは既存の [!DNL Marketo] ースリードまたは連絡先フィールドをターゲットにする必要があります。 マッピングされた [!DNL Marketo] フィールドが存在しない場合、更新はサイレントに失敗します。 宛先を設定する前に、[!DNL Marketo] のすべてのターゲットフィールドを事前に作成してください。

- **ガバナンスポリシーブロックの有効化：** 広告宛先に対してアクティブ化を行うと、アカウント属性フィールド（特にサードパーティのファームグラフィックデータ）のデータ使用ラベルがトリガーガバナンス違反を引き起こす可能性があります。 必要に応じて、フィールドマッピングをアクティブ化および調整して制限されたフィールドを除外する前に、ガバナンスコンプライアンスを評価します。

- **アカウントと人物データをバッチのみの評価と組み合わせるアカウントオーディエンス：** 人物レベルの行動イベントを参照するアカウントオーディエンス（「過去 30 日間に少なくとも 1 人の連絡先がメールを開封したアカウント」など）には、バッチ評価が必要です。 ユースケースでリアルタイム選定が想定されている場合、この制約が原因で予期しない待ち時間が発生する可能性があります。

### ベストプラクティス

最適な結果を得るには、次の推奨事項に従います。

- 少数の明確に定義されたアカウントオーディエンス（ICP 層、業界別、エンゲージメント層）から始めて、複雑な複数属性の定義に拡張します
- 既存の顧客、アクティブなオポチュニティ、最近関与したアカウントに対して専用の抑制オーディエンスを作成して、無駄な費用とチャネルの競合を回避します
- オーディエンス構成を使用して、集計したエンゲージメントスコアに基づいてアカウントをランキングし、階層化されたアカウントオーディエンス（高/中/低のエンゲージメント）を作成します。
- 調整されたマルチチャネルキャンペーンに対して、同じアカウントオーディエンスを複数の宛先に同時にアクティブ化（広告の [!DNL LinkedIn]、メールの [!DNL Marketo]、セールスの可視性の CRM など）
- ターゲティング条件と目的のチャネルを含む、アカウントオーディエンスの一貫性のある命名規則を実装します（例：「B2B_ICP_Enterprise_Tech_LinkedIn」、「B2B_Suppression_ActiveOpps」）
- オフピーク時のバッチアクティベーションをスケジュールして、ダウンストリームシステムへの影響を最小限に抑え、広告プラットフォームの処理ウィンドウに合わせます
- 最初のアクティベーション後に宛先ごとの一致率を監視し、ID フィールドマッピングを繰り返して、一致を改善します
- [!DNL Marketo Engage] を使用した双方向のデータフローの維持：[!DNL Marketo] から [!DNL RT-CDP] （ソースコネクタ）へのエンゲージメントデータの取り込み、閉じたループシステム用に [!DNL Marketo] （宛先コネクタ）にオーディエンスをアクティベートします

### トレードオフの決定

実装を決定する際は、次のトレードオフを考慮します。

>[!NOTE]
>**トレードオフ：オーディエンスの鮮度と評価の複雑さ**
>
>アカウント属性をユーザーレベルの行動データと組み合わせたアカウントオーディエンスは、最も正確なターゲティングを提供しますが、バッチ評価（毎日の更新）に限定されます。 アカウント属性のみに基づく単純なオーディエンスであれば、ほぼリアルタイムで更新されるストリーミング評価の対象となる場合があります。
>
>- **バッチ（複雑な条件）お気に入り：** ターゲティング精度、初歩的なシグナルと行動シグナルの組み合わせ、包括的なアカウントスコアリング
>- **ストリーミング（シンプルな条件）のメリット：** オーディエンスの更新を迅速化、アカウント変更に対するリアルタイムの応答、[!DNL Marketo] および CRM でのアクションまでの時間を短縮
>- **推奨事項：** 毎日の更新を許容できる（ほとんどの B2B のユースケース）プライマリターゲティングオーディエンスには、バッチ評価を使用します。 アカウントステータスの変化や価値の高いアカウントアラートなど、時間に依存するトリガーに対しては、ストリーミング評価を予約します。

>[!NOTE]
>**トレードオフ：一元化されたアクティベーションと宛先固有のオーディエンスの比較**
>
>単一の統合アカウントオーディエンスを複数の宛先に対してアクティブ化したり、各チャネルの機能と要件に合わせて宛先固有のオーディエンスを作成したりできます。
>
>- **一元的なオーディエンスのお気に入り：** チャネル間の一貫性、よりシンプルなオーディエンス管理、統一されたレポート
>- **宛先固有のオーディエンスのお勧め：** チャネルごとに最適化されたターゲティング（例：[!DNL Marketo] ーザーがリードレベルの詳細を必要とする一方で、会社レベルの属性から得られるメリットが [!DNL LinkedIn] まる）、チャネル固有の抑制ルール
>- **推奨事項：** 一貫性を保つため、一元化されたオーディエンスから始めて、チャネル要件が大きく異なる場合（広告とメールで異なる抑制ウィンドウなど）にのみ、宛先固有のバリアントを作成します。

>[!NOTE]
>**トレードオフ：リアルタイムの CRM 同期とバッチ CRM 更新の比較**
>
>CRM へのストリーミングにより、売上高をすぐに可視化できますが、CRM API の割り当ては消費されます。 バッチファイルの書き出しは効率的ですが、待ち時間が発生します。
>
>- **ストリーミング CRM 同期のお勧め：** 販売の応答性、即時のアカウントアラート、リアルタイムのパイプライン表示
>- **バッチ CRM 更新のお勧め：** API クォータの節約、一括更新の効率、CRM データ読み込みウィンドウとの連携
>- **推奨事項：** アカウントの優先順位が高い変更（アカウントの「販売準備完了」層への移行など）には、ストリーミング CRM 同期を使用します。 毎月のアカウント・スコアリング更新やテリトリ再割り当てリストなどの一括更新には、バッチ・ファイル・エクスポートを使用します。
