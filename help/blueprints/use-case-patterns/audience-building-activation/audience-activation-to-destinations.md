# 宛先への Audience Activation

このガイドでは、Adobe [!DNL Real-Time Customer Data Platform] （RT-CDP）から外部の宛先へのオーディエンスをアクティブ化するための完全な実装リファレンスを提供します。 オーディエンスセグメントを評価して、ターゲティング、抑制、類似モデリングまたは分析強化のために、広告プラットフォーム、クラウドストレージ、CRM システム、データパートナーに公開する必要があるソリューションアーキテクト、マーケティング技術者、実装エンジニア向けに設計されています。

トレードオフ分析、意思決定ガイダンス、UI ナビゲーションパス、Experience League ドキュメントのリファレンスなど、実行可能なすべての実装オプションを紹介します。

このガイドでは、オーディエンスセグメントの定義と評価から、宛先接続の設定とオーディエンスの公開、アクティベーションの正常性の監視、ガバナンスコンプライアンスの実施まで、オーディエンスアクティベーションのライフサイクル全体をカバーします。

## ユースケースの概要

組織は、有料メディアキャンペーンの強化、CRM レコードのエンリッチメント、パートナーとのデータの共有、ダウンストリーム分析のフィードを行うために、オーディエンスデータを外部システムに配信する必要があります。 宛先へのAudience Activationは、RT-CDP の基本的なアクティベーションパターンであり、ターゲットオーディエンスに適合するプロファイルを評価し、1 つ以上の外部宛先に接続し、プロファイル属性を宛先固有のフィールドにマッピングし、ダウンストリーム使用のためにオーディエンスを公開します。

このパターンは、オーディエンスデータを適切な形式で適切なタイミングで外部システムに送信することを目標とする場合に適用されます。 メッセージ配信、オンサイトのパーソナライゼーションまたは分析は含まれません。 これは、RT-CDP 実装の最も一般的な出発点であり、他のパターンが上に構成される構築ブロックとして機能します。

一般的な関係者には、有料メディアを管理するデジタルマーケティングチーム、ウェアハウスを充実させるデータチーム、キャンペーンの連絡先リストを準備する CRM チーム、アウトバウンドデータフローのガバナンスコンプライアンスを確保するプライバシーチームなどがあります。

ターゲティング効率の向上、獲得キャンペーンからの既存顧客の抑制、メディア支出の最適化を実現します。

**KPI：顧客獲得コスト、リードあたりのコスト、効率**

[顧客獲得コストの削減に関する詳細情報](/help/blueprints/business-objectives/cost-efficiency/reduce-customer-acquisition-cost.md)

### マーケティング費用と ROI の最適化

ターゲティング、アトリビューション、オーディエンス抑制、予算配分の改善を通じて、マーケティング投資回収率（ROI）を向上させます。

**KPI:** コスト削減、顧客獲得コスト、収益増

## 戦術的な使用例

- **広告プラットフォームオーディエンスのターゲティング** - キャンペーンターゲティング用に、選定されたセグメントを有料メディアプラットフォームにプッシュします
- **既存の顧客の有料メディア抑制** – 広告プラットフォームの獲得キャンペーンから既知の顧客を除外して、無駄な支出を排除します
- **類似シードオーディエンス** – 類似の機能拡張のシードオーディエンスとして、高価値のカスタマーセグメントを Facebook、Google Ads または The Trade Desk にプッシュします
- **セールスイネーブルメントのための CRM 同期** – 高インテントまたは高価値のオーディエンスをアクティブ化して、セールスチームがアウトリーチを優先できるようにします
- **データパートナーのオーディエンス共有** - Co-op ターゲティングや測定のために、適格なオーディエンスセグメントをデータパートナーと共有します
- **Data Warehouse エンリッチメントのためのクラウドストレージの書き出し** - オーディエンスメンバーシップとプロファイル属性をAmazon S3、Azure Blob、Google Cloud Storage または SFTP に書き出して、ダウンストリーム分析に使用します
- **オーディエンスのリターゲティングアクティベーション** - リターゲティングプラットフォームに変換しなかったサイト訪問者をアクティブ化します
- **メールサービスプロバイダーへの連絡先リスト同期** - オーディエンスメンバーシップをサードパーティのメールプラットフォームにプッシュして、連携したアウトリーチを行います

## 実装オプション

このユースケースパターンでは、次の実装オプションを使用できます。 各オプションと比較表を確認して、要件に最適なアプローチを決定します。

### オプション A：ストリーミング宛先のアクティベーション

**最適な対象：** Facebook Custom Audiences、Google Ads Customer Match、LinkedIn Matched Audiences、The Trade Desk など、広告プラットフォームやパートナーシステムに対するリアルタイムオーディエンスメンバーシップのアップデート。

**仕組み：**

ストリーミング宛先のアクティベーションでは、プロファイルがセグメントに適合または不適合になると、オーディエンスメンバーシップの変更がほぼリアルタイムで宛先に配信されます。 プロファイル属性が変更された場合や、プロファイルがオーディエンスにエントリまたはオーディエンスから離脱する行動イベントが発生した場合、メンバーシップの更新は数分以内に宛先にストリーミングされます。

このアプローチには、ストリーミング宛先コネクタ（ほとんどの主要な広告プラットフォームでこれをサポート）と、ストリーミングまたはエッジオーディエンスの評価方法が必要です。 オーディエンス評価は、選定されたプロファイルの変更を継続的に監視し、アクティベーションデータフローは、更新が発生するとプッシュします。 宛先は、完全なオーディエンススナップショットではなく、増分メンバーシップの変更を受け取ります。

ストリーミングアクティベーションは、RT-CDP カタログ内のほとんどの広告プラットフォーム宛先のデフォルトです。 宛先コネクタは、API 認証、レート制限、再試行ロジックを自動的に処理します。

**主な考慮事項：**

- オーディエンス評価では、ストリーミングまたはエッジ評価を使用する必要があります（バッチ専用オーディエンスは、ストリーミングの宛先をリアルタイムでフィードできません）
- すべてのセグメント式がストリーミング評価の対象となるわけではありません。複雑な集計、複数エンティティクエリ、特定の時間ベース関数にはバッチが必要です
- 宛先パートナー API のレート制限が、大規模なオーディエンスの選定イベント中のスループットに影響を与える可能性があります
- プロファイル属性の更新は、メンバーシップの変更と共に送信され、宛先データが最新の状態に保たれます

**メリット：**

- 宛先でのオーディエンスの鮮度がほぼリアルタイム（数時間ではなく数分）
- 増分更新により、完全な書き出しと比較してデータ転送の量が減ります
- 認定および不適格イベントの自動処理
- ほとんどの広告プラットフォーム宛先は、ストリーミングをネイティブにサポートしています

**制限事項：**

- ストリーミング対応のセグメント定義（制限付きセグメント関数セット）が必要です
- ファイル形式や書き出し構造を制御できない – 宛先コネクタによって決定されるデータ形式
- ファイルベースのストレージ宛先（S3、Azure Blob、SFTP）に書き出せない
- 宛先 API のレート制限により、大量の変更がスロットルされる可能性があります

### オプション B：バッチ宛先の有効化（ファイルエクスポート）

**最適な対象：** クラウドストレージ、データウェアハウス、またはファイルベースの読み込みを使用するシステム（Amazon S3、Azure Blob Storage、Google Cloud Storage、SFTP、データランディングゾーン）へのスケジュールされたオーディエンスの書き出し。

**仕組み：**

バッチ宛先アクティベーションでは、スケジュールされたサイクルでオーディエンスを評価し、結果をファイル（CSV、JSON または Parquet）として設定済みのストレージ宛先に書き出します。 エクスポートには、完全なオーディエンススナップショットまたは増分変更（前回のエクスポート以降の新しい選定および選定解除）を含めることができます。

この実装には、ストレージ資格情報、ファイル形式の環境設定（区切り文字、圧縮、命名規則）、書き出しのスケジュール（毎日、3 時間ごと、6 時間ごと等）を使用した、ファイルベースの宛先接続の設定が含まれます。 各書き出し実行では、オーディエンスメンバーシップとマッピングされたプロファイル属性を含むファイルが生成されます。

ファイルベースのデータ交換はユニバーサルにサポートされているので、このアプローチは、ダウンストリームのコンシューマーの最も広い範囲をサポートします。 また、クラウドストレージおよびデータウェアハウスのエンリッチメントのユースケースに対する唯一のオプションでもあります。

**主な考慮事項：**

- オーディエンス評価では、任意の方法（バッチ、ストリーミングまたはエッジ）を使用できます。書き出し自体は、関係なく、スケジュールに従って実行されます
- ファイル形式、圧縮、命名規則は、宛先ごとに設定できます
- 増分書き出し（変更のみ）と完全書き出し（完全なオーディエンススナップショット）の両方をサポートします
- スケジュール精度の範囲を 3 時間ごとから毎日に書き出す

**メリット：**

- 書き出しファイルの形式（CSV、JSON、Parquet）、圧縮、命名を完全に制御する
- ファイルを使用できるすべてのダウンストリームシステムと互換性があります
- 増分エクスポートモードと完全エクスポートモードの両方をサポート
- 任意の評価方法でオーディエンスを書き出すことができます（複雑なセグメント化ロジックを持つバッチ専用セグメントを含む）
- 通常のスケジュール外でのアドホック（オンデマンド）書き出しの実行をサポートします

**制限事項：**

- 本来、待ち時間は長くなります。オーディエンスデータは、リアルタイムではなく、スケジュールに従って宛先に到達します
- ファイルベースの書き出しでは、宛先システムがファイルを処理して取り込む必要があります
- 書き出されたファイルの保存先のストレージコスト
- 増分ストリーミング更新と比較して、エクスポートあたりのデータ量が多い

**Experience League:**

- [プロファイル書き出しのバッチ宛先に対するオーディエンスのアクティブ化](https://experienceleague.adobe.com/ja/docs/experience-platform/destinations/ui/activate/activate-batch-profile-destinations)
- [ファイルベースの宛先カタログ](https://experienceleague.adobe.com/en/docs/experience-platform/destinations/catalog/cloud-storage)

### オプション C：複数の宛先のアクティベーション

**最適な対象：** 複数の宛先に対して同じオーディエンスを同時にアクティブ化します。例えば、抑制リストをすべての広告プラットフォームにプッシュしたり、高値セグメントを CRM と広告プラットフォームの両方に同期したり、複数のメディアパートナー間でオーディエンスのリーチを調整したりします。

**仕組み：**

複数の宛先のアクティベーションは、個別の技術的なメカニズムではなく、複数の宛先接続にわたって同じオーディエンスに適用されるオプション A と B の構成です。 同じオーディエンスが 2 つ以上の宛先に対してアクティブ化され、それぞれに独自の接続設定、フィールドマッピング、スケジュールが設定されます。

各宛先接続は個別に動作します。Facebook へのストリーミングアクティベーションと S3 へのバッチ書き出しは、同じソースオーディエンスから同時に実行できます。 フィールドマッピングは宛先ごとに設定されるので、各宛先が必要とする内容と許可されるガバナンスポリシーに基づいて、異なる属性を異なるシステムに送信できます。

これは、複数の広告プラットフォームをまたいで運用する組織、メディアのアクティベーションと共に CRM の同期を維持する組織、または運用システムと分析システムの両方にオーディエンスデータを配布する必要がある組織にとって共通の実稼動パターンです。

**主な考慮事項：**

- 各宛先には、独立したフィールドマッピング、スケジュールおよびガバナンス評価があります
- ガバナンスポリシーは宛先ごとに適用されます。宛先タイプごとに異なる属性を使用できます
- 1 つのオーディエンスをストリーミング宛先とバッチ宛先の組み合わせに同時にアクティベートできます
- 新しい宛先の追加では、既存のアクティベーションを変更する必要はありません

**メリット：**

- 単一のオーディエンス定義から、すべての外部システムにわたってオーディエンス配分を調整します。
- 宛先ごとに独立した設定を使用すると、マッピングとスケジュールを最適化できます
- 宛先ごとのガバナンスの適用により、様々な宛先タイプにわたるコンプライアンスが確保されます
- 分散型アクティベーションをサポートしながら、オーディエンス管理を一元化

**制限事項：**

- 設定、認証、維持するその他の宛先接続
- 監視の複雑さは、宛先ごとに増加します。アクティベーションの失敗は、宛先ごとに追跡する必要があります
- ライセンス使用状況（アクティブ化されたプロファイル）は、使用権限に応じて、宛先ごとにカウントされる場合があります
- 複数の宛先間でのフィールドマッピングのメンテナンスには、調整が必要です

**Experience League:**

- [宛先の概要](https://experienceleague.adobe.com/ja/docs/experience-platform/destinations/home)
- [宛先カタログ](https://experienceleague.adobe.com/ja/docs/experience-platform/destinations/catalog/overview)

### 適切なオプションを選択

次の決定ロジックを使用して、適切な実装アプローチを選択します。

1. **必要な宛先の数** 2 つ以上の宛先に対して同じオーディエンスをアクティブ化する必要がある場合は、オプション C （宛先ごとにオプション A と B を構成する）を実装しています。 宛先ごとに 2 と 3 の質問に進みます。

2. **宛先はストリーミングをサポートしていますか？** 宛先が広告プラットフォーム（Facebook、Google広告、LinkedIn、The Trade Desk）またはストリーミング API を使用したパートナー統合の場合は、その宛先にオプション A を使用します。 移行先がクラウドストレージ（S3、Azure Blob、GCS、SFTP）またはファイルを使用するシステムの場合は、オプション B を使用します。

3. **オーディエンスはどのくらいの速さで宛先に到着する必要がありますか？** オーディエンスメンバーシップが数分以内に宛先で反映される必要がある場合（アクティブキャンペーン中のリアルタイム抑制など）、オプション A を使用します。毎日または複数の時間ごとの配信で十分な場合（例：毎週のデータウェアハウスの更新、CRM バッチ同期）、オプション B を使用します。

4. **オーディエンスは複雑なセグメント化ロジックを使用していますか？** オーディエンス定義に複数イベントの集計、大きな時間、ストリーミング評価の対象とならない関数が含まれる場合は、オプション B （バッチ宛先）を使用するか、オプション A の宛先がバッチ評価されたオーディエンスをスケジュールに従って受け取ることを確認します。

## 実装フェーズ

実装は、次のフェーズに従います。 各フェーズには、設定の詳細、判断ポイント、Experience League ドキュメントへのリンクが含まれます。

### フェーズ 1：オーディエンスの評価

**アプリケーション関数：** RT-CDP：オーディエンス評価、RT-CDP：オーディエンス構成

**設定内容：** 宛先に対してアクティブ化されるターゲットオーディエンスを定義します。 これには、オーディエンス条件の指定（プロファイルが選定されること）、評価方法の選択（メンバーシップの更新頻度）、オーディエンス母集団の検証が含まれます。 これは、すべてのアクティベーションの出発点です。定義され評価されたオーディエンスがない場合、アクティベーションするものは何もありません。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：オーディエンスの作成方法**
>
>ターゲットオーディエンスはどのように定義する必要がありますか。
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| セグメントビルダー（ルールベース） | プロファイル属性、行動イベントおよびセグメントメンバーシップ条件を使用した標準オーディエンス定義 | 最も柔軟なルール定義。対象となる式に対してストリーミングとエッジの評価をサポートし、単一条件のオーディエンスまたは適度に複雑なオーディエンスに最適です。 |
>| オーディエンス構成 | オーディエンスには、既存オーディエンスの組み合わせ、ランキング、分割または除外が必要です | 順次操作用の視覚的なキャンバス。結果はバッチ評価のみ。1 つのキャンバスにつき最大 10 個のコンポジションブロック |
>| Federated Audience の構成 | オーディエンス条件は、AEPに取り込まずに、外部データウェアハウスのデータに対して評価する必要があります | 外部データベースに対して直接クエリを実行し、データの重複を回避します。Federated Audience Composition ライセンスが必要です。 |

>
>**決定：評価方法**
>
>プロファイルは、オーディエンスに入る必要がある場合と出る必要がある場合で、どの程度迅速に行う必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| バッチ | スケジュール済みキャンペーン、毎日のオーディエンスの更新、またはストリーミングに適合しない複雑なセグメント化ロジック | ジョブあたり最大 2,400 万プロファイルを処理します。すべてのセグメント化機能をサポートします。スケジュールに従って実行します（毎日のデフォルトまたはカスタム cron スケジュール） |
>| ストリーミング | ストリーミング宛先のアクティベーションまたはイベントトリガーのユースケースには、リアルタイムオーディエンスメンバーシップの変更が必要です | ほぼリアルタイム（数秒から数分）、制限付きセグメント化関数セット（単純なイベント、属性の比較、制限付き時間ウィンドウのみ）、複数エンティティのクエリなし |
>| Edge | エッジで必要な同じページのパーソナライゼーションまたはインスタントオーディエンスの選定 | ミリ秒の待ち時間。最も制限が厳しいルールセット – プロファイル属性とシンプルなセグメントメンバーシップチェックのみ。主にオンサイトパーソナライゼーションに使用されます（宛先のアクティベーションではあまり一般的ではありません） |

>
>**決定：結合ポリシー**
>
>オーディエンス評価で使用する結合ポリシーはどれですか？
>

>| --- | --- | --- |
>| 順序付きタイムスタンプ（デフォルト） | ほとんどのユースケース – プロファイルフラグメントが競合する場合は、最新のデータを優先する必要があります | 最も一般的。最近更新された属性値を使用します。 |
>| データセットの優先順位 | タイムスタンプに関係なく、特定のデータソースがその他のデータソースを上書きする必要がある | データセットの優先順位の定義が必要です。レコードの CRM システムが常に web で収集されたデータを上書きする必要がある場合に役立ちます |

**UI ナビゲーション：** 顧客/オーディエンス/オーディエンスを作成/ルールを作成（セグメントビルダーの場合）またはオーディエンスを構成（オーディエンス構成の場合）

**キー設定の詳細：**

- ユースケース（プロファイル属性、行動イベント、セグメントメンバーシップまたは時間ベースの条件）に基づいてオーディエンス条件を定義します
- アクティブ化しないプロファイル（最近変換されたプロファイル、グローバルに購読解除されたプロファイルなど）を除外するために抑制ルールを適用
- アクティベーションに進む前に、オーディエンス母集団のサイズを検証します
- 評価方法がフェーズ 2 で選択した宛先タイプに適合していることを確認します

**オプションが分岐する場所：**

**オプション A （ストリーミング宛先のアクティベーション）の場合：**
リアルタイムのメンバーシップの更新を配信するには、オーディエンスはストリーミングまたはエッジ評価を使用する必要があります。 セグメントルール式がストリーミング評価の対象であることを確認する – 時間ベースの集計関数、複数エンティティクエリおよびバッチ専用セグメントへの `inSegment()` 参照を避けます。

**オプション B （バッチ宛先のアクティベーション）の場合：**
あらゆる評価方法が機能します。 バッチ評価は、書き出し自体がスケジュールに従って実行されるので、最も一般的な選択です。 サンドボックスにバッチ評価スケジュールが存在することを確認するか、バッチ評価スケジュールを作成します。

**オプション C （複数宛先のアクティベーション）の場合：**
評価方法は、最も要求の厳しい宛先に対応する必要があります。 任意の宛先でストリーミングが必要な場合、オーディエンスはストリーミング評価を使用する必要があります。 すべての宛先がバッチの場合は、バッチ評価で十分です。

### フェーズ 2：宛先の設定

**アプリケーション関数：** RT-CDP：宛先設定

**設定内容：** オーディエンスが公開される外部の宛先への認証済み接続を確立します。 これには、カタログからの宛先の選択、認証資格情報の提供、宛先固有のパラメーター（ファイル形式、ストレージの場所、書き出しスケジュールなど）の設定が含まれます。 各宛先には、独自の接続設定が必要です。

**このフェーズの決定ポイント：**

>
>**決定：宛先のタイプ**

>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 増分書き出し | 前回の書き出し以降に新しく作成された選定および選定解除のみ | ファイル・サイズが小さい。デスティネーションでの処理が高速。状態を維持するにはデスティネーション・システムが必要 |
>| 完全な書き出し | 各実行でのオーディエンススナップショットの完了 | ファイルのサイズが大きい場合。出力先は毎回の全体像を把握。完全な置き換えを実行するシステムの場合は単純 |

**UI ナビゲーション：** 接続/宛先/カタログ/[ 宛先を選択 ]/設定

**キー設定の詳細：**

- 宛先カタログを参照し、ターゲット宛先を選択する
- 認証資格情報を入力して、接続をテストします。
- バッチ宛先の場合：ファイル形式（CSV、JSON、Parquet）、圧縮、ファイル命名テンプレート、書き出しスケジュールを設定します
- ストリーミング宛先の場合：接続は通常、OAuth 認証フローを介して確立されます
- アクティベーションに進む前に、宛先接続のステータスがアクティブと表示されていることを確認します

**オプションが分岐する場所：**

**オプション A （ストリーミング宛先のアクティベーション）の場合：**
カタログからストリーミングの宛先を選択します（Advertisingまたはソーシャルカテゴリ）。 OAuth 認証フローを完了します。 認証が確認されると、接続をアクティブ化する準備が整います。

**オプション B （バッチ宛先のアクティベーション）の場合：**

**オプション C （複数宛先のアクティベーション）の場合：**
各宛先に対して、このフェーズを繰り返します。 各接続は独立しています。ストリーミング宛先とバッチ宛先が混在する場合もあります。 各接続の認証と設定を文書化して、操作上の参考にします。

### フェーズ 3:Audience Activation

**アプリケーション関数：** RT-CDP:Audience Activation

**設定内容：** アクティベーションデータフローを作成して、評価されたオーディエンスを設定された宛先に公開します。 これには、アクティブ化するオーディエンスの選択、プロファイル属性の宛先フィールドへのマッピング、書き出しスケジュールの設定が含まれます。 アクティベーションデータフローは、ソースオーディエンスをターゲット宛先に接続し、継続的なデータ配信を管理します。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：属性マッピング**
>
>アクティベーションに含める必要があるプロファイル属性
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| ID フィールドのみ | 宛先はプロファイルのみを一致させる必要があります（広告プラットフォームのオーディエンスメンバーシップなど） | 最小限のデータ転送、最もプライバシーに配慮、広告プラットフォームのターゲティングと抑制に典型的 |
>| ID とプロファイル属性 | 宛先には、パーソナライゼーションやセグメント化のためのエンリッチメント属性（CRM 同期、パートナー共有など）が必要です | より多くのデータが転送されます。各属性についてガバナンスレビューが必要です。データ使用ラベルを宛先マーケティングアクションと照合します |
>| ID と計算属性 | 派生スコアまたは集計（ライフタイム値層、パートナータゲット設定の傾向スコアなど）からの宛先のメリット | 計算属性を設定する必要があります。ダウンストリームシステム用の高価値のエンリッチメントです。 |

>オーディエンスデータはどのくらいの頻度で書き出す必要がありますか？

>| 日次 | 標準の更新頻度：ほとんどのバッチの使用例 | 鮮度と処理コストのバランスを取る（最も一般的なデフォルト） |

>| オンデマンド（アドホック） | 1 回限りのエクスポートまたは緊急のサイクル外アクティベーション | 手動のトリガーではスケジュールがバイパスされるので、テストや緊急のキャンペーンニーズに役立つ |

**UI ナビゲーション：** 接続/宛先/参照/[ 宛先を選択 ]/オーディエンスをアクティブ化

**キー設定の詳細：**

- 宛先に対してアクティブ化するオーディエンスを 1 つ以上選択
- プロファイル属性と ID フィールドの宛先固有のフィールドへのマッピング
- ストリーミング宛先の場合：ID 名前空間マッピングを確認します（例：ハッシュ化されたメールを Facebook の extern_id に対して）
- バッチ宛先の場合：書き出しスケジュールを設定し、増分または完全書き出しモードを選択して、最初の書き出し日を設定します
- 公開する前に、アクティブ化の概要を確認して、すべてのマッピングとスケジュールを確認します

**オプションが分岐する場所：**

**オプション A （ストリーミング宛先のアクティベーション）の場合：**
オーディエンスを選択し、ID 名前空間を宛先 ID フィールドにマッピングします。 アクティベーションは公開直後に開始されます。オーディエンスメンバーシップの変更は、ほぼリアルタイムで宛先にストリーミングされます。 書き出しスケジュールは必要ありません。アクティベーションは継続的に行われます。

**オプション B （バッチ宛先のアクティベーション）の場合：**
オーディエンスを選択し、プロファイル属性をマッピングし、書き出しスケジュールを設定します。 増分エクスポートモードと完全エクスポートモードのどちらかを選択します。 オプションで、通常のスケジュールの範囲外で即時配信するアドホック書き出しをトリガーします。

**オプション C （複数宛先のアクティベーション）の場合：**
各宛先に対してアクティベーションワークフローを繰り返します。 宛先ごとに異なる属性マッピングを持つ複数の宛先に対して、同じオーディエンスをアクティブ化できます。 例えば、ハッシュ化されたメールのみを広告プラットフォームに送信し、人口統計学的属性を CRM に含めます。

### フェーズ 4：ガバナンスの検証

**アプリケーション機能：** RT-CDP：同意とガバナンスの適用

**設定内容：** アクティベーションの前と最中に、ガバナンスポリシーと同意環境設定が正しく適用されていることを検証します。 このフェーズでは、制限されたデータ（PII、機密属性）が許可されていない宛先に送信されず、有効な同意のないプロファイルがアクティベーションから除外されます。 ガバナンスの適用はアクティベーション時に自動的に行われますが、プロアクティブな検証によって、ブロックされたアクティベーションやコンプライアンス違反を防ぐことができます。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：ガバナンス実施アプローチ**
>
>ガバナンスコンプライアンスはいつ検証されるべきですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| プロアクティブ（アクティベーション前） | すべての実装に推奨（特に、外部のサードパーティシステムをアクティブ化する場合） | フィールドマッピングを設定する前に、データ使用ラベルとポリシーを確認します。アクティベーションの失敗を防ぎ、最初からコンプライアンスに準拠できるようにします。 |
>| 事後対応（アクティベーション時） | ガバナンスポリシーが既によく確立されており、チームがコンプライアンスに自信がある場合 | Platform は、アクティベーション時にポリシーを自動的に適用します。違反では、アクティベーションがブロックされたり、制限付き属性が除外されたりします。ポリシーが適切に理解されていない場合、予期しないエラーが発生する可能性があります。 |

>[!NOTE]
>
>**決定：同意フィルタリング**
>
>同意環境設定はアクティベーションにどのように影響しますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 同意適用が有効 | 同意が法的に必要な広告やマーケティングの宛先に対してアクティブ化する際に必要です | 有効な同意のないプロファイル（consents.marketing.{channel}.val = &#39;y&#39;）は、アクティベーションから自動的に除外されます。同意データを取り込む必要があります |
>| 同意フィルターなし | データ転送に同意が適用されない場合の、内部使用の宛先または Analytics の書き出し | データの使用が、同意を必要とするマーケティングアクションを構成しない場合にのみ適切です。法務/プライバシーチームにお問い合わせください。 |

**UI ナビゲーション：** プライバシー/ポリシー/同意ポリシー（同意の確認用）、データガバナンス/ポリシー（データ使用ポリシーの確認用）

**キー設定の詳細：**

- 宛先に関連付けられたマーケティングアクションに対してガバナンスポリシーがアクティブであることを確認します（広告プラットフォームの場合は「サードパーティに書き出し」など）

## 実装に関する考慮事項

実装前と実装中に、次の考慮事項を確認してください。

### ガードレールと制限

- **セグメント定義制限：** サンドボックスあたり最大 4,000 個のセグメント定義 – [&#x200B; セグメント化ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/guardrails)
- **宛先ごとのデータフロー：** 宛先接続あたり最大 100 個のデータフロー – [&#x200B; 宛先ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/destinations/guardrails)
- **バッチ書き出しファイルサイズ：** ファイルベースの宛先には、書き出しファイルサイズの上限があります。大きなオーディエンスは、複数のファイルに自動的に分割されます
- **ストリーミング宛先スループット：** 1 秒あたりのスループットの制限は、各宛先パートナーによって設定されます。大量のオーディエンスの変更は調整される場合があります
- **バッチ評価処理能力：** デフォルトでは、セグメント評価ジョブあたり最大 2,400 万プロファイルです
- **オーディエンス構成：** 1 つのキャンバスにつき最大 10 個のコンポジションブロック。構成されたオーディエンスは、バッチ評価のみです
- **ID グラフ：** グラフあたり最大 50 個の ID — [ID サービスガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/identity/guardrails)
- **計算属性：** サンドボックスあたり最大 25 計算属性 – [&#x200B; 計算属性ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/computed-attributes/overview#guardrails)
- **アクティベーションガードレールの概要：**&#x200B;[&#x200B; アクティベーションガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/destinations/guardrails)

### よくある落とし穴

- **不適格なルールロジックを持つストリーミングセグメント：** 複雑な集計関数または複数エンティティクエリを使用するオーディエンスを定義し、ストリーミング評価を期待しています。 これらのセグメントは、バッチ評価にサイレントにフォールバックし、予期しない待ち時間を引き起こします。 防止：オーディエンスを定義する前に、ストリーミングの実施要件を確認します。

- **同意のフィルタリングが原因で書き出されたプロファイルがゼロ：** オーディエンス内のすべてのプロファイルに有効な同意値が不足しており、アクティベーション時にオーディエンス全体が除外されます。 防止：同意データの取り込みが動作可能であり、アクティブ化する前に同意フィールドに値が入力されていることを確認します。

- **ガバナンスポリシーによりアクティベーションが予期せずブロックされる：** スキーマフィールドのデータ使用ラベル トリガーポリシー違反によりアクティベーションが妨げられる。 防止：フィールドマッピングを設定する前に、ガバナンスコンプライアンスをプロアクティブに評価します（フェーズ 4）。 スキーマから継承されるラベルをレビューします。

- **宛先資格情報の有効期限：** OAuth トークンまたは API キーの有効期限が切れると、即時のアラートが表示されず、アクティベーションが失敗します。 防止：宛先のアクティブ化の失敗に関するアラートを設定し、資格情報のローテーションスケジュールを確立します。

- **ID 名前空間が一致しません：** アクティベーションマッピングで設定された ID 名前空間が、宛先が期待するものと一致しません（例えば、宛先が SHA-256 でハッシュ化されたメールを必要とする場合にプレーンテキストメールを送信する）。 防止：マッピングを設定する前に、必要な ID 形式に関する宛先ドキュメントを確認してください。

- **書き出しエラーを引き起こすフィールドマッピングエラー：** 互換性のないデータタイプまたは形式を使用して、プロファイル属性を宛先フィールドにマッピングします。 防止策：最初に、少数のオーディエンスでアクティベーションをテストし、実稼動オーディエンスにスケーリングする前に、マッピングエラーのデータフロー実行の詳細を確認します。

- **RT-CDP と宛先の間のオーディエンス母集団のずれ** RT-CDP のオーディエンスサイズは、ID の照合の違い、宛先の重複排除ロジック、タイミングにより、宛先のカウントと一致しません。 これは予期される動作です。防止：宛先ごとの予期される一致率ベンチマークを文書化し、重大な逸脱を監視します。

### ベストプラクティス

- **テストオーディエンスから開始する：** 実稼動オーディエンスをアクティブ化する前に、新しい各宛先に対して、理解が深い小さなテストオーディエンスをアクティブ化します。 フィールドマッピング、ID 照合および配信指標をテストオーディエンスで検証します。

- **レイヤーガバナンスの早期適用：** 宛先のアクティベーションを設定する前に、データ使用ラベルを適用し、ガバナンスポリシーを設定します。 これにより、ブロックされたアクティベーションを防ぎ、最初からコンプライアンスに準拠できるようにします。

- **バッチ宛先での増分エクスポートの使用：** 増分エクスポートは、ファイルサイズを削減し、宛先での処理を高速化し、ストレージコストを最小限に抑えます。 宛先システムで完全なオーディエンスの置き換えが必要な場合にのみ、完全な書き出しを使用します。

- **命名規則の標準化：** 組織全体で、オーディエンス、宛先接続、データフローの一貫した命名を確立します。 名前にユースケース、宛先、評価方法を含めます（例：「Suppression_ExistingCustomers_Facebook_Streaming」）。

- **マッチ率の監視：** 各宛先で一致したプロファイルに対する、RT-CDP から書き出されたプロファイルの割合を追跡します。 一致率が大幅に低下した場合は、ID 解決の問題、資格情報の問題、宛先 API の変更が発生している可能性があります。

- **宛先間で抑制を調整：** 抑制オーディエンスを使用する場合（獲得キャンペーンから既存の顧客を除外するなど）、一貫性を維持するために、関連するすべての広告プラットフォームに同時に抑制オーディエンスをアクティブ化します。

- **非アクティブなアクティベーションのレビューとプルーフ：** 宛先データフローを定期的にレビューし、不要になったオーディエンスを非アクティブ化します。 非アクティブなアクティベーションは、ライセンス容量を消費し、監視オーバーヘッドを追加します。

### トレードオフの決定

>[!NOTE]
>
>**トレードオフ：オーディエンスの鮮度とセグメント化の柔軟性**
>
>ストリーミング評価では、ほぼリアルタイムのオーディエンス更新が可能ですが、使用できるセグメントルールの関数は制限されます。 バッチ評価では完全なセグメントルール関数セットがサポートされますが、待ち時間（分ではなく時間）が発生します。
>
>- **ストリーミングのお気に入り：** オーディエンスメンバーシップが数分以内に宛先に反映される必要があるリアルタイムのユースケース（アクティブなキャンペーン抑制、リアルタイムリターゲティング）
>- **バッチ好み：** 集計関数、マルチエンティティクエリ、または大きな時間枠条件（ライフタイム値層、マルチタッチ行動セグメント）を必要とする複雑なオーディエンスロジック
>- **推奨事項：** リアルタイムの鮮度がビジネス価値を促進するストリーミング宛先に対してアクティブ化されたオーディエンスのストリーミング評価を使用します。 ファイルベースの宛先に対してアクティブ化された複雑なオーディエンスの場合、または毎日の更新で十分な場合は、バッチ評価を使用します。 両方を必要とするオーディエンスの場合は、2 つのバージョン（リアルタイムアクティベーションを可能にするシンプルなストリーミングセグメントと、定期的なエンリッチメントを可能にする包括的なバッチセグメント）の作成を検討します。

>[!NOTE]
>
>**トレードオフ：最小限のデータ書き出しとリッチ属性マッピング**
>
>ID フィールド（ハッシュ化されたメール、デバイス ID）のみを書き出すことで、データの漏洩を最小限に抑え、ガバナンスを簡素化できます。 追加のプロファイル属性（人口統計、価値層、エンゲージメントスコア）を書き出すと、宛先は豊かになりますが、ガバナンスの複雑さとデータの責任が増します。
>
>- **書き出しの支持を最小限に抑える：** プライバシーファーストのアプローチ、ガバナンスリスクの軽減、フィールドマッピングの簡素化。ほとんどの広告プラットフォームのターゲティングと抑制のユースケースに十分です
>- **リッチエクスポートのお気に入り：** 宛先でのダウンストリームパーソナライゼーション、CRM エンリッチメント、属性レベルの詳細が必要なパートナーデータ共有
>- **推奨事項：** 広告宛先の ID のみの書き出しを最小限に抑えるデフォルト。 プロファイル属性を追加するのは、宛先のユースケースで特に必要とされ、ガバナンスレビューでコンプライアンスが確認された場合のみです。 計算属性を使用して、生の PII ではなく、機密性の低い集計されたエンリッチメント値を提供します。

>[!NOTE]
>
>**トレードオフ：宛先ごとの単一オーディエンスと統合された複数オーディエンスデータフローの比較**
>
>各オーディエンスを個別のデータフローとしてアクティブ化すると、分離ときめ細かい監視が可能になります。 宛先に対して 1 つのデータフローを通じて複数のオーディエンスをアクティブ化すると、管理が簡素化されますが、分離が軽減されます。
>
>- **個別のデータフローにより優遇：** 独立した監視、容易なトラブルシューティング、他のユーザーに影響を与えることなく個々のオーディエンスのアクティベーションを一時停止する機能
>- **統合されたデータフローのメリット：** 維持する接続が少なくなり、認証情報の管理が合理化され、運用オーバーヘッドが削減される
>- **推奨事項：** 同じ宛先に関連する複数のオーディエンス（例：Facebook へのすべての抑制セグメント）をアクティブ化する場合は、統合されたデータフローを使用します。 オーディエンスの SLA や属性マッピングが異なる場合や、エラーの分離が重要な場合は、別のデータフローを使用します。
