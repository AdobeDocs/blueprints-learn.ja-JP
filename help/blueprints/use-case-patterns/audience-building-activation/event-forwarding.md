# イベントの転送

このガイドでは、Edge Networkを使用したサーバーサイドイベント転送の実装 [!DNL Adobe Experience Platform] ついて説明します。 Edge Networkを介して収集されたリアルタイムイベントデータをAdobe以外の宛先（サードパーティの分析プラットフォーム、クラウドストレージエンドポイント、広告ネットワーク、カスタム Web フックなど）に配布する必要があるソリューションアーキテクト、マーケティング技術者、実装エンジニア向けに設計されています。

イベント転送を設定するためのすべての実行可能なアプローチを紹介し、それらの間のトレードオフを説明し、詳細な手順ガイダンス [!DNL Adobe Experience League] 関するドキュメントへのリンクを示します。

## ユースケースの概要

[!DNL Adobe Experience Platform] Web SDK、Mobile SDK、Server API を使用して行動データを収集する組織は、多くの場合、同じイベントストリームをAdobe以外のシステム（[!DNL Google Analytics] や [!DNL Snowflake] などの分析プラットフォーム、コンバージョントラッキング用の広告ネットワーク、長期ストレージ用のデータウェアハウス、カスタム内部サービス）と共有する必要があります。 従来、これによりクライアントサイドのタグが急増する必要があり、ページの重みが増し、待ち時間が発生し、プライバシーとガバナンスのリスクが生じます。

イベント転送は、Edge Networkでサーバーサイドを操作することでこれを解決します。 訪問者インタラクションが Web SDKまたは Server API を通じてイベントをトリガーすると、そのイベントはデータストリームを通じてEdge Networkにルーティングされます。 専用のイベント転送プロパティに設定されたイベント転送ルールは、受信イベントデータを評価し、設定された 1 つ以上の宛先に選択的に転送します。 このサーバーサイドのアプローチにより、クライアントサイドのタグの増加を減らし、ページパフォーマンスを向上させ、データガバナンスを一元化し、Adobe エコシステムから残るデータを正確に制御できるようになります。

このパターンのターゲットオーディエンスには、[!DNL Adobe Experience Platform] Web SDKまたは Server API を既にデータ収集のためにデプロイ（またはデプロイ予定）していて、クライアントサイドのJavaScript タグを追加せずにAdobe以外のエンドポイントにイベントデータを配信することで投資を拡大したい組織が含まれます。

## 主なビジネス目標

このユースケースパターンでは、以下のビジネス目標がサポートされています。

### データ品質とガバナンスの向上

正確なターゲティング、無駄の削減、信頼性の高い分析のために、クリーンで完全な準拠データを確保します。 イベント転送は、サーバーサイドでデータ配信を一元化することで、どのようなデータが外部システムと共有されるかを一元的に制御できるようになり、データ漏洩のリスクを軽減できるほか、データが [!DNL Adobe] Edge Networkに送信される前にガバナンスポリシーが適用されることになります。

**KPI:** 効率、コスト削減

詳しくは、[&#x200B; データ品質とガバナンスの向上 &#x200B;](../../business-objectives/cost-efficiency/improve-data-quality-governance.md) を参照してください。

### マーケティングテクノロジーの統合と最新化

統合されたスケーラブルなプラットフォームに移行することで、ツールの断片化と技術的負担を軽減します。 イベント転送を使用すると、組織は複数のクライアントサイドベンダータグを単一のサーバーサイドのデータ配信メカニズムに置き換えて、ページ負荷のオーバーヘッドを削減し、テクノロジースタックをシンプル化できます。

**KPI:** コスト削減、効率性、市場投入までのスピード

詳しくは、[&#x200B; マーケティングテクノロジーの統合と最新化 &#x200B;](../../business-objectives/cost-efficiency/consolidate-modernize-marketing-technology.md) を参照してください。

## 戦術的な使用例

次に、このユースケースパターンが適用される一般的な戦術シナリオを示します。

- **サードパーティ分析の強化** - クライアントサイドタグを追加せずに、ページビュー、クリックおよびコンバージョンイベントをリアルタイムで [!DNL Google Analytics]、[!DNL Snowflake] またはその他の分析プラットフォームに転送します
- **Advertising コンバージョントラッキング** – 購入イベントとリードジェネレーションイベントを [!DNL Meta] Conversions API、[!DNL Google Ads]、[!DNL TikTok] または [!DNL Snap] に送信して、サーバーサイドのコンバージョンの測定と最適化を行います
- **データウェアハウスストリーミング** – 生のイベントデータをクラウドデータウェアハウス（[!DNL Google BigQuery]、[!DNL Amazon S3]、[!DNL Azure Event Hubs]）にルーティングして、長期保存とオフライン分析を実現します
- **カスタム Webhook 統合** - フィルタリングまたは変換されたイベントデータを、HTTP エンドポイントを介して、内部マイクロサービス、CRM システムまたはパートナープラットフォームに転送します
- **タグの軽減とページパフォーマンスの向上** – 複数のクライアントサイドベンダーJavaScriptタグを、1 つの Web SDK実装と、サーバーサイドイベント転送ルールに置き換えることで、ページの重みを軽減し、Core Web Vitalsを向上させます
- **プライバシー準拠のデータ共有** - イベントデータをサードパーティと共有する前に、データフィルタリングとフィールドレベルの編集規則をサーバーサイドで適用し、PII が外部システムに到達する前に削除またはハッシュ化されるようにします
- **マルチクラウドイベント配信** – 同じイベントストリームを、単一のサーバーサイドのルールセットから複数の宛先（分析、広告、Data Warehouse など）に同時に転送します
- **リアルタイムの不正シグナル転送**：価値の高いトランザクションイベントを不正検出システムに転送して、リアルタイムのリスクスコアリングとアラートを実現

## 主要業績評価指標

次の KPI は、このユースケースパターンの成功を測定するのに役立ちます。

- **ページ読み込み時間の短縮** - クライアントサイドタグをサーバーサイドイベント転送に移行した後の、ページ読み込み速度とCore Web Vitalsの改善を測定しました
- **データ配信成功率** - エラーやタイムアウトなしで宛先エンドポイントに正常に転送されたイベントの割合
- **タグ数の削減** — サーバーサイドに相当する機能を実装した後に削除された、クライアントサイドのベンダータグの数
- **データの鮮度/待ち時間** - クライアントでイベントが発生してから宛先エンドポイントにイベントが到着するまでの時間（ターゲット：秒未満）
- **ガバナンスコンプライアンス率** - サーバーサイドフィルタリングルールを通過する送信データ共有の割合。PII や制限されたデータが権限のない宛先に到達しないようにします
- **運用効率** - クライアントサイドのタグのデプロイメントの管理と、タグの競合のトラブルシューティングに費やす開発者の時間を短縮

## 実装オプション

ここでは、イベント転送を実装するために使用できるアプローチと、適切なオプションを選択する際のガイダンスについて説明します。

### オプション A：拡張機能ベースのイベント転送

**最適な対象：** 適切にサポートされている宛先プラットフォーム（[!DNL Meta]、[!DNL Google]、[!DNL AWS]、[!DNL Azure]、[!DNL Snowflake] など）を使用するチーム データ収集カタログで使用できる事前定義済みのイベント転送拡張機能がある。

**仕組み：**

拡張機能ベースのイベント転送では、Adobeまたはサードパーティパートナーが管理する事前定義済みの統合機能を活用します。 各拡張機能は、特定の宛先用に専用に構築され、認証、ペイロードの書式設定およびエンドポイント通信を処理します。 実装者は、イベント転送プロパティに拡張機能をインストールし、認証資格情報を設定して、XDM データ要素を拡張機能のアクションフィールドにマッピングするルールを作成します。

このアプローチでは、拡張機能が宛先の API 要件を抽象化するので、カスタム開発を最小限に抑えることができます。 例えば、[!DNL Meta] Conversions API 拡張機能は、XDM コマースイベントを想定され [!DNL Meta] 形式に変換し、PII フィールドのハッシュ処理、重複排除パラメーターおよびアクセストークン管理を行います。 同様に、[!DNL Google Cloud Platform] または [!DNL AWS] 拡張機能は、それぞれのクラウドサービスの認証およびペイロードの書式設定を処理します。

トレードオフは、拡張機能の可用性によって、サポートされる宛先が決まります。 ターゲット宛先用の拡張機能が存在しない場合は、代わりにオプション B （Custom Webhook）を使用する必要があります。

**主な考慮事項：**

- 拡張機能の可用性はさまざまです。計画する前に、[&#x200B; データ収集拡張機能カタログ &#x200B;](https://experienceleague.adobe.com/en/docs/experience-platform/tags/extensions/server/overview) を確認してください
- 拡張機能は、Adobeまたはパートナーによって管理されます。更新によって、ルールの調整が必要な重大な変更が生じる場合があります
- 一部の拡張機能は、特定のイベントタイプのみをサポートするか、特定の XDM フィールドマッピングを必要とします
- 拡張機能は、設定 UI 内で認証と資格情報の管理を処理します

**メリット：**

- サポートされる宛先の実装までの最速の時間
- 事前定義済みのペイロードのフォーマット設定により、マッピングエラーを軽減します
- 拡張機能で処理される認証と資格情報の管理
- Adobeまたは認定パートナーが維持更新
- カスタムコードの削減とメンテナンス負担の軽減

**制限事項：**

- 拡張機能が使用可能な宛先に制限
- カスタム Web フックと比較して、ペイロードのカスタマイズの柔軟性が低い
- 拡張機能の更新には、ルールの再設定が必要になる場合があります
- 一部の拡張機能では、すべての宛先 API 機能がサポートされているとは限りません

### オプション B：カスタム Webhook （Fetch API）イベント転送

**最適な対象：** 事前定義済みの拡張機能のない宛先にイベントを転送する必要があるチーム、または HTTP リクエストペイロード、ヘッダーおよび認証メカニズムを完全に制御する必要があるチーム。

**仕組み：**

カスタム Webhook ベースのイベント転送は、（デフォルトで含まれる） [!DNL Adobe Cloud Connector] 拡張機能を使用して、任意のエンドポイントに対して任意の HTTP リクエストを行います。 この実装担当者は、受信した XDM イベントから値を抽出して変換するデータ要素を定義したあと、Cloud Connector の「Make Fetch Call」アクションタイプを使用してルールアクションを設定します。 このアクションを使用すると、HTTP メソッド、URL、ヘッダーおよびリクエスト本文を完全に制御できます。

リクエスト本文は、通常、データ要素とカスタムコードを使用して構築され、宛先の API 仕様に従ってペイロードをフォーマットします。 このアプローチは、HTTP でアクセス可能な任意のエンドポイント（REST API、Webhook、クラウド関数、内部サービス）をサポートしており、最も柔軟なオプションとなっています。

トレードオフは、より高い実装作業と継続的なメンテナンスです。 実装者は、宛先 API を理解し、認証を手動で処理し（認証ヘッダーの設定、トークン更新の管理など）、宛先 API が進化した場合はペイロード形式を維持する必要があります。

**主な考慮事項：**

- 宛先 API の仕様（HTTP メソッド、URL 構造、ペイロード形式、認証）を理解している必要があります
- 認証資格情報は、データ要素またはシークレットで手動で管理する必要があります
- ペイロード変換（ハッシュ、エンコーディング、再構築）にカスタムコードが必要な場合があります
- 宛先 API の変更時に自動更新されない – 手動メンテナンスが必要
- データ収集の秘密鍵機能は、API キーとトークンを安全に保存できます

**メリット：**

- HTTP でアクセス可能な任意のエンドポイントをサポート – 拡張機能の依存関係なし
- リクエストペイロード、ヘッダーおよび認証を完全に制御
- 内部サービス、カスタム API、またはニッチプラットフォームに転送可能
- カスタムコードを使用して複雑なペイロード変換を有効にします
- カスタムコード内に再試行ロジックとエラー処理を実装可能

**制限事項：**

- 初期の実装作業の増加
- ペイロードの形式と認証に関する継続的なメンテナンス責任
- 事前定義済みのエラー処理や証明書のローテーションなし – 手動で実装する必要がある
- HTTP プロトコルおよび宛先 API の詳細に関する開発者の専門知識が必要です

### オプション C：ハイブリッド（拡張機能+ カスタム Web フック）

**以下に最適：** 組織がイベントを複数の宛先に転送する場合、一部の宛先には事前定義済みの拡張機能があり、他の宛先にはカスタム統合が必要です。

**仕組み：**

ハイブリッドアプローチは、適切にサポートされた宛先に対する拡張機能ベースの転送と、拡張機能がない宛先に対するカスタム webhook アクションを組み合わせます。 単一のイベント転送プロパティには複数のルールが含まれています。一部は拡張機能アクション（広告コンバージョントラッキング用の [!DNL Meta] Conversions API 拡張機能など）を使用し、他は Cloud Connector の取得アクション（内部データレイクエンドポイントへの転送など）を使用しています。

このアプローチは、不要なカスタム開発を最小限に抑えながら、カバレッジを最大化します。 各ルールは個別に動作するので、拡張機能ベースのルールには事前定義済みのペイロード形式というメリットがありますが、カスタムルールは完全な柔軟性を維持します。

**主な考慮事項：**

- プロパティの複雑さは、ルールと宛先の数に伴って増加します
- テストとデバッグには、拡張機能ベースのルールとカスタムルールで異なるアプローチが必要になる場合があります
- 公開の変更は、プロパティのすべてのルールに影響します。変更を安全にステージングするには、ライブラリと環境を使用します
- 拡張機能ベースのアクションとカスタムアクションを区別するために、明確な命名規則を使用してルールを整理することを検討してください

**メリット：**

- 様々な宛先タイプにわたる最高のカバレッジ
- 事前に構築された拡張機能を利用可能な場所で活用し、作業を軽減
- カスタム宛先の柔軟性を維持
- 単一イベント転送プロパティは、すべての転送ロジックを管理します

**制限事項：**

- 複数のルールタイプによるプロパティの複雑さの増加
- 混在メンテナンスモデル – 一部のルールは拡張機能を介して自動更新され、その他のルールは手動での維持が必要です
- デバッグするには、拡張機能の設定とカスタム取得呼び出しパターンの両方に関する知識が必要です

### 適切なオプションを選択

まず、ターゲットの宛先を調べ、それぞれに事前定義済みのイベント転送拡張機能が存在するかどうかを確認します。

1. **すべての宛先に拡張機能がある場合** — オプション A を選択します。これにより、メンテナンスの負担が最も少なく、最速の実装が可能になります。 拡張機能は、認証、ペイロード形式、および API バージョン管理を処理します。

2. **宛先に拡張機能がない場合、または完全なペイロード制御が必要な場合** — オプション B を選択します。クラウドコネクタ拡張機能を使用して、任意のエンドポイントにカスタム HTTP リクエストを行います。 これは、複雑な変換やカスタムハッシュを適用したり、内部サービスに送信したりする必要がある場合にも適しています。

3. **サポートされている宛先とサポートされていない宛先が混在している場合** — オプション C を選択します。[!DNL Meta]、[!DNL Google]、[!DNL AWS] などのプラットフォームには拡張機能を、その他のすべてに対してはカスタム Web フックを使用します。 これは、様々な分析や広告スタックを持つ組織で最も一般的な実稼動シナリオです。

### フェーズ 3：イベントルールの定義

**Application Function:** AEP：イベントルール定義、AEP：宛先マッピング

**設定する内容：** 受信イベントデータを評価し、転送するイベントをフィルタリングするための条件を適用し、データを宛先エンドポイントに送信するアクションを定義するルール。 各ルールは、条件（実行するタイミング）とアクション（実行するアクション）で構成されます。 データ要素は、ルール条件やアクション設定で使用するために、XDM イベントペイロードから値を抽出して変換します。

**このフェーズの決定ポイント：**

>[!NOTE]

>| --- | --- | --- |

>| データ要素とカスタムコードの組み合わせ | 直接マッピングされるフィールドもあれば、変換が必要なフィールドもあります | 単純なマッピングにはデータ要素を使用し、複雑な変換にはカスタムコードブロックを使用します。 メンテナンス性と柔軟性のバランス |

>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| PII フィールドを完全に除外 | 宛先は PII を必要とせず、ガバナンスポリシーは共有を制限します | 転送されたペイロードから PII フィールドを省略するようにルールを設定します。 プライバシーコンプライアンスのための最もシンプルなアプローチ。 |
>| 転送前に PII フィールドをハッシュ化 | 宛先には、ハッシュ化された識別子が必要です（例えば、[!DNL Meta] では、Conversions API 用に SHA-256 でハッシュ化されたメールが必要です） | カスタムコードデータ要素を使用して、SHA-256 ハッシュを適用します。 一部の拡張機能は自動的にハッシュを処理します。 |
>| 契約上の基礎を含むフォワード PII | 宛先にはデータ処理契約があり、PII を共有するための法的根拠があります | データ使用ラベルとガバナンスポリシー（S3）で共有が許可されていることを確認します。 法的根拠を文書化します。 |

**UI ナビゲーション：** [!DNL Experience Platform]/データ収集/イベント転送/プロパティを選択/データ要素/ルール

**キー設定の詳細：**

- **データ要素** パスプレフィックス `arc.event.xdm.` を使用して、受信 XDM イベントを参照します（例えば、ページ URL の `arc.event.xdm.web.webPageDetails.URL`）
- **ルール条件** データ要素の値を評価して、ルールを実行するかどうかを決定します
- **ルールアクション** 拡張機能固有のアクション（オプション A の場合）または Cloud Connector の「フェッチ呼び出しを行う」アクション（オプション B の場合）を使用して、データを宛先に送信します
- 各ルールには複数のアクションを含めることができ、1 つのイベントを複数の宛先に転送できます
- 同じイベントに対して複数のルールが実行される可能性がある場合、ルールの順序を使用して、評価のシーケンスを制御します
- 実稼動環境に公開する前に、開発環境でルールを十分にテストします

**オプションが分岐する場所：**

**オプション A （拡張機能ベース）の場合：**

宛先拡張機能の事前定義済みアクションタイプを使用して、ルールアクションを設定します。 例えば、[!DNL Meta] Conversions API 拡張機能には、XDM フィールドを [!DNL Meta] のイベントパラメーター（event_name、event_time、user_data、custom_data）にマッピングする「イベントを送信」アクションが用意されています。 この拡張機能は、ペイロードの書式設定、ハッシュ化、API 通信を処理します。

**オプション B （カスタム Webhook）の場合：**

Cloud Connector 拡張機能の「Make Fetch Call」アクションを使用して、ルールアクションを設定します。 宛先 URL、HTTP メソッド（通常は POST）、リクエストヘッダー（認証を含む）を指定し、データ要素やカスタムコードを使用してリクエスト本文を作成します。 宛先 API で想定されるペイロード形式を正確に一致させる責任があります。

**オプション C （ハイブリッド）の場合：**

宛先ごとに個別のルールを作成します。 拡張機能ベースのルールでは、拡張機能のアクションタイプを使用します。カスタムルールでは、Cloud Connector のフェッチ呼び出しを使用します。 すべてのルールは同じプロパティに共存し、各受信イベントに対して個別に評価されます。

## 実装に関する考慮事項

このセクションでは、実装時に念頭に置くべきガードレール、一般的な落とし穴、ベストプラクティス、トレードオフの決定について説明します。

### ガードレールと制限

- イベント転送は、Edge Networkでリアルタイムにイベントを処理します。デフォルトでは、失敗した配信のバッチモードや再試行キューはありません
- Edge Networkのレート制限は、データストリームごとに処理される合計イベントボリュームに適用されます – [Edge Network ガードレール &#x200B;](https://experienceleague.adobe.com/en/docs/experience-platform/edge-network/guardrails)
- イベント転送ルールは、サーバーサイドで実行され、クライアントサイドのリソース（DOM、cookie、localStorage）にアクセスできません
- イベント転送ルールのカスタムコードは、サンドボックス化された環境で実行されます。すべてのブラウザーJavaScript API が使用できるわけではありません
- Cloud Connector の取得の呼び出しにはタイムアウト制限があり、応答が遅い宛先がタイムアウトの原因となる可能性があります
- イベント転送は、Edge Networkの地理的ルーティングの対象となり、最も近いEdgeの場所で処理されます
- 転送されたリクエストの最大ペイロードサイズは、Edge Networkの制限によって制御されます

### よくある落とし穴

- **フィルタリングなしですべての XDM フィールドを転送：** XDM イベントペイロード全体を、少数のフィールドのみが必要な宛先に送信すると、帯域幅が浪費され、宛先コストが増加し、誤って PII を共有する可能性があります。 転送ルールでは、常に必須フィールドのみをマッピングします。

- **秘密鍵で認証情報を保護しない：** データ要素またはルールアクションで API キーまたはトークンをハードコーディングすると、セキュリティリスクが発生します。 資格情報を安全に保存し、ルールで参照するには、常にデータ収集シークレット機能を使用します。

- **宛先レート制限の無視：** サードパーティの宛先には、多くの場合、API レート制限があります。 イベントの量が宛先の処理能力を超えると、イベントがドロップされたり、API アクセスが制限されたりする場合があります。 レート制限に関する宛先ドキュメントを確認し、必要に応じてイベント量を減らすためのフィルタリングを実装します。

- **ステージング環境を使用せずに実稼動環境に直接公開：** ステージング環境をスキップすると、エラーは実稼動環境でのみ検出され、重要な宛先でデータが失われる可能性があります。 常に、最初にライブトラフィックを使用してステージングで検証します。

- **HTTP 応答コードを監視しない：** エラーなく起動するルールは、宛先がデータを正常に処理したことを保証しません。 宛先応答コードを監視し（イベント転送監視ツールで使用可能）、2xx 以外の応答を調査します。

- **XDM パス参照の設定に誤りがあります：** データ要素は、`arc.event.xdm.` プレフィックスを使用して受信イベントフィールドを参照します。 誤ったパス（ネストのレベルがない場合など）の場合は、エラーがスローされずに null 値がサイレントに生成される。 デバッガーを使用してデータ要素値を検証します。

### ベストプラクティス

- **1 つの宛先から開始して徐々に拡張** – 追加のルールと宛先を追加する前に、1 つの宛先でエンドツーエンドのイベント転送を検証します。 これにより、デバッグが簡素化され、インフラストラクチャの信頼性が構築されます。

- **一貫した命名規則の使用** – 宛先、イベントタイプ、環境を特定する明確な規則（「ルール：Meta – 購入イベント」、「DE：注文合計」など）を使用して、データ要素、ルール、ライブラリに名前を付けます。

- **プライバシーに対応したフィールドレベルのフィルタリングの実装** – 宛先が PII を適切に処理すると主張している場合でも、機密フィールドがEdge Networkを離れる前に、サーバーサイドのフィルタリングを適用して、機密フィールドを削除またはハッシュ化します。 これは、クライアントサイドタグよりもイベント転送を使用する場合のガバナンス上の主なメリットです。

- **設定のバージョン管理** - ライブラリ公開ワークフローを使用して、イベント転送設定のバージョン管理されたスナップショットを管理します。 監査とロールバックの目的で、各ライブラリバージョンに含まれる内容を文書化します。

- **Platform Debugger を使用したテスト** - [!DNL Adobe Experience Platform Debugger] 拡張機能は、クライアントサイド SDKからEdge Network処理に至るまでのイベントライフサイクルを可視化します。 開発およびトラブルシューティングの際に使用します。

- **イベント転送ルールと XDM スキーマの設計の調整** - イベント転送が既知の要件である場合、宛先に必要なフィールドを含めるように XDM スキーマとイベント分類を設計します。 デプロイメント後にスキーマの変更を調整すると、より混乱を招く。

### トレードオフの決定

>[!NOTE]
>
>**トレードオフ：拡張機能のシンプルさと Webhook の柔軟性の比較**
>
>事前定義済みの拡張機能は、実装の労力とメンテナンスを最小限に抑えますが、サポートされる拡張機能とペイロード形式に制限されます。 カスタム Webhook は完全な制御を提供しますが、より多くの開発と継続的なメンテナンスが必要です。
>
>- **拡張機能ベース（オプション A）のメリット：** 市場投入までのスピード、開発者依存の低減、自動資格情報管理、メンテナンスの低下
>- **Webhook ベース（オプション B）のお気に入り：** 完全なペイロード制御、任意の HTTP エンドポイントのサポート、カスタム変換ロジック、拡張機能リリースサイクルからの独立性
>- **推奨事項：** 利用可能で十分な場合は、拡張機能を使用します。 宛先に拡張機能がない場合、または拡張機能が必要な特定の API 機能をサポートしていない場合にのみ、カスタム Web フックにフォールバックします。 ハイブリッドアプローチ（オプション C）は、ほとんどの組織にとって実用的な選択です。

>[!NOTE]
>
>**トレードオフ：すべてのイベントを転送と選択的フィルタリングの比較**
>
>すべてのイベントを宛先に転送することで、完全性は保証されますが、データ量、宛先のコストおよびプライバシーのリスクが増加します。 選択的フィルタリングはノイズを減らしますが、価値のあるイベントを見逃すリスクがあります。
>
>- **すべてのイベントのお気に入りを転送：** データの完全性、シンプルさ、将来保証（後で必要に応じてデータが存在する）
>- **選択的なフィルタリングのメリット：** コスト効率、プライバシーリスクの軽減、宛先データのクリーン化、データ最小化原則への準拠
>- **推奨：** イベントタイプとビジネスの関連性に基づく選択フィルタリングをデフォルトで選択します。 各宛先が実際に必要とするイベントとフィールドのみを転送します。 これは、データ最小化原則（GDPR 第 5 条）に従っており、運用コストを削減します。

>[!NOTE]
>
>**トレードオフ：単一のプロパティと複数のプロパティの比較**
>
>単一のイベント転送プロパティは管理がより簡単ですが、すべてのルールが 1 つの公開ワークフローを共有することを意味します。 複数のプロパティを指定すると、分離は可能ですが、管理オーバーヘッドが増加します。
>
>- **単一のプロパティのメリット：** シンプルさ、統合公開、共有データ要素、容易なデバッグ
>- **複数のプロパティが有効：** チームレベルのアクセス制御、独立した公開サイクル、宛先エラーの分離
>- **推奨事項：** 多くの実装では、まず 1 つのプロパティを使用します。 異なるチームが異なる宛先統合を所有し、独立したリリースサイクルを必要とする場合、または規制要件によりデータフロー間の厳密な分離が必要な場合にのみ、複数のプロパティに分割します。
