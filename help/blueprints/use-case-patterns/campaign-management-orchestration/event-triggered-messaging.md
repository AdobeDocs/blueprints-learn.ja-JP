# イベントトリガーメッセージ

このガイドでは、[!DNL Adobe Journey Optimizer] （AJO）、[!DNL Real-Time Customer Data Platform] （RT-CDP）および [!DNL Adobe Experience Platform] （AEP）を使用したイベントトリガーメッセージングの包括的な実装ブループリントを提供します。 行動やシステムのイベントに応じてコンテキストに沿ったリアルタイムのメッセージを配信する必要がある、ソリューションアーキテクト、マーケティング技術者、実装エンジニア向けに設計されています。

このガイドを使用すると、何を設定するか、実装の選択肢が存在する場所、各決定を左右するトレードオフを理解できます。

このガイドでは、イベントの取り込みとジャーニーの作成からメッセージ配信とパフォーマンスレポートまで、ライフサイクル全体をカバーし、3 つの異なる実装オプションに関する詳細な決定ガイダンスを示します。

## ユースケースの概要

イベントトリガーメッセージは、リアルタイムの行動イベントまたはシステムイベントに応答してコンテキストメッセージを提供します。 事前に評価されたオーディエンスにスケジュールに従って送信するバッチアウトバウンドメッセージアクティベーションとは異なり、このパターンでは、条件を満たすイベント（買い物かごの放棄、参照セッション、フォーム送信、システムステータス変更など）をリッスンし、条件を評価してメッセージを配信するジャーニーにトリガープロファイルをすぐに入力します。

パターンは、AEPへのリアルタイムイベントストリーミング（Web SDK、Mobile SDKまたはサーバーサイド API 経由）、AJOでの単一イベントエントリを持つジャーニー、送信するかどうかと送信する内容を判断する条件評価ロジックに依存します。 通常、メッセージはトリガーイベントから数分以内に送信されるので、このパターンは、時間依存の、文脈的に関連性の高い通信に最適です。

組織はこのパターンを使用して顧客のアクションにリアルタイムで対応するため、スケジュールされたバッチ通信と比較して関連性が高まり、エンゲージメントとコンバージョン率が高くなります。 一般的なシナリオには、買い物かごの放棄された回復、購入後のフォローアップ、登録後のウェルカムメッセージ、支払い失敗や価格低下アラートなどの時間依存の通知が含まれます。

## 戦術的な使用例

次のシナリオでは、イベントトリガーメッセージを様々なビジネスコンテキストにわたって適用する方法を示します。

- **買い物かご放棄メールまたは SMS** – 顧客が買い物かごに項目を追加したが、定義された期間内に購入を完了しなかった場合にリマインダーメッセージを送信します
- **閲覧放棄のフォローアップ** – 製品またはコンテンツを表示したが、コンバージョンアクションを実行しなかった訪問者を再エンゲージします
- **購入後の感謝またはクロスセル** – 購入イベントの直後に確認とクロスセルのレコメンデーションを配信します
- **体験版の有効期限リマインダー** – 更新またはコンバージョンメッセージを含む無料体験版の終了が近づいているユーザーに通知します
- **登録後のウェルカムメッセージ** – 新規ユーザーがアカウントを登録または作成したときに直ちにオンボーディングメッセージを送信します
- **フォーム送信確認** - コンテキストに基づく確認を使用して、フォーム送信（問い合わせリクエスト、申請、登録）を確認します。
- **支払い失敗通知** – 定期的な支払いが失敗したときに顧客にアラートを表示し、支払い情報を更新するように促します
- **アプリのアンインストールの取り消しプッシュ通知** - モバイルアプリケーションのアンインストール時に取り消しメッセージをトリガーします
- **予約または予定の確認** – 予約、予約、予定がスケジュールされた後、すぐに確認を送信します
- **ウィッシュリスト登録済みアイテムの価格低下アラート** — ウィッシュリストの製品が価格が低下した場合に顧客に通知します

## 主要業績評価指標

次の KPI は、イベントトリガーメッセージ実装の有効性を測定するのに役立ちます。

| KPI | 説明 | 測定 |
| --- | --- | --- |
| コンバージョン率 | 目的のアクション（購入、サインアップ、更新）を完了したトリガーメッセージ受信者の割合 | コンバージョン/配信メッセージ * 100 |
| 増分収益 | 送信されないコントロールグループと比較した、イベントトリガーメッセージに起因する追加売上高 | トリガー送信からの売上高 – コントロール母集団ベースライン |
| 開封率 | 受信者が開いた配信メッセージの割合 | 開封数/配信数* 100 |
| クリックスルー率（CTR） | 少なくとも 1 回のクリックを生成する配信メッセージの割合 | クリック数/配信数* 100 |
| コンバージョンまでの時間 | メッセージ配信とコンバージョンイベントの間の平均経過時間 | Avg （コンバージョンタイムスタンプ – 配信タイムスタンプ） |
| ジャーニーの完了率 | ジャーニーにエントリしメッセージ配信ステップに到達する（条件または離脱によってドロップされない）プロファイルの割合 | 配信に到達したプロファイル/ジャーニーにエントリしたプロファイル * 100 |
| メッセージ抑制率 | フリークエンシーキャップ、同意または条件の評価が原因で抑制された選定中のプロファイルの割合 | 抑制されたプロファイル/絞り込み済みプロファイルの合計* 100 |
| バウンス率 | ハードバウンスまたはソフトバウンスが原因で配信できなかったメッセージの割合 | バウンス /送信* 100 |

## アプリケーション

このユースケースパターンでは、次のAdobe アプリケーションが使用されています。

- **[!DNL Adobe Journey Optimizer]（AJO）** – 単一ジャーニーエントリ、条件評価、待機ステップ、メッセージオーサリング、チャネル設定、頻度ガバナンスおよび配信レポートを使用したイベントオーケストレーション

## アプリケーション関数

このプランは、アプリケーション機能カタログから次の機能を実行します。 関数は、番号付きステップではなく、実装フェーズにマッピングされます。

| 関数 | 実装フェーズ | 説明 |
| --- | --- | --- |
| Journey Orchestration | ジャーニーの作成と設定 | 単一イベントエントリを持つジャーニーの作成、選定イベントの設定、条件ノード、待機ステップ、メッセージアクション、終了条件および再エントリルールの追加 |
| チャネル設定 | チャネルサーフェスの設定 | サブドメインのデリゲーション、IP プール、送信者設定、抑制リスト管理などのチャネルサーフェス（メール、SMS、プッシュ）を設定または検証します |
| メッセージオーサリング | メッセージコンテンツの作成 | イベント駆動型パーソナライゼーション、条件付きコンテンツブロック、再利用可能なフラグメントを使用して、コンテキストメッセージコンテンツを作成します |

| 関数 | 実装フェーズ | 説明 |

## 前提条件

実装を開始する前に、次の手順を完了してください。

- [ ジャーニー ] 作成および公開権限をプロビジョニングされたAJO サンドボックス （F1）
- [ リアルタイム顧客プロファイル（F2）に対して有効な、コンテキストフィールドを使用したトリガーイベントをキャプチャする ] XDM ExperienceEvent スキーマ
- [ ] 正しいAEP データセットへのデータストリームルーティングイベントを使用して Web SDK、Mobile SDKまたはEdge Network Server API を介して設定されたリアルタイムイベントストリーミング（F3）
- [ トリガー ] ベントの識別子用に設定された ID 名前空間。ID リンクルールが設定されています（F4）
- [ チ ] ネルサーフェス （メール、SMS またはプッシュ）が設定および検証され、テスト送信が成功しました
- [ サンプルプロファイルを使用して作成、レビュー、テストされた ] メッセージコンテンツ
- [ ターゲットチャネルのプロファイル（`consents.marketing.email.val` など）に入力された ] の同意フィールド
- [ ] 条件ベースのオーディエンスフィルタリング（オプション B/C）を使用する場合、ストリーミング評価されたオーディエンスセグメントが定義され、アクティブに評価されます

## 実装オプション

ここでは、イベントトリガーメッセージの 3 つの実装オプションについて説明します。 各オプションは前のオプションに基づいて構築され、追加の機能が追加されます。

### オプション A：単純なイベントトリガーメッセージ

**最適な対象：** 遅延や条件付きロジックが必要なく、即時に単一メッセージで応答します（注文確認、ウェルカムメール、フォーム送信の確認、予約確認）。

**仕組み：**

ジャーニーは、単一イベントエントリノードを介して、選定されたイベントをリッスンします。 イベントを受信すると、プロファイルはジャーニーにエントリし、最小限の条件評価（同意確認、プロファイルの存在の検証）を通過し、設定されたチャネルアクションノードを介して単一のメッセージを直ちに受信します。

これは最も単純な実装バリアントです。 ジャーニーキャンバスには、イベントエントリノード、基本的な実施要件チェック用のオプションの条件ノード、単一のメッセージアクションノードおよび終了ノードが含まれています。 待機ステップ、複雑なブランチ、コンバージョンチェックは行われません。 メッセージは通常、トリガーイベントから数秒から数分以内に配信されます。

トリガーイベントからのイベントデータ（製品名、注文合計、フォーム詳細）は、パーソナライゼーションエディターのコンテキスト属性を使用して、メッセージのパーソナライゼーションに使用できます。 このアプローチにより、配信までの時間を最大化し、ジャーニーの複雑さを最小限に抑えることができます。

**主な考慮事項：**

- イベントの後、顧客が自己変換するかどうかに関係なく、メッセージが送信されます（コンバージョンチェックなし）
- 本質的に即座の対応が必要なイベント（確認、確認）に最適
- 再エントリのルールは慎重に設定する必要があります。確認スタイルのメッセージの場合は、各イベントでメッセージが生成されるように再エントリを許可し、ウェルカムスタイルのメッセージの場合は、再エントリを防ぎます

**メリット：**

- 最速の配信時間（イベント後の数秒から数分）
- 可動部分を最小限に抑えたシンプルなジャーニー設定
- ジャーニーの失敗やプロファイルの停止のリスクが最も低い
- テストとメンテナンスが容易

**制限事項：**

- 送信前に顧客が自己変換したかどうかを確認する機能がありません
- 時間遅れの条件付きロジックを組み込むことはできません
- 頻度ガバナンスを使用せずにイベントが頻繁に発生する場合、不要なメッセージが生成される可能性がある

### オプション B：待機を含む条件付きイベントトリガーメッセージ

**最適な対象：** 顧客がメッセージを受信する前に、自己変換する時間が必要な、遅延した条件付き応答。買い物かごの放棄（1～4 時間待ってから、買い物かごがまだ放棄されているかどうかを確認します）、閲覧の放棄（24 時間待ってから、購入があったかどうかを確認します）、体験版の有効期限（有効期限が近づくまで待ちます）。

**仕組み：**

ジャーニーは、選定されたイベントをリッスンしてプロファイルにエントリし、条件を評価する前に待機期間を導入します。 待機の後、条件ノードは、顧客が自己変換したか（例えば、購入を完了したか、サブスクリプションを更新したか）、または元のコンテキストが引き続き有効かどうかを確認します。 条件が満たされた（例：買い物かごがまだ放棄された）場合にのみ、ジャーニーはメッセージ配信に進みます。

ジャーニーキャンバスには、イベントエントリノード、待機ノード（設定可能な期間または特定の日時）、コンバージョンイベントまたはプロファイル状態の変更を確認する条件ノード、分岐パス（送信メッセージと送信なしの終了）、修飾パス上のメッセージアクションノード、各分岐の終了ノードが含まれています。 終了条件を設定すると、待機期間中にコンバージョンイベントが発生した場合にジャーニーからプロファイルを自動的に削除できるので、明示的な条件チェックが不要になります。

このアプローチは、顧客が自己変換する時間を確保し、顧客体験を向上させ、送信されるメッセージの関連性を高めることで、不要なメッセージを減らします。

**主な考慮事項：**

- 待機時間はコンバージョン率に大きく影響します。待機時間が短いと（1～2 時間）、緊急度は高くなりますが不要な送信が多くなります。待機時間が長いと（24～48 時間）、ボリュームは少なくなりますが、関連度が遅延する場合があります
- 条件評価では、コンバージョンイベントをAEPでキャプチャし、同じプロファイル ID に関連付ける必要があります
- 終了条件を使用すると、明示的な条件ノードの代わりに、変換チェックをより明確に行うことができます
- 再エントリルールでは、待機期間を考慮する必要があります。プロファイルは、既に待機している間にジャーニーに再エントリしないでください

**メリット：**

- セルフコンバージョンを確認することで、不要なメッセージを減らします
- より質の高い、より関連性の高いコミュニケーションを実現
- 設定可能な遅延ウィンドウで時間依存のユースケースをサポート
- 終了条件により、コンバージョン時のジャーニーの自動削除が有効になる

**制限事項：**

- 待機ノードと条件ノードによるジャーニーの複雑さの増加
- プロファイルが待機期間中にジャーニースロットを占有します（ジャーニータイムアウトは 91 日の場合に適用されます）
- 正確な条件評価のために、コンバージョンイベントを待機ウィンドウ内でAEPに取り込む必要がある場合
- オプション A と比較して、配信までの時間が長い

### オプション C：頻度ガバナンスでイベントトリガー

**最適な対象：** ページ表示、製品表示、検索クエリ、買い物かごへの繰り返しの追加など、メッセージ疲労が懸念される高頻度のイベントソース。 オプション A またはオプション B の上にオーバーレイとして適用できます。

**仕組み：**

このオプションは、オプション A またはオプション B に頻度ガバナンスを追加して、過度のメッセージングを防ぎます。 高頻度の行動イベント（製品表示や買い物かごへの繰り返しの追加など）は、短期間にジャーニーを複数回トリガーにすることができます。 頻度ガバナンスがないと、プロファイルが重複したり、過剰なメッセージを受け取ったりする場合があります。

頻度ガバナンスは、2 つの補完的なメカニズムによって実装されます。1 つのチャネルにつき、プロファイルが受け取るメッセージの数を期間ごとに制限するAJO ビジネスルール（頻度キャップ）と、プロファイルが同じジャーニーに再エントリできるようにする前にクールダウン期間を定義するジャーニーレベルの再エントリルールです。 ビジネスルールは組織レベルで機能し、すべてのキャンペーンとジャーニー（週に最大 3 通のメールなど）で上限が適用されますが、再エントリのクールダウンは個々のジャーニーレベルで機能します（例えば、プロファイルは最後のエントリから 72 時間以内にこの特定のジャーニーに再エントリできません）。

さらに、競合と優先度の管理を設定して、トリガーされたジャーニーに優先度スコアを割り当てることができ、同じプロファイルで複数のジャーニーやキャンペーンが競合する場合は、最も優先度の高い通信が優先されます。

**主な考慮事項：**

- フリークエンシーキャップは、疲労の防止と重要なトリガーされたメッセージが確実に配信されるようにとの間でバランスを取る必要があります
- チャネル固有のキャップ（電子メール、SMS およびプッシュ）は、異なる疲労しきい値を持つことが推奨されています
- ジャーニーの再エントリのクールダウンと組織レベルのフリークエンシーキャップは連携して機能するので、両方を設定する必要があります
- 優先度スコアによって、上限に達したときの通信の勝者が決まります。優先度の高いジャーニーには、より高いスコアを割り当てる必要があります

**メリット：**

- 高頻度のイベントソースによるメッセージ疲労を防ぐ
- ブランドの評判と購読者の満足度を維持
- 組織レベルの上限は、すべてのキャンペーンとジャーニーにわたってセーフティネットを提供します
- 優先度のスコアリングにより、最も重要なメッセージが制限に達した際に確実に配信されます

**制限事項：**

- 一部の選定中のプロファイルは抑制され、関連するメッセージが欠落する可能性があります
- フリークエンシーキャップの設定により、管理がより複雑
- キャップは、同じチャネルを共有する他のアクティブなキャンペーンやジャーニーと予期せず相互作用する場合があります
- メッセージングポートフォリオの変更に合わせて、継続的な監視と調整が必要

### 適切なオプションを選択

次の決定ガイダンスを使用して、適切な実装オプションを選択してください。

1. **メッセージを遅延なく直ちに送信する必要はありますか？** ある場合は、**オプション A** から開始します。確認メッセージ、ウェルカムメールおよび確認には、通常、待機期間は必要ありません。

2. **お客様は、メッセージを受信する前に自己変換する時間を確保する必要がありますか？** 「はい」の場合は、「**オプション B**」を選択します。買い物かごの放棄、閲覧の放棄および体験版の有効期限のユースケースには、顧客が自分でアクションを完了できる待機期間のメリットがあります。

3. **トリガーイベントは高頻度ですか（同じプロファイルに対してセッションまたは 1 日に複数回発生します）?** 使用できる場合は、オプション A または B の上にオーバーレイとして **オプション C** を追加します。製品ビュー、ページビュー、検索イベントでは、周波数対策が必要なトリガーが大量に生成される可能性があります。

4. **サンドボックスで複数のトリガージャーニーとキャンペーンがアクティブですか？** ある場合は、イベントの頻度に関係なく **オプション C** を追加して、クロスジャーニー通信の負荷を管理し、チャネル疲労を防ぎます。

実際には、ほとんどの実稼動実装では、放棄スタイルのユースケース（頻度ガバナンスによる条件付き待機）にはオプション B + C を使用し、確認スタイルのユースケース（頻度ガバナンスをセーフティネットとして使用する即時送信）にはオプション A + C を使用します。

>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| メールアドレス | 認証済み web セッションまたはメールが取り込まれたフォームからのイベント | 連絡可能な ID に直接解決。メール配信への最も簡単なパス |
>| CRM ID | CRM ID がプライマリ識別子である、認証済みシステムからのイベント | CRM ID 名前空間の作成が必要です。プロファイルには、メッセージ配信用のリンクされたメールまたは電話が必要です |
>| ECID （Experience Cloud ID） | 認証済み ID を使用できない匿名の web イベントまたはアプリイベント | ECID を既知の ID にリンクするための ID ステッチが必要です。メッセージは ECID のみに送信できません |
>| 電話番号 | モバイルまたはコールセンターインタラクションからのイベント | SMS 配信をサポート。電話名前空間が必要です |

**UI ナビゲーション：** ID/ID 名前空間を作成；スキーマ/スキーマを選択/フィールドを選択/ID/プライマリ ID として設定；プロファイル/結合ポリシー

**キー設定の詳細：**

- 匿名イベント（ECID のみ）は、ECID が ID グラフを介して既知の接触可能な ID （メール、電話）にリンクされるまで、メッセージ配信をトリガーできません
- AJOで使用される結合ポリシーは、オーディエンスの評価に使用される結合ポリシーと一致している必要があります
- Web ベースのトリガーメッセージの場合は、ID グラフが、ログインイベントを介して ECID を認証済み ID にリンクしていることを確認します

### フェーズ 5：ジャーニーの作成と設定

**応用機能：** AJO:Journey Orchestration、AJO:Frequency &amp; Business Rules （Option C）、AJO:Conflict &amp; Priority Management

**設定する内容：** トリガイベントをリッスンしてメッセージ配信を調整するジャーニー。 これは、イベント入力ノード、条件ノード、待機ステップ（オプション B の場合）、メッセージアクションノードおよび終了条件を使用してジャーニーキャンバスが設計されるコア実装フェーズです。 このフェーズでは、周波数ガバナンス（オプション C）とコンフリクト/プライオリティの設定も扱います。

**このフェーズの決定ポイント：**

>[!NOTE]
>
>**決定：再入国ルール**
>
>トリガーイベントが再び発生した場合、プロファイルはこのジャーニーに再度エントリできますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 再エントリを許可（クールダウン付き） | このイベントは合法的に繰り返し発生する可能性があり、発生するたびにメッセージが保証されます（例：買い物かごの放棄ごとに、復旧メッセージがトリガーされる場合） | クールダウン期間（5 分以上）を設定して、重複イベントからの迅速な再入場を防ぎます。通常のクールダウンは 1 時間から 72 時間です |
>| 再エントリなし | イベントの繰り返し（例：初回登録後のウェルカムメッセージ）に関係なく、プロファイルごとにメッセージを 1 回だけ送信する必要があります | プロファイルは、最初のジャーニーの完了後、永続的に除外されます。これは、1 回限りのライフサイクルメッセージに適しています |

>[!NOTE]
>
>**決定：待機期間（オプション B のみ）**
>
>条件を評価してメッセージを送信するまで、ジャーニーはどのくらい待てばよいですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 短時間（1～4 時間） | 即時のフォローアップが効果的な、買い物かごの放棄などの緊急性の高いユースケース | メッセージ量が多い（一部のセルフコンバータが検出される可能性がある）。価値の高い買い物かごの回復に最適 |
>| Medium待機（12 ～ 24 時間） | 閲覧放棄やコンテンツエンゲージメントなど、ある程度の緊急度を持つユースケース | バランスの取れたアプローチ：関連性を維持しながら不要な送信を削減 |
>| 長い待機（2 ～ 7 日） | 試用版の有効期限のリマインダーや定期的なチェックインなど、緊急度の低いユースケース | メッセージ量の減少：コンテキストに基づく関連性を失うリスクが高い |
>| カスタムの日時 | 特定の日付に関連付けられたユースケース（例：試験の有効期限の 3 日前に送信） | 計算された日時まで待機します。カスタム式エディターを使用します |

>[!NOTE]
>
>**決定：終了条件**
>
>メッセージアクションに到達する前に、プロファイルをジャーニーから削除する必要がある条件は何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| コンバージョンイベント（購入完了など） | 目標がコンバージョンの場合の買い物かごまたは参照放棄 | コンバージョンイベントが検出されると、プロファイルは自動的に終了します。オプション B で最もクリーンなアプローチ |
>| オーディエンスメンバーシップの変更 | 選定セグメントを離れる場合、プロファイルは終了する必要があります | ほぼリアルタイムで更新されるストリーミング評価済みオーディエンスが必要です |
>| ジャーニータイムアウト（最大 91 日） | デフォルトの動作 – 最大ジャーニー時間が経過するとプロファイルが終了します | セーフティネット。短時間のみ有効なトリガージャーニーの場合は、主な出口メカニズムにしない |
>| 明示的な終了条件がありません | すべての選定中のプロファイルがメッセージを受け取る簡単な確認（オプション A） | プロファイルは、メッセージアクションと終了ノードの後に自然に終了します |

>[!NOTE]
>
>**決定：頻度ガバナンス（オプション C）**
>
>プロファイルが期間ごとに受信するトリガーメッセージの数
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| チャネル固有の上限（例：週に 3 件のメール、1 日に 1 件の SMS） | チャネルが異なると、疲労しきい値も異なります | 推奨アプローチ。各チャネルの侵入度レベルを尊重します |
>| グローバルな上限（例：すべてのチャネルで週 5 件のメッセージ） | すべてのコミュニケーションタイプにわたるガバナンスの簡素化 | 管理が容易であるが操作に制限が少ない（押し付けがましくない経路を過剰に制限する可能性がある） |
>| ジャーニーレベルの再エントリのクールダウンのみ | この特定のジャーニーには頻度ガバナンスが必要ですが、組織全体では必要ありません | 最もシンプルな実装：ジャーニー間の疲労を防げない |

**UI ナビゲーション：**&#x200B;ジャーニー/ジャーニーを作成（ジャーニー作成用）、ジャーニーキャンバス/イベントをドラッグ（イベント入力用）、ジャーニーキャンバス/待機アクティビティをドラッグ（待機設定用）、ジャーニーキャンバス/条件をドラッグ（条件分岐用）、ジャーニープロパティ/終了条件（終了設定用）、管理/ビジネスルール/ルールを作成（頻度制限用）

**キー設定の詳細：**

- イベントスキーマを選択し、選定条件を定義して、ジャーニーイベントを設定します
- オプション B の場合、イベントエントリの直後、条件評価の前に待機ノードを追加します
- プロファイル属性、イベントデータまたはオーディエンスメンバーシップチェックを使用した条件ノードの設定
- メッセージアクションノードをチャネルサーフェスにリンクし、フェーズ 3 と 4 で作成したコンテンツをリンクします
- ジャーニーのタイムアウトを適切な期間に設定します（トリガージャーニーの場合は 91 日未満）。
- オプション C の場合は、ジャーニーを公開する前に、管理/ ビジネスルールを使用して頻度ルールを作成します
- 他のキャンペーンやジャーニーが重複するオーディエンスをターゲットにしている場合は、ジャーニーに優先度スコアを割り当てます

**オプションが分岐する場所：**

**オプション A （シンプルなイベントトリガー）の場合：**
ジャーニーキャンバスは最小限です。イベントエントリ/オプションの同意/実施要件/メッセージアクション/終了です。 待機ステップやコンバージョンチェックは不要です。 イベントが発生するたびにメッセージを生成する必要があるかどうかに基づいて、再入場ルールを設定します。

**オプション B （待機あり条件付き）:**
イベントエントリの後に待機ノードを追加します。 待機の後、条件ノードを追加して、コンバージョンを確認するか（ジャーニーエントリ後にイベントが発生したかどうか `commerce.purchases` 確認するなど）、終了条件を使用して、コンバージョンされたプロファイルを自動的に削除します。 「変換されていない」パスのメッセージアクションと「変換された」パスの終了ノードに分岐します。

**オプション C （頻度ガバナンス）の場合：**
公開する前に、管理/ビジネスルールを使用して、組織レベルのフリークエンシーキャップを設定します。 ジャーニープロパティでジャーニーレベルの再エントリのクールダウンを設定します。 オプションで、ジャーニープロパティで優先度スコアを割り当て、キャップに達したときの通信の勝者を制御します。 これは、オプション A またはオプション B の上に適用できます。

## 実装に関する考慮事項

このセクションでは、イベントトリガーメッセージ実装のガードレール、一般的な落とし穴、ベストプラクティスおよびトレードオフの決定について説明します。

### ガードレールと制限

イベントトリガーメッセージの実装には、次のプラットフォームガードレールと制限が適用されます。

- **単一イベントのスループット：** 単一イベントジャーニーのサンドボックスごとに最大 5,000 イベント/秒 – [Journey Optimizer ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/journey-optimizer/using/get-started/guardrails)
- **ライブジャーニーの制限：** サンドボックスあたり最大 500 個のライブジャーニー – [Journey Optimizerガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/journey-optimizer/using/get-started/guardrails)
- **ジャーニーキャンバスの制限：** ジャーニーキャンバスあたり最大 50 個のアクティビティ
- **ジャーニータイムアウト：** ジャーニーの最大期間は 91 日です（グローバルタイムアウト）
- **再入場クールダウン：** 再入場クールダウンの最短は 5 分
- **フリークエンシーキャップ設定：** サンドボックスあたり最大 10 個のキャッピング設定
- **チャネルサーフェス：** サンドボックスあたりのチャネルタイプあたり最大 10 個のチャネルサーフェス
- **ストリーミング取り込み：** HTTP 接続あたり最大 20,000 レコード/秒 – [&#x200B; 取り込みガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/ingestion/guardrails)
- **計算属性：** サンドボックスあたり最大 25 計算属性 – [&#x200B; 計算属性ガードレール &#x200B;](https://experienceleague.adobe.com/ja/docs/experience-platform/profile/computed-attributes/overview#guardrails)
- **コンテンツフラグメント：** メッセージあたり最大 30 個のコンテンツフラグメント
- **ライブレポートの更新：** ライブレポートは 60 秒ごとに更新され、過去 24 時間のデータが表示されます
- **履歴レポートの待ち時間：** 履歴（常に）レポートは、実行が終了した後に完全に入力されるまでに最大 2 時間かかる場合があります

### よくある落とし穴

イベントトリガーメッセージを実装する際には、次の一般的な問題に注意してください。

- **ID 解決を伴わない匿名イベント：** ECID （匿名の Cookie ID）のみを持つイベントをトリガーしても、メッセージ配信の接続可能なプロファイルに解決することはできません。 トリガーイベントに認証済み ID が含まれているか、ID グラフが ECID を既知の連絡先にリンクしていることを確認します。 ID 解決がない場合、プロファイルはジャーニーにエントリしますが、メッセージ配信ステップで失敗します。

- **オプション B の条件チェックにコンバージョンイベントがありません：** コンバージョンイベント（購入など）がAEPに取り込まれていない場合、またはトリガーイベントと同じプロファイル ID に関連付けられていない場合、条件チェックは誤って「コンバージョンなし」と評価され、不要なメッセージが送信されます。 コンバージョンイベントがリアルタイムでAEPに送られ、トリガーイベントと ID を共有することを確認します。

- **過度に許容度の高い再エントリルール：** 高頻度のイベントソースでクールダウン期間のない再エントリを許可すると、同じプロファイルが数分以内に複数のメッセージを受信する可能性があります。 ビジネスコンテキストに合った再エントリのクールダウンを常に設定します（例：買い物かごの放棄の 4 時間のクールダウン、閲覧の放棄の 24 時間のクールダウン）。

- **待機ステップのタイムゾーンの不一致：** 待機ステップはデフォルトで UTC で処理されます。 ジャーニーが複数のタイムゾーンをまたいでプロファイルをターゲットにし、待機が受信者のローカルタイムに合う必要がある場合は、それに応じてジャーニーのタイムゾーン設定を構成します。

- **他のキャンペーンと競合するフリークエンシーキャップ：** 組織レベルのフリークエンシーキャップは、すべてのキャンペーンとジャーニーに適用されます。 プロファイルが既に他のアクティブなキャンペーンの上限に達している場合、新しくトリガーされたジャーニーは抑制される場合があります。 抑制率を監視し、優先度スコアを調整して、重要なトリガーメッセージに優先順位を付けます。

- **依存関係がないため、ジャーニーの公開に失敗しました：** ジャーニーキャンバスのすべての分岐は、終了アクティビティで終了する必要があります。 すべてのアクションノードは、アクティブなメッセージコンテンツを持つ有効なチャネルサーフェスを参照する必要があります。 公開する前に、すべての依存関係を確認します。

### ベストプラクティス

イベントトリガーメッセージを正常に実装するには、以下の推奨事項に従ってください。

- **シンプルなメッセージから繰り返し開始：** 新しいトリガーメッセージのユースケースの場合は、オプション A から開始します。 イベントパイプラインとメッセージ配信を検証した後にのみ、待機と条件のロジック（オプション B）を追加します。 複数のトリガージャーニーがアクティブな場合は、頻度ガバナンス（オプション C）を追加します。

- **コンバージョンチェックに条件ノードの代わりに終了条件を使用：** 終了条件は、対象となるイベント（購入など）が発生するとジャーニーからプロファイルを自動的に削除するので、待機ステップ後に明示的な条件ノードを使用する必要がなくなります。 これにより、よりクリーンなジャーニーキャンバスとよりレスポンシブなコンバージョン検出が可能になります。

- **開発用サンドボックス内の実際のイベントデータを使用したテスト：** 実際のイベントペイロード（テストプロファイルだけでなく）を使用して、イベントスキーマフィールド、パーソナライゼーショントークン、条件式が、実稼動環境のようなデータで正しく機能することを検証します。

- **イベント固有の再エントリのクールダウンを設定：** クールダウン期間を、トリガーイベントのビジネスコンテキストに一致させます。 買い物かごの放棄ジャーニーは通常、4～24 時間のクールダウンを使用します。確認ジャーニーでは即時の再エントリが許可される場合があります。オンボーディングジャーニーでは、再エントリが完全に防ぐ必要があります。

- **抑制率をヘルス指標として監視：** 抑制率（プロファイルは適合するが、頻度の制限や条件によってメッセージを受信しない）が 30～40% を超える場合は、制限が厳しすぎるか、トリガーイベントが定義されている範囲が広すぎるかを確認します。

- **ライブジャーニーを編集する代わりにバージョンジャーニーを使用する：** ライブジャーニーは直接編集できません。 ジャーニーを複製し、変更を加えてから、古いバージョンを停止して新しいバージョンを公開することで、新しいバージョンを作成します。 これにより、現在のジャーニー内のプロファイルが中断するのを回避できます。

- **条件ノードにフォールバック/デフォルトパスを含める：** 予期しないプロファイル状態を処理するために、常に条件ノードにデフォルト（else）パスを設定します。 条件の分岐に一致しないプロファイルは、停止したままにするのではなく、適切に終了する必要があります。

### トレードオフの決定

イベントトリガーメッセージの実装を設計する際は、次のトレードオフを考慮します。

>[!NOTE]
>
>**トレードオフ：メッセージ速度とメッセージの関連性**
>
>即時メッセージ配信（オプション A）は速度を最大化しますが、自己変換を行おうとしている顧客にメッセージを送信できます。 遅延条件付き配信（オプション B）は、セルフコンバータを除外することで関連性を向上させますが、待ち時間が発生して緊急度が低下する可能性があります。
>
>- **オプション A のお勧め：** すべてのイベントでメッセージが保証される、スピード、シンプルさ、確認スタイルのユースケース
>- **オプション B がお気に入り：** 関連性、メッセージ疲労の軽減、セルフコンバージョンが一般的な放棄スタイルのユースケース
>- **推奨事項：** 速度が主要な値であるトランザクションメッセージと確認メッセージには、オプション A を使用してください。 関連性が速度を上回る場合は、オプション B を使用して回復メッセージと再エンゲージメントメッセージを実行します。 待機時間を試して、各ユースケースに最適なバランスを見つけます。

>[!NOTE]
>
>**トレードオフ：周波数保護とメッセージカバレッジの比較**
>
>厳密なフリークエンシーキャップ（オプション C）は疲労を防ぎますが、プロファイルがキャップに近い場合、重要なトリガーメッセージが抑制される可能性があります。 制限が許容範囲内または無制限の場合はカバレッジが最大化されますが、メッセージの過剰送信やチャネル疲れが発生するリスクがあります。
>
>- **厳格な制限の適用：顧客体験、ブランド評判、配信品質の** 化（スパムの苦情を減らす）
>- **許容上限の適用：** メッセージの対象範囲、コンバージョンの機会、トリガーの見逃しを回避
>- **推奨事項：** 業界標準の上限（1 週間に 3～5 通のメール、1 週間に 1～2 通の SMS、1 日に 2～3 通のプッシュ）から開始し、エンゲージメント指標と購読解除率に基づいて調整します。 上限に達するとバッチキャンペーンに勝つように、トリガーメッセージに優先度の高いスコアを割り当てます。

>[!NOTE]
>
>**トレードオフ：ジャーニーのシンプルさとジャーニーの洗練**
>
>シンプルなジャーニー（ノードが少なく、ブランチがない）は、作成、テスト、保守が容易ですが、コンテキストに合わせた適応機能は限られています。 高度なジャーニー（複数の条件、分岐パス、セグメントチェック）により、より詳細な応答が可能になりますが、複雑さが増し、エラーのリスクが高まります。
>
>- **シンプルなジャーニーのメリット：** 実装の迅速化、デバッグの容易化、メンテナンス負荷の軽減
>- **洗練されたジャーニーのメリット：** より正確なターゲティング、パーソナライゼーションの向上、不要な送信の削減
>- **推奨事項：** ビジネス要件を満たす最も単純なジャーニーから始めます。 パフォーマンスデータに基づいて段階的に複雑さを追加していきます。 確実に機能する単純なジャーニーは、エラーが検出されない複雑なジャーニーよりも価値があります。
