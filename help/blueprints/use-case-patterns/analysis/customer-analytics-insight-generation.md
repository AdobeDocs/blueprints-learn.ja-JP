# Customer analytics &amp; insightの生成

このガイドでは、Customer Analytics とinsightの生成に関する完全な実装リファレンスを提供します。 [!DNL Adobe Experience Platform] データセットを [!DNL Customer Journey Analytics] に接続する方法、データビューを設定する方法、フリーフォーム分析ワークスペースを作成する方法、計算指標を作成する方法、ダッシュボードとモバイルスコアカードを公開する方法およびオプションでCJAで定義したオーディエンスを [!DNL Adobe Experience Platform] に公開してアクティブ化する方法について説明します。

これは、すべての実行可能な実装パス、それらの間のトレードオフ、および各段階で必要な設定決定を理解する必要があるソリューションアーキテクト、マーケティングテクノロジスト、実装エンジニア向けに設計されています。

分類の他のパターンは、アクティブ化とエンゲージメント（メッセージの送信、コンテンツのパーソナライズ、オーディエンスのアクティブ化）に重点を置いていますが、このパターンは、顧客の行動の分析、キャンペーンのパフォーマンスの測定、トレンドの識別、戦略と最適化の決定に役立つインサイトの生成などを理解することに重点を置いています。 これは、最も一般的に構成されるパターンであり、ほぼすべてのアクティベーションまたはパーソナライゼーションパターンとペアになっています。

## ユースケースの概要

組織は、チャネルをまたいだ顧客の行動、キャンペーンのパフォーマンス、顧客がジャーニーに立ち寄る場所、共鳴するコンテンツ、様々なセグメントが時間の経過と共にどのように維持されるかを理解する必要があります。 Customer analytics とinsightのジェネレーションは、[!DNL Adobe Experience Platform] の豊富なクロスチャネルデータを [!DNL Customer Journey Analytics] に接続することで、このニーズに応えます。アナリストは、フリーフォームワークスペースの作成、カスタム指標の作成、アトリビューションモデルの設定、関係者が利用するためのダッシュボードの公開を行えます。

パターンは、詳細な探索的分析を必要とするマーケティングアナリスト、パフォーマンスダッシュボードを必要とするキャンペーンマネージャー、エンゲージメントとリテンションのインサイトを必要とする製品マネージャー、一目で KPI スコアカードを確認できるエグゼクティブなど、複数のオーディエンスにサービスを提供します。 実装のアプローチは、主要な分析的フォーカス（キャンペーンパフォーマンス測定、クロスチャネルジャーニー分析、分析主導のオーディエンスのアクティベーション、ガイド付きの製品インサイト）によって異なります。

## 主なビジネス目標

このユースケースパターンでは、以下のビジネス目標がサポートされています。

**分析とレポートの改善**

統合ダッシュボードとセルフサービスツールを通じて、レポート機能を強化し、より迅速で実用的なマーケティングインサイトを得ます。

- **KPI:** 効率、生産性

このビジネス目標について詳しくは、[ 分析とレポートの改善 ](/help/blueprints/business-objectives/analytics-insights/improve-analytics-reporting.md) を参照してください。

**データに基づく意思決定の有効化**

セルフサービス分析、リアルタイムの顧客インサイト、戦略の指針となる AI を活用した予測で、チームを強化します。

- **KPI:** 効率、生産性

このビジネス目標について詳しくは、[ データ駆動型意思決定の有効化 ](/help/blueprints/business-objectives/analytics-insights/enable-data-driven-decision-making.md) を参照してください。

**マーケティングアトリビューションの改善**

マーケティングタッチポイント、チャネルおよびキャンペーンがコンバージョンや収益の成果に与える影響を正確に測定します。

- **KPI:** 効率、増分収益

このビジネス目標について詳しくは、[ マーケティングアトリビューションの改善 ](/help/blueprints/business-objectives/analytics-insights/improve-marketing-attribution.md) を参照してください。

**マーケティング費用と ROI の最適化**

最も高いリターンを実現するチャネルとキャンペーンを把握して、マーケティング予算の配分を最適化します。

- **KPI:** 効率、増分収益

このビジネス目標について詳しくは、[ マーケティング費用と ROI の最適化 ](/help/blueprints/business-objectives/cost-efficiency/optimize-marketing-spend-roi.md) を参照してください。

## 戦術的な使用例

このパターンで実装できる戦術的なユースケースの例を以下に示します。

- キャンペーンパフォーマンスダッシュボード – メール、SMS、プッシュ、有料メディアキャンペーン全体での配信指標、エンゲージメント率、コンバージョンおよび収益アトリビューション
- カスタマージャーニーのフォールアウト分析 – 購入、登録、オンボーディングファネルで、顧客がどこで離脱するかを特定します
- コホート保持分析 – 様々な獲得コホートが数週間、数か月、四半期でどの程度保持しているかを測定します
- チャネルアトリビューションモデリング – ファーストタッチ、ラストタッチ、線形、タイムディケイアトリビューションを比較して、コンバージョンを促進するチャネルを理解します
- コンテンツのパフォーマンス分析：セグメント、チャネル、ライフサイクルの各ステージで、最も共感を得られるコンテンツを特定します
- 製品の使用状況と導入分析 – 機能の採用、エンゲージメントの頻度、ユーザーの増加のトレンドを追跡します
- カスタマーライフサイクルステージ分析 – ライフサイクルステージ（新規、アクティブ、危険、失効）別に顧客をセグメント化して分析します。
- マーケティングミックス最適化ダッシュボード – チャネル投資と収益貢献度の比較
- クロスチャネルエンゲージメントスコアおよびレポート - web、アプリ、メールおよびキャンペーンのインタラクションから複合エンゲージメントスコアを作成します

## 主要業績評価指標

次の KPI は、このユースケースパターンの成功を測定するのに役立ちます。

| --- | --- | --- |
| 効率 | insightまでの時間の短縮と手動レポートの手間の軽減 | CJAの実装前後にアナリストがレポートの作成に費やした時間の追跡 |
| 生産性 | ビジネスユーザーによって作成されたセルフサービス分析の数 | Workspace プロジェクトの作成とダッシュボードの使用状況の監視 |
| 増分収益 | インサイト主導の最適化の決定によって生じる収益 | CJAの分析に基づいて最適化されたキャンペーンからの売上高増加を測定します |
| コンバージョン率 | 主要なカスタマージャーニーにおけるFunnelの完了率 | CJA フォールアウトビジュアライゼーションを使用して、ジャーニーステップごとにフォールアウト率を追跡する |
| エンゲージメント | チャネルをまたいだ顧客インタラクションの深度と頻度 | CJAでのエンゲージメントスコアリングの計算指標の作成 |
| 定着 | 定義された期間における顧客の返品率 | CJA コホート分析を使用したリテンションカーブの測定 |

## ユースケースパターン

**Customer Analytics およびinsightの生成**

クロスチャネル分析のワークスペース、計算指標およびダッシュボードを作成して、顧客の行動とキャンペーンのパフォーマンスを把握します。

## アプリケーション

このユースケースパターンでは、次のアプリケーションが使用されています。

## 前提条件

このユースケースパターンを実装するには、次の前提条件を満たす必要があります。

- CJA製品の使用権限が組織用にプロビジョニングされます
- CJA製品プロファイルは適切なユーザーアクセス（ワークスペースの作成、データビューアクセス）で設定されます
- AEP サンドボックスには、データフロー（web イベント、アプリイベント、キャンペーンデータ、CRM レコード）を含んだターゲットデータセットが含まれています
- XDM スキーマは、適切なフィールドグループを持つすべてのソースデータセットに対して定義されます
- 人物 ID フィールドは識別され、接続されるすべてのデータセットで一貫して使用できます
- ID 名前空間は、AEP接続ステッチで使用されるユーザー ID に対してCJAで設定されます
- 関係者の要件については、どの KPI、どのオーディエンスがダッシュボードを使用するか、どのレベルの詳細かが文書化されています
- モバイルスコアカードの場合：関係者は [!DNL Adobe Analytics] ダッシュボードモバイルアプリをインストールしています
- オプション C （オーディエンス公開）の場合：ターゲットサンドボックスでAEP リアルタイム顧客プロファイルが有効になっている
- オプション D （ガイド付き分析）の場合：CJA SKU にはガイド付き分析機能が含まれています

## 実装オプション

この節では、このユースケースパターンで使用可能な実装オプションについて説明します。

### オプション A:Campaign パフォーマンス分析

**ベストプラクティス：** メールキャンペーンダッシュボード、ジャーニーfunnel分析、チャネルパフォーマンス比較、マーケティング ROI レポートなど、キャンペーンとジャーニーの効果の測定と最適化。

**仕組み：**

このオプションは、AJO campaign およびジャーニーデータセットをCJAに接続し、配信指標とエンゲージメント指標（送信、配信、開封数、クリック数、バウンス数、購読解除数）を使用してデータビューを設定し、キャンペーンのパフォーマンスダッシュボードを作成して、マーケティング関係者向けのスコアカードを公開します。 焦点は、マーケティングキャンペーンがチャネル間および時間の経過と共にどのように実行されるかを理解することです。

データビューは、キャンペーン固有のディメンション（キャンペーン名、ジャーニー名、チャネルタイプ、メッセージバリアント）と配信指標で設定されます。 計算指標は、開封率、クリックスルー率、コンバージョン率、メッセージあたりの売上高など、導出された指標に対して作成されます。 ダッシュボードは、トレンド分析の比較期間と共にこれらの KPI を表示します。

**主な考慮事項：**

- AEPにAJO キャンペーンおよびジャーニーイベントデータセットが必要です
- アトリビューションモデルは、組織のキャンペーン測定理念に合わせる必要があります
- AJOのネイティブレポート（運用上の配信指標の場合）とCJA（クロスチャネルビジネスへの影響の場合）の両方を含めることを検討します

**メリット：**

- キャンペーンの測定と最適化に特化
- キャンペーン間の比較およびチャネルミックス分析を有効にします
- 計算指標は、すべてのキャンペーンで標準化された KPI 定義を提供します
- モバイルスコアカードにより、マーケティングリーダーは一目でパフォーマンスを確認できます

**制限事項：**

- キャンペーンおよびジャーニーデータに限定。カスタマージャーニーの完全なコンテキストを提供しない
- ジャーニーパス、フォールアウト、コホート分析は含まれない
- アトリビューションは、完全なカスタマージャーニーではなく、キャンペーンタッチポイントを対象としています

### オプション B:Customer Journey Analytics

**最適な対象：** Web、アプリ、メール、CRM、オフラインのタッチポイントをまたいだクロスチャネルのカスタマージャーニー（フォールアウト分析、パス分析、コホート保持、アトリビューションモデリング、ライフサイクルステージ分析）について。

**仕組み：**

このオプションは、複数のAEP データセット（web イベント、アプリイベント、CRM データ、キャンペーンインタラクション、トランザクションレコード）を接続して、カスタマージャーニーの統一されたクロスチャネルビューを構築します。 データビューは、すべてのチャネルにまたがるディメンションと指標で設定されます。 CJAのフロー、フォールアウト、コホート、アトリビューションビジュアライゼーションを使用して、顧客がジャーニー内をどのように移動するか、離脱した場所、様々なセグメントの保持方法、コンバージョンにクレジットを付与する必要があるチャネルを分析します。

これは、エンドツーエンドのカスタマーエクスペリエンスに深いinsightを提供する、最も包括的な分析オプションです。 また、実装も最も複雑で、データセット間のステッチに慎重な人物 ID 設定が必要になり、適切なディメンションと指標を公開するための思慮深いデータビューデザインが必要になります。

**主な考慮事項：**

- 正確なクロスチャネル分析のために、接続されたすべてのデータセットで一貫したユーザー ID が必要です
- AEPのスキーマデザインは、CJA分析の質と深度に直接影響します
- 接続内のデータセットが増えると、分析は豊富になりますが、バックフィル時間が長くなります
- アトリビューションモデリングには、明確なコンバージョンイベント定義が必要です

**メリット：**

- クロスチャネルカスタマージャーニーの表示を完了
- CJA ビジュアライゼーションの完全なスイート：フロー、フォールアウト、コホート、アトリビューション、フリーフォームテーブル
- 単一チャネルレポートでは見えないインサイトの検出を有効にします
- 顧客の行動とライフサイクルに関する複雑な分析の質問をサポート

**制限事項：**

- マルチデータセット接続とクロスチャネルステッチにより、実装が複雑になる
- データ表示の設定および派生フィールドに関する事前計画が必要
- 大規模なマルチデータセット接続のバックフィルには、数日かかる場合があります
- 分析の品質は、基になるデータの完全性と一貫性に依存します

### オプション C：オーディエンス公開を使用した Analytics

**最適な対象：** 分析主導のアクティベーション - CJA分析を通じて興味深いセグメントを見つけ、RT-CDP 宛先、AJO キャンペーン、AJO ジャーニーを通じてアクティベーションのためにAEPに公開します。

**仕組み：**

このオプションは、オプション A またはオプション B をCJAからのオーディエンスの公開で拡張します。 アナリストは、クロスチャネルの行動データとCJA フィルターの完全な分析機能を使用してCJAでセグメントを作成し、ダウンストリームのアクティベーションのためにそれらのオーディエンスをAEP リアルタイム顧客プロファイルに公開します。 これにより、insightとアクションの間のギャップが埋まります。探索的分析で見つかったセグメントは、AEP セグメントビルダーで手動で再作成しなくても、実用的なオーディエンスになります。

公開されたオーディエンスは、接触チャネルが「CJA」のAEP オーディエンスポータルに表示され、任意の RT-CDP 宛先に対してアクティブ化したり、AJOでキャンペーンターゲットとして使用したり、ジャーニーエントリ条件として使用したりできます。

**主な考慮事項：**

- AEP リアルタイム顧客プロファイルがターゲットサンドボックスで有効になっている必要があります
- CJA接続には、AEP ID 名前空間に解決される有効なユーザー ID が必要です
- 公開済みのオーディエンスが、組織のAEP オーディエンスの使用権限にカウントされます
- 更新ケイデンスは、アクティベーション要件（1 回、4 時間ごと、毎日、毎週）に基づいて設定する必要があります。

**メリット：**

- 分析とアクティブ化の間のループを閉じます
- CJAのクロスチャネル行動データを使用して、高価値セグメントを検出できます。
- CJAで定義されたオーディエンスは、AEP セグメントビルダーでは使用できないディメンションとフィルターを活用できます
- 分析インサイトに基づくオーディエンス条件の反復的な絞り込みをサポートします。

**制限事項：**

- CJAのお客様 1 人あたり最大 75 人の公開済みオーディエンス。
- 大規模なデータセットの場合、初期オーディエンスの評価に最大 24 時間かかる場合があります
- CJAで公開されたオーディエンスは、AEPでは編集できません。変更は、CJAで行う必要があります
- 基本分析以外にも、追加の ID 名前空間およびプロファイル設定が必要です

- [オーディエンスの作成と公開](https://experienceleague.adobe.com/en/docs/analytics-platform/using/cja-components/audiences/publish)

### オプション D：製品チーム向けのガイド付き分析

**最適な対象：** 製品エクスペリエンスのインサイト – 機能の導入、ユーザーエンゲージメントのトレンド、保持率の分析、funnelのコンバージョン、リリースの影響分析を、CJAのガイド付き分析ワークフローを使用して行います。複雑なフリーフォームのWorkspace プロジェクトを設定する必要はありません。

**仕組み：**

このオプションでは、構造化されたテンプレート化されたinsight生成にCJA ガイド分析を使用します。 ガイド付き分析では、funnel、トレンド、保持率、ユーザーの増加率、エンゲージメント頻度、リリースの影響、初回使用、タイムラインなどの事前定義済みの分析タイプを提供し、アナリストが構造化されたワークフローを通じて、特定の製品やエクスペリエンスに関する質問に答えられるようにします。 フリーフォームプロジェクトをゼロから作成することなく、迅速で焦点を当てたインサイトを必要とする製品マネージャーやアナリストに最適です。

この実装では、AEP データセットをCJAに接続し、イベントレベルのディメンションと指標を使用してデータビューを設定したあと、ガイド付き分析ワークフローを使用してインサイトを生成します。 結果は、Workspace プロジェクト内にパネルとして保存し、さらにカスタマイズできます。

**主な考慮事項：**

- Guided analysis には、ガイド付き分析機能を含むCJA製品の使用権限が必要です
- キャンペーンパフォーマンス測定ではなく、製品およびエクスペリエンス分析に最適
- アナリストでないユーザーが利用しやすい構造化されたワークフローを提供します
- フリーフォームWorkspace分析と組み合わせて、より深い探索を行うことができます

**メリット：**

- 入力に対する障壁の軽減 – 構造化されたワークフローにより、ユーザーは分析を進めることができます。
- 製品エクスペリエンスに関する質問（funnel、リテンション、成長、影響）に対応する専用のソリューション
- 一般的な分析質問のinsightまでの時間を短縮
- 保存済みの分析を、フリーフォーム分析と共にWorkspace プロジェクトに埋め込むことができます

**制限事項：**

- フリーフォームWorkspace分析よりも柔軟性が低い
- 事前定義済みの分析タイプ（funnel、トレンド、保持率、増加率、頻度、影響、タイムライン）に制限
- セグメント比較では、最大 3 つのセグメントを同時にサポートします
- Funnel分析は、最大 15 のステップをサポートします

### 適切なオプションを選択

次のガイダンスを使用して、ニーズに最適な実装オプションを選択してください。

- **主な目標がキャンペーンの有効性の測定であり** AJO キャンペーンデータがAEPに送られる場合は、**オプション A** から始めます。マーケティングパフォーマンスレポートの価値創出を最速で実現します。

- **Web、アプリ、メール、オフラインのタッチポイントをまたぐ完全なカスタマージャーニーを理解する必要があり** 一貫したユーザー ID を持つ複数のデータセットがある場合は、「**オプション B**」を選択します。最も深い分析機能を提供しますが、データビュー設定にはより多くの先行投資が必要です。

- **インサイトに基づいて行動する場合** CJAで検出されたセグメントをAEPに公開して RT-CDP またはAJOでアクティブ化するには、**オプション C** を選択します。これにより、オプション A または B にオーディエンスの公開が追加されるので、AEP リアルタイム顧客プロファイル設定が必要になります。

- **フリーフォームのWorkspace プロジェクトが複雑になることなく** チームが迅速で構造化された製品インサイトを必要としており、CJA SKU にガイド付き分析が含まれている場合は、「**オプション D**」を選択します。特定の製品エクスペリエンスの質問に回答するための最速のパスです。

- **多くの組織は複数のオプションを実装しています**：マーケティングチームのキャンペーンダッシュボードのオプション A、分析チームのクロスチャネル分析のオプション B、製品チームのセルフサービスインサイトのオプション D。 これらのオプションは、同じCJA接続とデータビューインフラストラクチャを共有します。

## 実装フェーズ

この節では、このユースケースパターンの実装フェーズについて順を追って説明します。

### フェーズ 1：データ接続

**Application function:** CJA:Data Connection

このフェーズでは、分析のために 1 つ以上のCJA データセットをCJAにバインドするAEP接続を設定します。 接続は、CJAに取り込むデータセット、ユーザー ID を介してデータセット間でイベントをステッチする方法、履歴データとストリーミングデータの取り込み方法を定義します。 これは、AEPの Data Lake とCJAの間の基本的なリンクです。

#### 決定ポイント

このフェーズでは、次の決定が必要です。

>[!NOTE]
>**決定：AEP サンドボックスの選択**
>
>ソースデータセットを含むAEP サンドボックスはどれですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 実稼動サンドボックス | 実稼動レポート用のライブ顧客データ | 実稼動ダッシュボードおよび関係者レポートに使用 |
>| 開発用サンドボックス | 実稼動デプロイメント前のテストとイテレーション | 実稼動環境に昇格する前の初期設定および検証に使用します |

>[!NOTE]
>**決定：データセットの選択とタイプの指定**
>
>接続に含めるAEP データセットと、割り当てる必要があるデータセットのタイプは何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| イベントデータセット | タイムスタンプ付きの行動データ（web インタラクション、アプリイベント、キャンペーンインタラクション、トランザクション） | タイムスタンプフィールドが必要です。ほとんどの分析の中核を形成します |
>| 参照データセット | キー値の参照データ（製品カタログ、キャンペーンメタデータ、ストアの場所） | 共有キーを介してイベントデータに結合（最新の状態のみが使用されます） |
>| プロファイルデータセット | ユーザーレベルの属性（ロイヤルティ層、ライフタイム値、CRM 属性） | ユーザーレベルでエンリッチメントを提供：最新の状態のみが使用されます |

>[!NOTE]
>**決定：ユーザー ID の設定**
>
>クロスデータセットステッチのユーザー ID として機能するフィールドは何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| CRM ID | 組織には、チャネル間で一貫した CRM 識別子があります | 既知の顧客に対して最も正確なクロスチャネルステッチを提供 |
>| ECID （Experience Cloud ID） | 主に匿名の web/アプリの動作の分析 | デバイススコープ。ID 解決なしでデバイス間でステッチされない |
>| メール （ハッシュ化） | メールは、データセット間で共通の識別子です | タッチポイントをまたいでメールが一貫して取り込まれる場合に適しています |
>| カスタム名前空間 | 組織は独自の識別子を使用します | オーディエンス公開用のAEP ID 名前空間と一致する必要があります（オプション C） |

>[!NOTE]
>**決定：バックフィル範囲**
>
>接続にインポートする履歴データの量
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| すべての既存データ | 前年比の比較および長期的なトレンドに必要な履歴の深度を最大化 | 大きなデータセット（数十億件のレコード）のバックフィルは、完了するまで数日かかる場合があります |
>| カスタムの日付範囲 | 関連するのは最近の履歴のみか、ストレージの最適化が問題 | 分析に使用できる履歴の深度を制限 |
>| バックフィルなし | 前向きな分析だけが必要です | 最速の接続設定。新しいデータが流入するまで履歴データは利用できない |

>[!NOTE]
>**決定：ストリーミングの有効化**
>
>新しいデータは、ほぼリアルタイムでCJAに送られるべきですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| ストリーミングを有効にする | ほぼリアルタイムのレポートが必要（データはAEPの取り込み後 90 分以内に利用可能） | 本番環境の接続で最も一般的であり、タイムリーな分析が可能 |
>| バッチのみ | 定期的な更新で十分であり、ストリーミングは必要ありません | よりシンプルな設定：バッチ処理後にデータを利用可能 |

#### データ接続の設定

**UI ナビゲーション：** CJA/接続/新しい接続を作成

- 接続名と説明は、組織の命名規則に従う必要があります

- 接続に追加する前に、人物 ID フィールドが存在し、各データセットに入力されていることを確認します

### フェーズ 2：データ表示の設定

**Application function:** CJA:Data View Configuration

このフェーズでは、分析での接続データの表示方法を定義するデータビューを設定します。 データビューは、ディメンションおよび指標として公開されるスキーマフィールド、値のアトリビューションおよび持続化方法、セッションの定義方法、生のデータを分析可能なコンポーネントに変換する派生フィールドを決定します。 異なる分析観点に対して、1 つの接続から複数のデータビューを作成できます。

#### 決定ポイント

このフェーズでは、次の決定が必要です。

>[!NOTE]
>**決定：コンテナの命名**
>
>ビジネスドメインに一致させるためにコンテナで使用する用語は何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| デフォルト （ユーザー/ セッション / イベント） | 標準分析の用語はチームが理解します | ほとんどの実装に対応 |
>| カスタム名（例：買い物客/訪問/インタラクション） | ビジネスドメイン固有の用語により、ユーザーの採用が促進されます | 技術者以外の関係者が分析の範囲を理解するのに役立ちます |
>| B2B 名（アカウント、エンゲージメント、タッチポイントなど） | アカウントレベルの分析が重視される B2B 分析 | コンテナ範囲を B2B ビジネスコンセプトと連携 |

>[!NOTE]
>**決定：セッションタイムアウト**
>
>無操作状態がセッション境界を定義する期間はどのくらいですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| 30 分（デフォルト） | 標準 web 分析セッション定義 | 業界標準。ほとんどの分析ベンチマークと連携 |
>| 15 分 | ユーザーがタスクを迅速に完了する短時間のコンテンツまたはトランザクションサイト | より多くのセッションを作成し、個別のユーザーインテントをより的確にキャプチャできます。 |
>| 60 分以上 | 長い形式のコンテンツ、複雑な B2B インタラクションまたは大量の研究ジャーニー | 少ないセッション。拡張されたリサーチを 1 つのセッションとしてキャプチャ |
>| 新規セッションイベントを使用したカスタム | 特定のイベント（アプリの起動、キャンペーンのクリックスルーなど）は、常に新しいセッションを開始する必要があります | ビジネス・ロジック主導のセッション境界を提供 |

>[!NOTE]
>**決定：属性モデルのデフォルト**
>
>コンバージョン指標に適用する必要があるデフォルトのアトリビューションモデルは何ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| ラストタッチ （デフォルト） | コンバージョン前に、クレジットは最新のタッチポイントに移動する必要があります | シンプルで直感的。認識チャネルの価値が低い可能性がある |
>| ファーストタッチ | 最初の認識と獲得を推進するチャネルの把握 | 獲得分析に役立ち、育成タッチポイントを無視 |
>| 線形 | すべてのタッチポイントは同等のクレジットを共有する必要があります | 公正な配分。主要なタッチポイントの影響を弱める可能性があります。 |
>| タイムディケイ | 最近のタッチポイントは、離れたタッチポイントよりも多くのクレジットを受け取る必要があります | 最新性と貢献度履歴のバランスを設定します |
>| U字型 | 最初と最後のタッチポイントは、最もクレジットに値します | 獲得チャネルとクローズチャネルの両方を理解するのに適しています |
>| アルゴリズム | CJAの AI モデルを使用したデータドリブン アトリビューション | 最も正確であるが、十分なコンバージョンデータ量が必要 |

>[!NOTE]
>**決定：派生フィールドのロジック**
>
>生データを分析に対応するディメンションに変換するには、カスタムのビジネスルールが必要ですか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| マーケティングチャネル分類（条件の場合） | 生のトラッキングコードは、チャネルグループに分類する必要があります | 最も一般的な派生フィールドのユースケース：チャネル分析にとって重要 |
>| 値のバケット化 | 連続値は、範囲（注文値層など）にグループ化する必要があります | 継続的な指標の分析を簡素化 |
>| フィールドの結合 | 複数のソースフィールドを 1 つのディメンションに組み合わせる必要があります | データセット間で異なるスキーマパスに同じ概念が存在する場合に役立ちます |
>| 正規表現ベースの抽出 | 構造化された値は解析する必要があります（例：キャンペーンコードからのキャンペーンタイプの抽出） | 強力ですが、慎重な正規表現パターン設計が必要 |

#### データ表示の設定

**UI ナビゲーション：** CJA/データビュー/新しいデータビューを作成

主な設定の詳細：

- フェーズ 1 で作成した親接続を選択します。
- レポート要件に合わせたタイムゾーンとカレンダータイプの設定
- XDM スキーマフィールドを適切な永続性（配分と有効期限）設定でディメンションにマッピングします
- 形式（小数、整数、通貨、パーセンテージ、時間）およびアトリビューション設定を使用して、XDM スキーマフィールドを指標にマッピングします
- ディメンションに含める/除外するルールを設定して、無関係な値を除外
- 必要に応じて指標の重複排除を有効にして、重複カウントを防ぐ
- マーケティングチャネル分類、値のバケット化またはフィールド結合の派生フィールドを作成
- 1 つのデータビューにつき最大 5,000 のディメンションと 5,000 の指標
- 1 つのデータビューにつき最大 100 個の派生フィールド

#### オプションが異なる場所

**オプション A （Campaign パフォーマンス分析）の場合：**

キャンペーン固有のディメンション（キャンペーン名、ジャーニー名、チャネルタイプ、メッセージバリアント、件名）をマッピングします。 配信指標のマッピング：送信、配信、開封数、クリック数、バウンス数、購読解除数。 キャンペーン測定理念に基づいてコンバージョン指標のアトリビューションを設定します。

**オプション B （Customer Journey Analytics）の場合：**

クロスチャネルディメンション（ページ名、アプリ画面、チャネル、キャンペーン、製品、コンテンツタイプ）をマッピングします。 すべてのチャネルにわたってエンゲージメントとコンバージョン指標をマッピングします。 比較分析用に複数のアトリビューションモデルを設定します。 チャネル分類およびジャーニーステージ ID 用の派生フィールドを作成します。

**オプション D （ガイド付き分析）の場合：**

製品エクスペリエンス分析に関連するイベントレベルのディメンションと指標（機能名、ユーザーアクション、エンゲージメントタイプ）をマッピングします。 funnelの手順、リテンション条件、成長シグナルを定義するイベントに焦点を当てます。

### フェーズ 3：分析と指標の作成

**Application function:** CJA:Workspace Analysis、CJA：ガイド付き分析、CJA：計算指標の作成

このフェーズでは、分析ワークスペース（フリーフォームプロジェクトまたはガイド付き分析）、派生 KPI の計算指標、セグメント化された分析のフィルター、主要イベントの注釈を構築します。 ここで、ビジネス上の質問に答えるテーブル、ビジュアライゼーション、指標の作成など、分析値が実現されます。

#### 決定ポイント

このフェーズでは、次の決定が必要です。

>[!NOTE]
>**意思決定：分析アプローチ**
>
>この分析では、フリーフォームWorkspace プロジェクトまたはガイド付き分析ワークフローを使用する必要がありますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| フリーフォームWorkspace（オプション A、B、C） | 詳細な探索的分析、カスタムレイアウト、複雑な分類、高度なビジュアライゼーション | 最大限の柔軟性。アナリストのスキルが必要。すべてのビジュアライゼーションタイプをサポート |
>| ガイド付き分析（オプション D） | 構造化された製品インサイト、特定の質問に対する迅速な回答、技術ユーザーの削減 | insightまでの時間を短縮。事前定義済みの分析タイプに制限。Workspaceに保存してさらにカスタマイズ可能 |
>| 両方 | 組織には、詳細な分析と迅速で構造化されたインサイトの両方が必要です | 一般的な質問にはガイド付き分析を、深い探求にはWorkspaceを使用 |

>[!NOTE]
>**決定：ビジュアライゼーションのタイプ**
>
>このユースケースでは、どのビジュアライゼーションがインサイトを最もよく伝えていますか？
>
>| オプション | 選択すべき状況 | 検討事項 |
>| --- | --- | --- |
>| フリーフォームテーブル | ディメンションの分類を使用した詳細なデータ調査 | ほとんどの分析の基盤。最大 10 の分類レベルをサポート |
>| フロービジュアライゼーション | パスの動作（ページフロー、チャネル遷移）について | ジャーニーパスの検出に最適。基数が高く複雑になる場合があります。 |
>| フォールアウトビジュアライゼーション | 定義された一連のタッチポイントを使用したコンバージョンの測定 | funnel分析に最適（各手順のドロップオフを明確に示す） |
>| コホートテーブル | 獲得コホートによる経時的なリテンション分析 | さまざまなグループがどの程度維持されているかを示す。ライフサイクル分析に重要 |
>| アトリビューションパネル | コンバージョン指標のアトリビューションモデルの比較 | 並列モデル比較。明確なコンバージョンイベント定義が必要です。 |
>| 数値/変更の概要 | エグゼクティブ KPI の表示と期間ごとの比較 | 指標をきれいに一目で確認できる、ダッシュボードヘッダーに最適 |

>[!NOTE]
>**決定：計算指標の数式**
>
>ベースデータビュー指標の他に、計算指標が必要なビジネス KPI は何ですか。
>
>| 指標パターン | 数式の例 | 使用するタイミング |
>| --- | --- | --- |
>| 比率/率 | 注文件数/訪問回数 | コンバージョン率、クリックスルー率、バウンス率 |
>| フィルター適用済み指標 | 売上高（チャネル = 「電子メール」） | チャネル固有またはセグメント固有の測定値 |
>| 項目別平均 | 売上高/注文件数 | 平均注文額、訪問あたりの売上高 |
>| 数式を複合 | （売上高 – コスト） /売上高 | 利益率、ROI 計算 |
>| エンゲージメントスコア | インタラクションの加重合計 | チャネル全体の複合エンゲージメントスコアリング |

#### 分析と指標の設定

**UI ナビゲーション：**

- Workspace:CJA/Workspace/プロジェクト/プロジェクトを作成/空のプロジェクト
- ガイド付き分析：CJA/ホーム/ガイド付き分析（またはWorkspace/作成/ガイド付き分析）
- 計算指標：CJA/コンポーネント/計算指標/作成
- フィルター：CJA/ コンポーネント / フィルター/ フィルターを作成

主な設定の詳細：

- フェーズ 2 で作成したデータ ビューをプロジェクト データ ビューとして選択します
- 分析に適した日付範囲と比較期間を設定する
- ディメンションを行に、指標を列にドラッグして、フリーフォームテーブルを作成します
- ディメンションの分類を追加して、より深いレベル（キャンペーン別のチャネル、製品別のページなど）のデータを調べます
- オーディエンス固有の分析（ユーザーレベル、セッションレベルまたはイベントレベルの範囲）の再利用可能なフィルター（セグメント）を作成する
- 主要なビジネスイベント（製品発売、キャンペーン、インシデント）をマークする注釈を追加します。
- 計算指標の形式（小数、パーセント、通貨、時間）と極性（増加は良好/増加は不良）を設定します
- 表示権限または編集権限でCJA ユーザーとワークスペースプロジェクトを共有する

#### オプションが異なる場所

**オプション A （Campaign パフォーマンス分析）の場合：**

配信指標とエンゲージメント指標に基づいてキャンペーンディメンションが分類されたフリーフォームテーブルを作成します。 開封率、クリックスルー率、コンバージョン率、メッセージあたりの売上高、キャンペーン ROI の計算指標を作成します。 トレンドビジュアライゼーションを追加して、キャンペーンのパフォーマンスを経時的に追跡します。 セグメント比較を使用してキャンペーンのバリアントを比較します。

**オプション B （Customer Journey Analytics）の場合：**

フォールアウトビジュアライゼーションを作成して、ジャーニーのドロップオフポイントを特定します。 フロービジュアライゼーションを作成して、チャネル間のナビゲーションパターンを検出します。 獲得コホートでリテンションを測定するコホートテーブルを作成します。 アトリビューションパネルを設定して、コンバージョン指標のアトリビューションモデルを比較します。 ジャーニー完了率、クロスチャネルエンゲージメントスコアおよびライフサイクルステージコンバージョンの計算指標を作成します。

**オプション C （Analytics とオーディエンスの公開）の場合：**

オプション A または B から分析ワークスペースを構築し、分析中に価値の高いセグメントやパフォーマンスの低いセグメントを特定します。 フェーズ 5 で公開用にこれらのセグメントを取り込むCJA フィルターを作成します。

**オプション D （ガイド付き分析）の場合：**

ビジネス上の質問に基づいて、適切なガイド付き分析タイプを選択します。 主要なイベント、日付範囲、カウント方法およびセグメント比較を設定します。 さらにカスタマイズするために、完了した分析をWorkspace プロジェクトのパネルとして保存する。

### フェーズ 4：ダッシュボードの公開

**アプリケーション機能：** CJA：ダッシュボードとスコアカードの公開

このフェーズでは、関係者に KPI の可視性を提供するインタラクティブなダッシュボード（Workspace プロジェクト）とモバイルスコアカードを作成します。 ダッシュボードでは、概要数値、トレンドライン、分類、注釈を使用して、エグゼクティブや運用上の可視性を提供します。 モバイルスコアカードは、[!DNL Adobe Analytics] ダッシュボードモバイルアプリを介して、一目でパフォーマンスデータを提供します。

#### 決定ポイント

このフェーズでは、次の決定が必要です。

>[!NOTE]
>**決定：ダッシュボードタイプ**
>

## 実装に関する考慮事項

この節では、このユースケースパターンのガードレール、一般的な落とし穴、ベストプラクティスおよびトレードオフの決定について説明します。

### ガードレールと制限

この実装には、次のガードレールと制限が適用されます。

- **接続の制限：** 組織あたりの最大接続数は、CJA SKU の使用権限によって制限されます。 1 つの接続には、1 つのAEP サンドボックスからのデータセットのみを含めることができます。 — [CJA ガードレール ](https://experienceleague.adobe.com/en/docs/analytics-platform/using/cja-admin/guardrails)
- **データビューの制限：** データビューあたり最大 5,000 個のディメンションと 5,000 個の指標。 最大 5 レベルのネストされた関数を含む、データビューあたり最大 100 個の派生フィールド。
- **Workspaceの制限：** プロジェクトあたり最大 40 個のパネル。 フリーフォームテーブルでは、最大 10 個のディメンション分類をサポートしています。 レポートリクエストあたり最大 50,000 行。
- **スコアカードの制限：** モバイルスコアカード 1 つにつき最大 16 個のタイル。
- **ストリーミングの待ち時間：** ストリーミングデータは、通常、CJAの取得から 90 分以内にAEPで使用できます。
- **オーディエンス公開の制限：** CJA ユーザーあたり最大 75 個の公開済みオーディエンス。 最小更新頻度は 4 時間ごとです。
- **ガイド付き分析の制限：** Funnel分析は、最大 15 ステップをサポートします。 セグメント比較では、最大 3 つのセグメントを同時にサポートします。

### よくある落とし穴

このパターンを実装する際には、次の一般的な問題に注意してください。

- **データセット間でのユーザー ID の不一致：** 接続内のすべてのデータセットでは、データセット間分析に一貫したユーザー ID フィールドを使用する必要があります。 ユーザー ID が一致しないと、顧客表示が断片化され、同じ人物が複数の人物として表示されます。 接続を作成する前に、ユーザー ID の一貫性を確認します。

- **バックフィルに予期せず時間がかかる：** 数十億件のレコードを含む大きなデータセットは、バックフィルに数日かかる場合があります。 実装タイムライン中にこれを計画し、早くバックフィルを開始します。 接続の詳細表示で進行状況を監視します。

- **ほとんどのディメンション値で「未指定」と表示されるデータビュー：** マッピングされたスキーマフィールドは、ソースデータに散発的に入力される場合があります。 設定エラーを想定する前に、ソースデータセットでデータ品質を確認してください。 null 値を処理するには、派生フィールドの使用を検討してください。

- **セッション数が正しく表示されない：** セッションタイムアウトの設定は、セッションスコープの指標に大きく影響します。 タイムアウトを非常に短くすると、作成されるセッションの数が増え、タイムアウトを非常に長くすると、作成されるセッションの数が少なくなります。 新しいセッション開始イベントでも、予期せずセッションがフラグメント化される場合があります。 既知のユーザー行動パターンに照らしてセッション設定を確認およびテストします。

- **アトリビューションモデルが期待どおりに適用されない：** アトリビューションモデルは指標にのみ適用され、ディメンションには適用されません。 ルックバックウィンドウがビジネスサイクルに対して適切に設定されていることを確認します。 短いルックバックウィンドウで、早期funnelのタッチポイントを見逃す可能性があります。

- **ゼロまたは予期しない値を返す計算指標：** 式で参照されるベース指標に、選択した日付範囲のターゲットデータビューのデータがあることを確認します。 比率指標でゼロ除算をチェックします。 指標の定義を取得し、式の構造を確認します。

- **オーディエンスの公開に失敗します（オプション C）:** CJA接続には、AEP ID 名前空間に解決される有効なユーザー ID が必要です。 ID 名前空間の設定と、AEP リアルタイム顧客プロファイルがターゲットサンドボックスで有効になっていることを確認します。

### ベストプラクティス

実装を成功させるには、次のベストプラクティスに従います。

- **単一の包括的な接続から開始：** 関連するすべてのデータセットを含む 1 つの接続を作成してから、様々な分析の観点で複数のデータビューを作成します。 これにより、接続の急増を回避し、管理を簡素化できます。

- **マーケティングチャネル分類に派生フィールドを使用：** 生のトラッキングコードに依存するのではなく、Case When ロジックを使用して派生フィールドを作成し、トラフィックをマーケティングチャネルに分類します。 これにより、すべての分析で一貫したチャネルレポートが作成されます。

- **指標ディクショナリの作成：** すべての計算指標を、その式、使用目的および期待値の範囲と共にドキュメント化します。 この辞書を分析チームと共有して、プロジェクト間で一貫した指標の使用を確保します。

- **オーディエンスのデータビューの設計：** 様々な関係者グループに対して個別のデータビュー（キャンペーンに焦点を当てたディメンションと指標を含むマーケティングデータビュー、および機能とエンゲージメントディメンションを含む製品データビュー）を作成します。 これにより、各ユーザーグループのコンポーネントリストが簡単になります。

- **すべてに注釈を付ける：** キャンペーンの開始、サイトの変更、技術的なインシデント、季節性、データの傾向を説明する可能性のあるイベントに関する注釈を作成します。 注釈は、数か月後にダッシュボードを確認する際に、重要なコンテキストを提供します。

- **計算指標を手動の計算と比較してテストする：** ダッシュボードの計算指標を利用する前に、計算指標とそのベースコンポーネントを並べてレポートを実行します。 計算された値が手動計算と一致することを確認します。

- **フィルターの戦略的な使用：** 一般的なセグメントパターン（新着と再来訪、モバイルとデスクトップ、地域による）に対して再利用可能なフィルターを作成します。 すべてのフリーフォームテーブルに埋め込むのではなく、パネルレベルのフィルターとして適用します。

- **接続の正常性を定期的に監視します：** 接続の詳細表示で、スキップされたレコード、失敗したバッチ、ストリーミングの遅延を確認します。 接続レベルでのデータ品質の問題は、すべてのダウンストリーム分析に影響を与えます。

### トレードオフの決定

実装を計画する際は、次のトレードオフを考慮してください。

>[!NOTE]
>**トレードオフ：分析の深度とinsightまでの時間**
>
>オプション B （Customer journey analytics）は、クロスチャネルに関して最も深いインサイトを提供しますが、接続設定、データビューの設計、派生フィールドの作成には事前に大きな投資が必要です。 オプション D （ガイド付き分析）は、構造化されたワークフローでinsightまでの時間を短縮しますが、分析の柔軟性は低くなります。
>
>- **オプション B のお気に入り：** 包括的な理解、複雑なマルチチャネル分析、アトリビューションモデリング、カスタム KPI 開発
>- **オプション D のメリット：** スピード、アナリスト以外のユーザー向けのアクセシビリティ、構造化された製品体験の質問
>- **推奨事項：** オプション B のインフラストラクチャを並行して構築しながら、すぐに製品を把握するには、オプション D から始めてください。 多くの組織では、異なるチームに対して両方を同時に実行しています。

>[!NOTE]
>**トレードオフ：バックフィルの完全性と接続準備**
>
>すべての履歴データを読み込むと、前年比の比較や長期的なトレンド分析に最大限の分析深度が得られますが、大規模なデータセットのバックフィルには数日かかる場合があります。 バックフィルを最近の期間に制限すると、接続の準備が早くなりますが、履歴分析は制限されます。
>
>- **すべてのデータの恩恵：** 長期的なトレンド分析、前年比の比較、履歴の長いコホート分析
>- **限られたバックフィルのメリット：** 接続への対応の迅速化、ダッシュボードの初回起動までの時間の短縮、ストレージの最適化
>- **推奨事項：** 戦略的分析をサポートする実稼動用の接続のデータをすべてバックフィルします。 開発用の接続と概念実証の実装には、限定的なバックフィルを使用します。

>[!NOTE]
>**トレードオフ：単一の包括的なデータビューと複数の焦点を絞ったデータビュー**
>
>すべてのディメンションと指標を含んだ単一のデータビューは、統合された分析ワークスペースを提供しますが、コンポーネントリストを使用するとユーザーを圧倒できます。 複数のフォーカスされたデータビュー（チームまたはユースケースごとに 1 つ）を使用すると、コンポーネントエクスペリエンスが簡素化されますが、複数の設定を維持する必要があります。
>
>- **単一のデータビューのお勧め：** 統合分析、クロスドメイン分類、シンプルな管理
>- **複数のデータビューをお勧めします。** Cleaner のコンポーネントリスト、チーム固有の用語、ユースケースごとに異なるセッション定義
>- **推奨事項：** 1 つのプライマリデータビューから始めて、コンポーネントリストの複雑さが採用の障壁になる場合は、追加の焦点を当てたデータビューを作成します。 すべてのデータビューで同じ接続を参照できます。

>[!NOTE]
>**トレードオフ：リアルタイムストリーミングとバッチのみの取り込み**
>
>CJA接続でストリーミングを有効にすると、ほぼリアルタイムのデータ（AEPの取得後 90 分以内）が提供されますが、より多くのデータが継続的に処理されます。 バッチのみの取り込みは、データを定期的に処理し、遅延が発生する可能性があります。
>
>- **ストリーミングのお気に入り：** タイムリーなレポート、アクティブなキャンペーンの監視、ほぼリアルタイムのダッシュボード
>- **バッチのみのお気に入り：** 設定がシンプルで予測可能な処理ウィンドウが実現し、毎週または毎月のレポートに十分です。
>- **推奨事項：** 実稼動接続のストリーミングを有効にする。 アクティブなキャンペーンの監視や運用ダッシュボードのタイムリーなデータの価値と比較すると、増分処理コストは最小限です。
